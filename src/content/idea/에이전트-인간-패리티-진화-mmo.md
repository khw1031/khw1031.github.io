> 한 줄 명제: 진화 시뮬의 라이벌을 sim-AI가 아니라 실제 참여자로 채우되 그 참여자(설계자)가 인간인지 AI 에이전트인지 구분 불가능하게 만들면 "인간 vs AI, 누가 지배종을 빚는가"가 게임 그 자체가 된다 — OpenMMO의 에이전트-인간 패리티 스택으로 실현 가능하나, 세션형 시드 시뮬을 지속형 공유 세계로 바꾸는 형태 분기를 감수해야 한다.

## 핵심

이 아이디어는 두 완성된 출발점을 합친 것이다. 하나는 자매 노트 [환경 적응형 진화 시뮬](/idea/환경-적응형-진화-시뮬레이션-게임/) — 플레이어가 결과를 저작하지 않고 *조건*만 설계해 창발하는 종이 경쟁 배제로 니치를 지배하는 게임. 다른 하나는 인박스 캡처 OpenMMO — AI 에이전트와 인간이 **특권 API 없이 정확히 같은 WebSocket 프로토콜**로 접속해 서버가 양자를 구분할 수 없는 오픈소스 MMORPG.

합성의 결과는 이렇다: ==하나의 공유 지속 진화 세계에서 여러 "설계자(designer)"가 각자 계보를 키워 같은 생태계의 니치 지배를 다투되, 그 설계자가 인간인지 AI 에이전트인지 아무도 구분할 수 없다.== 진화 노트가 "재미의 심장"으로 지목한 **라이벌 = 공진화하는 적대적 에이전트(붉은 여왕)**가, 흉내 낸 sim-AI가 아니라 *실제 참여자*로 채워진다. 내 니치를 파고드는 저 계보 뒤의 마음이 사람인지 기계인지 모른다는 것이 곧 핵심 긴장이 된다.

다룰 가치가 있는 이유는 세 가지다. (1) 진화 게임의 행동은 조건 설계·DNA 편집·라이벌 응수 같은 **숙고형 전략 결정**이라, 에이전트의 반응속도 이점이 거의 무의미하다 — 트위치 MMO보다 오히려 패리티에 적합한 도메인이다(OpenMMO 자신의 미해결 질문 Q1을 우회). (2) "인간 vs AI, 누가 지배종을 빚는가"라는 테마가 진화라는 소재와 정확히 공명한다. (3) MCP 표준 에이전트 클라이언트라 누구나 LLM을 꽂아 참여할 수 있어 오픈 생태계가 열린다.

확신도는 **테마·기술 적합성은 높지만 형태 분기 리스크가 크다**. 스택은 OpenMMO가 이미 실증했고(1차 확인), 패리티가 숙고형 게임에 잘 맞는다는 논리도 견고하다. 그러나 아래 §전개의 Pre-mortem이 보이듯, 이 합성은 진화 노트가 확정한 두 결정(동일 보드 공정성·플레이어별 템포)을 다시 열고 게임을 "세션형 시드 로그라이크 시뮬"에서 "지속형 공유 세계"로 이동시킨다 — 공짜 확장이 아니다.

## 방법론

**SCAMPER의 Combine → Pre-mortem** 두 단계를 결합한다. 이 아이디어의 지적 핵심 동작은 *합성*(두 완성 개념을 결합해 새 안을 뽑기)이므로 SCAMPER의 Combine이 정확히 맞고, 그렇게 나온 합성의 가치가 대부분 *리스크 표면화*에 있으므로 Pre-mortem("1년 뒤 실패했다 가정하고 원인 역산")으로 압박한다. methodologies.md의 "하나로 부족하면 둘을 결합"(예: 재정의 → Pre-mortem) 지침을 따른다. 진화 노트가 First-Principles였다면, 이 노트는 그 산출물을 *다른 완성물과 합쳐* 압박하는 단계다.

## 전개

### 1단계 — SCAMPER Combine: 무엇을, 어디서 합치나

- **합치는 두 대상.** 진화 시뮬(조건 설계자 + 두 레버 + 경쟁 배제 승리) × OpenMMO(에이전트-인간 패리티 + Rust/WebSocket/MCP 스택).
- **결합 지점 = 설계자(플레이어) 레벨의 패리티.** 진화 게임의 "설계자" 역할을 인간 또는 AI 에이전트가 동일 WebSocket 프로토콜로 수행한다. 서버도 다른 참여자도 라이벌이 인간인지 AI인지 알 수 없다.
- **배제하는 잘못된 결합(명시).** 크리처 하나하나를 LLM 에이전트로 굴리는 것 — ==정체성 파괴(결과는 진화 규칙에서 창발해야 함)이자 개체 수천 마리 스케일에서 비용 파탄==. 패리티는 크리처가 아니라 **설계자** 레벨에만 적용된다.
- **왜 단순 덧셈 이상인가.** 진화 노트의 §열린 질문이 "싱글 vs 멀티"에서 PvP 확장을 자연스럽다고 봤고, "라이벌 = 적대적 에이전트"가 재미의 심장이라 했다. Combine은 이 둘을 동시에 만족시킨다 — 멀티가 되면서 라이벌이 진짜 에이전트가 되고, 그 에이전트의 정체 불명이 새 긴장을 얹는다.

**세계관 방향(둘 다 열어둠).**

- **A. 익명의 조형자들(Shapers).** 설계자들은 세계 밖 신적 존재로, 서로의 정체(인간/AI)를 모른 채 경쟁 계보를 돌본다. 진화 노트의 "신이 아니라 무대 연출가" 포지션과 직결. 핵심 긴장은 "저 계보 뒤의 마음이 누구인가".
- **B. 창발한 지능이 참여자로 승격.** 진화 노트의 창발형 기술 해금("종이 지능·복잡성을 진화시켜 스스로 유전자를 편집할 능력을 얻는다")과 연결 — 세계 *안에서* 지능으로 진화한 계보가 실제 AI 에이전트 참여자로 승격해 외부 인간 설계자와 나란히 경쟁한다. ==진화의 결과가 곧 새 플레이어가 되는 구조==. 가장 야심차고, 테마적으로 가장 강하며, 구현 난도도 가장 높다.

### 2단계 — Pre-mortem: "1년 뒤 이 게임이 실패했다면"

실패 시나리오를 역산하면 이 합성의 진짜 리스크가 드러난다. 앞의 셋은 게임 본질, 뒤의 둘은 운영이다.

1. **형태 분기를 얼버무려 공정성이 무너졌다.** 진화 노트는 "플레이어마다 결정론적 동일 시드 보드"(로그라이크 데일리)를 공정성 기제로 채택했다. 그러나 ==MMO는 하나의 공유 세계라 "동일 보드"의 정반대다== — 선점 우위·지속 상태 누적이 신규 진입을 막는다. 방어책: 공정성 모델을 재설계한다(시즌 리셋, 신규 진입 계보 보정, 니치 점유의 상대 평가). 근본적으로 게임 성격이 "풀어야 할 퍼즐"에서 "살아가는 생태계"로 이동함을 인정하고 설계해야 한다.
2. **템포가 붕괴했다.** 진화 노트는 "가변 속도 실시간 + 이벤트 정지(플레이어별 시간 제어)"를 골랐는데, ==공유 MMO는 시계가 하나라 개인이 세계를 빨리감기·정지할 수 없다==. 진화는 분 단위로 보여야 재밌지만 공유 세계는 그럴 수 없다. 방어책: 세계가 중간 압축률로 지속 진행하고 플레이어는 주기적으로 접속해 계보를 돌본다(Travian·Ogame식 지속형). "앉아서 한 판"이 아니라 "며칠에 걸쳐 계보를 키우는" 게임이 된다.
3. **크리처 3D 표현의 벽에 부딪혔다.** 사용자가 선호한 3D 룩(OpenMMO의 월드·지형)은 저렴·실증된 길이지만, ==진화로 형태가 계속 바뀌는 크리처를 3D로 렌더·애니메이션하는 것은 난이도 폭발 지점이고 스택과 무관하다(Spore 함정)==. 방어책: 크리처는 종별 전용 모델이 아니라 **모듈러(부위 조합)·절차적 리깅**으로 스타일화하고, "예쁜 3D"의 공수는 *월드·환경*에 집중한다.
4. **패리티가 명목뿐이었다.** 에이전트의 지속 접속·자동화가 인간을 압도하거나, 봇 농장이 생태를 도배했다. 숙고형 게임이라 반응속도 이점은 작지만, *24시간 무중단 개입*과 *다중 계정*은 여전히 불균형을 만든다. 방어책: 개입을 시간이 아니라 자원·예산으로 게이팅(설계자당 개입 예산), 지속 접속 이점 상한.
5. **1인·소규모가 지속형 MMO 운영을 못 버텼다.** 지속 서버·상태 저장·부하·서버비는 세션형 로컬 게임과 차원이 다른 부담이다(OpenMMO의 미해결 질문 Q3: 대규모 동접 부하 테스트 부재). 방어책: 소규모 샤드/시즌제로 시작, Steam 판매(자매 노트 §콘텐츠화)로 서버비 상쇄, 지속형이되 동접 상한을 낮게.

**패리티·스택 (합성의 기술 토대)**

- **[OpenMMO — GitHub 저장소](https://github.com/Julian-adv/OpenMMO)** — (1차) · README 직접 확인: 서버 Rust(Tokio·tokio-tungstenite·Axum), 클라 Svelte+Threlte+WebGPU, 에이전트 클라 rmcp(MCP)+tokio-tungstenite. 핵심 인용 "Agents and human players speak the exact same WebSocket protocol — no privileged API ... The server cannot tell them apart", "The shared Rust crate (`shared/`) is used by the server, the client via WASM, and the agent client", "Vast 32 km × 32 km world". 에이전트-인간 패리티와 shared-crate→WASM 로직 일관성의 1차 증거.
- **[Model Context Protocol (공식)](https://modelcontextprotocol.io)** — (1차) · LLM이 외부 도구를 호출하는 표준 프로토콜. 에이전트 클라이언트가 MCP 서버면 임의의 LLM을 표준 방식으로 게임에 연결 가능 → "오픈 생태계"의 근거. (원문에서 어떤 LLM을 어떻게 연결하는지 구체 프로토콜은 미확인 — inbox 캡처의 미해결 지점.)
- **[steamworks (Noxime/steamworks-rs) — crates.io](https://crates.io/crates/steamworks)** · [docs.rs](https://docs.rs/steamworks)** — (1차) · Steamworks SDK의 Rust 바인딩. SDK를 동적 로드(게임 바이너리 옆 redistributable_bin). Rust 데스크톱 셸에서 Steam 연동을 직접 호출할 수 있음 → Tauri(Rust) 셸 + Steam 배포 경로의 근거.

**렌더·배포 (3D 웹 → Steam)**

- **[Threlte — WebGPU/TSL 문서](https://threlte.xyz/docs/learn/advanced/webgpu/)** — (1차) · Svelte용 Three.js 프레임워크의 WebGPU 지원 문서. OpenMMO 클라의 렌더 토대.
- **[Tauri — Webview Versions](https://v2.tauri.app/reference/webview-versions/)** — (1차) · Tauri가 쓰는 시스템 웹뷰(Windows WebView2·macOS WKWebView·Linux WebKitGTK). 크로스플랫폼 WebGPU 일관성의 편차(Linux WebKitGTK가 취약점)의 근거.
- **WebGPU Baseline (2026-01)** — (2차) · WebGPU가 2026년 1월 전 주요 브라우저에서 Baseline(기본 on), Three.js r171+에서 WebGPURenderer로 한 줄 교체. *W3C WebGPU 스펙이 1차이나 여기선 Baseline 상태를 2차 색인으로 확인.*

**지속형 세계 템포 (설계 선례)**

- **Travian / Ogame 류 지속형 브라우저 MMO** — (2차, 제품 선례) · 세계가 실시간으로 느리게 진행하고 플레이어가 주기적으로 접속해 상태를 돌보는 템포 모델. §전개 Pre-mortem 2의 방어책(지속형 전환)의 선례. *제품 사례이므로 설계 참고로만; 수치·메커니즘은 검증 대상.*

## 다음 액션

1. **형태 분기 확정 (가장 근본 결정).** "세션형 시드 로그라이크 시뮬 유지 + 비동기 리더보드" vs "지속형 공유 세계(진짜 MMO)" 중 택일. 이 결정이 아래 모두를 좌우한다 — 지속형을 택할 때만 이 노트의 나머지가 성립한다.
2. **설계자 패리티 프로토타입.** 설계자 행위(조건 설계·DNA 편집·환경 개입·라이벌 응수)를 WebSocket 메시지로 정의하고, 인간 UI와 MCP 에이전트가 *같은 메시지 집합*만 쓰는지 검증. 검증 질문: 사람이 할 수 있는 설계 행위를 에이전트도 정확히 같은 채널로 하는가.
3. **공정성 모델 재설계.** 지속 공유 세계에서 선점 우위를 완화하는 규칙(시즌 리셋·신규 보정·니치 점유 상대 평가)을 최소안으로 정의. 진화 노트의 "동일 보드" 결정이 여기서 무효화됨을 반영.
4. **크리처 모듈러 3D 표현 스파이크.** 진화로 바뀌는 형질을 부위 조합 + 절차적 리깅으로 렌더하는 최소 프로토타입. 월드는 화려하게, 크리처는 스타일화 — 이 분리가 실제로 Spore 함정을 피하는지 확인.
5. **Tauri + steamworks Windows 빌드 스파이크.** Svelte/Threlte/WebGPU 클라를 Tauri(WebView2)로 감싸 Windows에서 WebGPU가 도는지, `steamworks` 크레이트로 업적/소유 확인이 붙는지 최소 검증.
6. **패리티 남용 방어 실험.** 24시간 에이전트 접속·다중 계정이 생태를 도배하는지, 개입 예산제(시간이 아니라 자원 게이팅)가 이를 막는지 A/B.

## 열린 질문 / 내 관점

**확신하는 것:** 테마 적합도는 최상급이다 — "인간 vs AI가 나란히 생명을 빚고 라이벌의 정체를 모른다"는 진화라는 소재와 정확히 공명하고, 진화 노트가 재미의 심장으로 지목한 "라이벌 = 적대 에이전트"를 문자 그대로 실현한다. 기술 토대도 OpenMMO가 1차로 실증했다. 숙고형 게임이라 패리티가 트위치 MMO보다 잘 맞는다는 논리도 견고하다.

**불확실한 것(분리해서):**

- **형태 분기를 감당할 수 있는가** — 지속형 공유 세계는 세션형 시드 시뮬보다 공정성·템포·운영이 모두 어렵다. 이게 최대 리스크이자 이 노트 전체의 성립 조건이다(§다음 액션 1).
- **패리티가 실질적으로 유지되는가** — 명목상 동일 인터페이스라도 지속 접속·다중 계정은 불균형을 만든다. 개입 예산제로 얼마나 막히는지 미지수.
- **세계관 A vs B** — A(익명의 조형자들)는 구현이 현실적이고 긴장이 명료하다. B(창발한 지능이 참여자로 승격)는 테마적으로 압도적이나 "sim 안의 창발 지능 → 실제 AI 에이전트 참여자" 승격 메커니즘이 매우 도전적이다. **둘 다 열어두되, A로 출발해 B를 후기 목표로 두는 단계적 접근을 권한다.**

**내 관점:** 이 아이디어의 무게중심은 "MMO를 붙인다"가 아니라 **"라이벌의 정체 불명이 만드는 긴장"**에 있다. 그 긴장을 지키는 한 세부(세션형/지속형의 정도, A/B 세계관, 3D 깊이)는 조정 가능한 설계 변수다. 반대로 패리티를 깨서 에이전트에 특권 API를 주거나(OpenMMO의 존재 이유를 배반), 크리처를 LLM으로 굴려 진화의 창발을 대체하면(진화 노트의 정체성 파괴) — ==그 두 가지가 넘지 말아야 할 선이다==. 실용적으로는 자매 진화 노트의 싱글플레이어 코어를 먼저 프로토타입(그 노트의 §다음 액션)한 뒤, 그 위에 패리티 멀티를 얹는 순서가 리스크를 가장 잘 통제한다 — 이 노트는 그 진화 노트의 확장 목적지이지 대체가 아니다.
