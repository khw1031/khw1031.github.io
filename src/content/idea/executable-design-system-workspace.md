> 한 줄 명제: Astryx의 "하나의 원본 → 여러 소비자" 구조는 옳지만 소비자가 개발자·에이전트뿐이다 — 같은 척추에 **디자이너의 시안 surface**와 **기획의 정책 뷰**를 얹으면 "디자인 시스템 = 실행 가능한 지능"을 세 역할이 함께 쓰는 워크스페이스가 된다.

## 핵심

이 아이디어는 디자이너·개발자·기획이 **디자인 시스템이 반영된 시안**을 함께 만들고 합의하도록 돕는 SaaS다. 출발점은 Meta의 Astryx가 증명한 구조 하나 — 컴포넌트마다 하나뿐인 문서 원본(`.doc.mjs`)에서 CLI·문서 사이트·MCP 서버가 **동시에 파생**되어, "사람과 AI 에이전트가 같은 참조를 읽는다"가 슬로건이 아니라 실제 코드 구조가 된다는 것이다. 그리고 Astryx의 MCP는 코드를 생성하는 서버가 아니라 에이전트의 컨텍스트 예산에 맞춰 사전 스코어링된 **문서 조회 서버**(`search`/`get`)라는 점이 중요하다.

문제는 Astryx의 소비자가 **개발자와 에이전트뿐**이라는 데 있다. 디자이너가 시각적으로 시안을 구성할 surface도, 기획이 요구사항·제약을 얹을 자리도 없다. 게다가 Astryx는 이미 코드로 작성된 디자인 시스템(`.doc.mjs`)을 전제하므로 Meta 규모가 아닌 팀에는 도입 비용이 크다. "meta의 방식을 더 사용성 높게 차용한다"는 이 아이디어의 목표는 정확히 이 사용성 격차를 메우는 것이다 — **같은 단일-원본 척추를 유지하되, 그 위에 디자이너·기획용 소비자를 추가하고, 기존 디자인 시스템을 흡수**해 코드-퍼스트 재작성 없이 시작하게 한다.

이는 Figma 전략 리포트가 말한 "변혁적 경로"와 같은 방향이다 — ==캔버스가 진실의 중심이 아니라, 디자인 시스템·코드·정책이 진실==이고 시안(캔버스)은 그것을 **파생한 뷰**가 된다. 시안이 실제 승인된 컴포넌트로 구성되므로 "디자인 시스템 반영"이 사후 검증이 아니라 **구성 방식 자체로 보장**된다.

## 방법론

- **JTBD** — 세 역할(디자이너·개발자·기획)이 각각 무슨 진전을 "고용"하는지로 제품 경계를 잡는다.
- **First-Principles** — Astryx의 메커니즘을 프리미티브 단위로 분해해 *유지할 것*과 *이 트리오에 부족한 것*을 가른다(단순 벤치마킹이 아니라 왜 그 구조인지부터).
- **Pre-mortem** — "1년 뒤 실패했다"를 가정해 도입 비용·페르소나 함정·incumbent 리스크를 선제 발굴한다.

## 전개

### 1) JTBD — 세 페르소나의 진전

- **디자이너**: "디자인 시스템을 지키면서 빠르게 시안을 만들고 싶다." 지금은 Figma에서 그린 뒤 개발이 재해석 → 어긋남. 원하는 진전 = **실제 컴포넌트로 구성되는 시안**(UXPin Merge류)이 자동으로 DS를 지킴.
- **개발자**: "시안을 우리 실제 토큰·컴포넌트로 정확히 구현하고, 에이전트가 우리 DS를 알고 코드를 짜게 하고 싶다." 원하는 진전 = Code Connect/Astryx MCP처럼 **에이전트가 승인된 구현을 참조**.
- **기획**: "시안과 요구사항·제약(승인 패턴, deprecated, 접근성, 규제)을 한곳에서 합의하고 싶다." 원하는 진전 = 시안에 **정책·요구사항이 연결된 합의 공간**. Figma 리포트가 "다음 워크스페이스 = IDE + 공간 협업 + AI 오케스트레이션"이라 본 지점이며, 대개 제품들이 비워두는 자리다.

### 2) First-Principles — Astryx에서 유지할 것 / 부족한 것

| Astryx 프리미티브 | 유지 가치 | 이 트리오엔 |
| --- | --- | --- |
| 컴포넌트당 단일 원본 → 다중 파생 | 세 소비자가 절대 어긋나지 않음 | **유지**(척추) |
| MCP = 문서 조회(`search`/`get`), 코드 생성 아님 | 컨텍스트 예산 절약, 환각 감소 | **유지** |
| 자기서술 CLI·매니페스트·안정 에러코드 | 에이전트 친화 | **유지** |
| 소비자 = 개발자·에이전트뿐 | — | **부족 → 디자이너·기획 surface 추가** |
| 코드로 이미 쓰인 DS(`.doc.mjs`) 전제 | Meta 규모엔 OK | **부족 → 기존 DS 흡수(Figma/shadcn)** |
| 시안/화면 조립 기능 없음 | — | **부족 → 실제 컴포넌트 시안 surface** |
| 단일 원본 = 단일 실패점(원본 부실 시 3곳 동시 부실) | — | **리스크 → 원본 검증·거버넌스 필요** |

### 3) 제품 개념 — 단일 원본, 세 소비자

디자인 시스템(컴포넌트·토큰·패턴·정책)을 **하나의 구조화된 원본**으로 두고 세 소비자를 파생한다:

1. **디자이너 시안 surface** — 실제 승인 컴포넌트로 화면을 조립(그려서 나중에 매핑하는 게 아니라 처음부터 코드 컴포넌트로). "DS 반영"이 구성 방식으로 보장.
2. **개발자·에이전트 MCP** — `search`/`get` 조회 + Code Connect식 실제 구현 매핑. 에이전트가 승인된 컴포넌트로 시안을 구현.
3. **기획 정책 뷰** — 승인/deprecated/접근성/규제 제약과 요구사항을 원본에 얹어, 시안 검토 시 위반을 즉시 노출.

도입 비용은 **흡수**로 낮춘다 — Figma Code Connect나 shadcn/v0 레지스트리 같은 기존 자산을 원본으로 빨아들여 `.doc.mjs` 재작성을 요구하지 않는다.

### 4) 차별화

| 대상 | 그들의 방식 | 이 아이디어의 차이 |
| --- | --- | --- |
| **Figma (Dev Mode MCP + Code Connect)** | 캔버스가 진실, 코드를 캔버스에 연결 | 진실은 DS/코드/정책, 시안은 파생 뷰(변혁적 경로) |
| **Astryx** | 단일 원본 → CLI/문서/MCP (개발자·에이전트) | 같은 척추 + 디자이너·기획 surface, 기존 DS 흡수 |
| **UXPin Merge** | 디자이너가 실제 컴포넌트로 디자인 | + 에이전트 MCP 척추 + 기획 정책층 + 단일 원본 파생 |
| **Builder.io** | 비주얼 편집 → 프로덕션 코드 | + DS를 "실행 가능한 지능"으로 다루는 정책·에이전트 계층 |

### 5) Pre-mortem (1년 뒤 실패했다면)

- **도입 비용**: 팀이 DS를 재작성하지 않는다. 흡수(Figma/레지스트리)가 얕으면 아무도 갈아타지 않음 → 흡수 깊이가 사활.
- **3-페르소나 함정**: 각 역할이 이미 강한 도구(Figma/IDE/Jira·Notion)에 묶여 있어, "지원 레이어"가 어느 쪽도 완전히 흡수 못 하고 얇게 걸침.
- **기획 surface의 모호함**: 가장 큰 제품 리스크. "정책 뷰"가 실제 JTBD가 아닐 수 있음 → 초기엔 디자이너↔에이전트 쐐기부터.
- **Incumbent**: Figma가 Dev Mode MCP·Make·Code Connect로 이미 이 격차를 좁히는 중 — 시간 싸움.
- **단일 실패점**: Astryx의 SPOF(원본 부실 시 전 소비자 부실)가 세 페르소나로 증폭.

## 다음 액션

1. **쐐기 확정**: 3-페르소나를 한 번에 노리지 말고 **디자이너↔에이전트**부터. 기존 DS(shadcn 레지스트리 또는 Figma Code Connect) 하나를 흡수 → (a) 실제 컴포넌트로 시안 조립, (b) 그 시안을 `search`/`get` MCP로 에이전트에 노출.
2. **핵심 가설 실험**: 흡수한 DS로 구성한 시안을 에이전트가 **승인 컴포넌트로 충실히 구현**하는지(어긋남 없이) 검증. 실패하면 "단일 원본→다중 소비자"의 사용성 이점이 없다.
3. **기획 JTBD 검증**: "정책 뷰"가 실제 고용되는 진전인지 인터뷰로 확인 — 아니면 디자이너+개발자 2-페르소나로 축소.
4. **Incumbent 해자 점검**: Figma Code Connect 커버리지·락인이 이 쐐기를 얼마나 잠식하는지, 흡수 깊이로 우회 가능한지.

## 열린 질문 / 내 관점

- **가장 불확실한 것**: 기획(planner) surface가 진짜 JTBD인가. 확신 낮음 — 실험 3이 이 아이디어의 3-페르소나 야심을 유지할지 2-페르소나로 접을지 가른다.
- **내 관점**: Astryx에서 베낄 것은 컴포넌트 카탈로그가 아니라 **"단일 원본 → 다중 소비자 + 에이전트가 읽는 조회형 MCP"라는 아키텍처**다. "사용성 높게 차용"의 핵심은 그 척추에 **디자이너와 기획이라는, Astryx가 비워둔 소비자**를 붙이고, 도입을 재작성이 아니라 **흡수**로 바꾸는 것. 시안이 실제 승인 컴포넌트로 구성되면 "디자인 시스템 반영"은 검증 대상이 아니라 구성의 부산물이 된다 — 이게 Figma "변혁적 경로"의 실사용 버전이다.
- **미검증 가정**: Figma/UXPin/Builder의 세부 기능·커버리지는 벤더 주장(1차지만 자기 유리)이다. 사업화 단계에선 실제 Code Connect 커버리지 한계와 레지스트리 흡수 가능 범위를 직접 검증해야 한다.
