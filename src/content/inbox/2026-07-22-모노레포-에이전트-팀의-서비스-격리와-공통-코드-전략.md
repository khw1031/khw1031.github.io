## TL;DR
- 같은 공간(모노레포)에서 서비스를 나누는 격리는 3층: **디렉토리 경계 + CODEOWNERS + import 경계 lint**(서비스 간), **git worktree + 짧은 브랜치**(작업 간), **merge queue + 기계 검증 게이트**(합류 지점).
- 공통 부분은 "모두의 것"인 shared 폴더가 아니라 **명시적 소유자와 공개 API(index.ts)를 가진 internal package**로 만들고, 소비자는 계약만 알게 한다.
- Human == Agent 가정이 성립하면 사람용 협업 규율(소유권·리뷰·계약)이 그대로 적용되고, 에이전트 쪽은 병렬성·볼륨 때문에 그 규율이 더 기계적으로 강제될 뿐이다.

## 격리 3층 구조
- **① 공간 구조 — 디렉토리 = 소유권 경계.** `apps/service-a`, `apps/service-b`, `packages/shared-*` 구조에서 디렉토리 자체가 code-ownership boundary가 되도록 설계하고, CODEOWNERS로 경로별 승인자를 리뷰 게이트로 강제한다. 에이전트가 팀원이면 "service-a 담당 에이전트"가 그 경로의 owner이고, 다른 서비스 경로를 건드리는 PR은 해당 경로 owner(사람이든 에이전트든)의 승인을 거친다.
- **② 작업 격리 — git worktree.** 각 에이전트/세션이 독립 working directory + index를 갖고 `.git` object store만 공유. 서로의 미커밋 변경을 절대 볼 수 없다. Claude Code 공식 지원: `claude --worktree <name>`, 서브에이전트 frontmatter `isolation: worktree`, `.worktreeinclude`로 `.env` 자동 복사. 실무 권장 동시 세션은 2–4개(리뷰 처리량·rate limit).
- **③ 통합 규율 — trunk-based + merge queue.** 브랜치가 오래 살수록 main과의 가정이 어긋나므로 수 시간 수명의 짧은 브랜치 + main 상시 릴리스 가능 상태 유지. 에이전트 볼륨용 merge queue 설계: 경로가 겹치지 않는 PR은 병렬 머지, 겹치는 PR은 자동 직렬화, 충돌은 그 코드를 쓴 에이전트에게 회송하고 사람은 에스컬레이션만 처리.

## 공통 코드 전략
- **shared 폴더 안티패턴** — "모두가 소유"하는 공유 폴더는 아무도 책임지지 않는 dumping ground가 된다.
- **internal package로 승격** — `packages/ui`, `packages/auth-client`처럼 이름·목적·공개 API(`index.ts`)가 있는 단위로 만들고, `enforce-module-boundaries` 같은 lint로 공개 API 외 내부 파일 import를 기계적으로 차단. 서비스는 계약만 알고 내부 구현은 모른 채 바뀔 수 있어야 한다.
- **소수 유지자 모델** — 도메인 모듈은 그 서비스 팀 전체 소유가 가능하지만, 공통 라이브러리는 소수의 지정 유지자가 낫다. 에이전트 팀으로 번역하면: 공통 패키지 전담 에이전트(또는 전담 태스크 큐)를 두거나, 누구나 PR을 올리되 공통 패키지 owner의 리뷰 게이트를 통과하는 Google OWNERS 스타일 기여자 모델.
- **모노레포 내부 단일 버전 + affected 검증** — 내부 패키지를 npm처럼 버전 발행하지 않고 소스 의존으로 두되, 공통 변경 시 의존 그래프 기반 selective test(affected)로 모든 소비자 서비스를 같은 PR에서 검증한다.
- **공통 변경은 직렬화 지점** — 공통 + 서비스 변경이 동시에 필요하면 공통을 먼저 머지하고 서비스 적용을 후속 태스크로 쪼갠다(stacked). 병렬 에이전트 관점에서 공통 패키지는 hotspot이므로 공통 변경 태스크는 병렬로 뿌리지 말고 순차 큐로 처리.

## Human == Agent가 바꾸는 것 / 안 바꾸는 것
- **안 바뀜**: 소유권(CODEOWNERS), 경계 lint, 리뷰 게이트, trunk-based 통합. 에이전트도 자기 worktree/branch에서 작업하고 merge queue를 통과한다.
- **강화됨**: ① 브랜치 수명 더 짧게 — 몇 시간 단위로 머지 가능한 크기로 태스크 스코프. ② "done" = 기계 검증 통과(eval 게이트). ③ 충돌은 작성 에이전트에게 자동 회송, 사람은 예외만.
- **주의** — worktree 격리는 *파일* 충돌만 막는다. 두 에이전트가 서로 모순되는 가정(같은 공통 계약을 다르게 변경)으로 작업하는 **의미 충돌**은 태스크 스코핑(spec에서 겹침 제거)과 공통 패키지 직렬화로만 막을 수 있다. 목표는 "격리 최대화"가 아니라 "짧은 격리 + 잦은 통합".

## 도구별 경계 강제 비교 (Nx vs Turborepo, 확인일 2026-07-22)
- **경계 강제의 성숙도가 갈린다.** Nx `@nx/enforce-module-boundaries`는 초기부터 있던 성숙한 태그 기반 ESLint 규칙(앱→앱 금지, feature→feature 금지, 공개 API 외 deep import 차단, 다국어 conformance 규칙까지). Turborepo `Boundaries`는 2.4(2025-04) 도입, **오늘 공식 문서 기준 여전히 experimental** — 기본은 "패키지 디렉토리 밖 파일 import 금지 + 미선언 의존성 import 금지", 태그(dependencies/dependents allow/deny, 전이 적용)도 되지만 실험적. 그래서 노트의 "공개 API 외 내부 import 차단"은 Nx면 기본기, Turborepo면 experimental Boundaries + `dependency-cruiser`/`eslint-plugin-*` 보조가 현실적.
- **오늘자 상태.** Turborepo 2.10.5(≈2026-07-14), 코어 Rust 이식 진행, **2026-03-25 릴리스에서 Git worktrees·Agent Skill 공식 지원**(이 노트의 worktree 격리 주제와 직접 맞물림). Nx 23.0.1(≈2026-07-19), **Rust 코어 이식 완료**(그 위에 새 Terminal UI 등), **22.5에서 Nx Agent Skills**, 22.7에서 Self-Healing CI(GitHub/GitLab/Bitbucket/Azure). → 2026년 두 도구의 공통 흐름은 "에이전트가 워크스페이스에서 일하는 법"을 1차 기능화.
- **주체/트레이드오프.** Nx = Nrwl(현 Nx사) 소유 OSS + 유료 Nx Cloud, `@nrwl`→`@nx` 스코프 이전 완료. Turborepo = Vercel 소유. 둘 다 벤더 중립이 아니라 특정 회사 상업 전략에 로드맵이 묶임.
- **폴리글롯(Python+React) 판단.** Turborepo는 **JS/TS 전용** → Python은 후보 아님. Nx는 폴리글롯 지향이나 **Python은 공식이 아닌 커뮤니티 플러그인 `@nxlv/python`**(개인 유지, 구 공식 파이썬 플러그인은 아카이브)에 의존 → 지속성 리스크 감안. 두 언어가 느슨하게만 공존하면 Nx의 학습·설정 비용이 과할 수 있어 경량 조합이 낫다.

## 경량 조합 (Nx 없이)의 boundary — 4평면
pnpm workspace(JS) + uv/Poetry(Python) + 루트 오케스트레이션. boundary는 한 도구가 아니라 **네 평면**으로 나눠 걸고, 어떤 도구도 전 평면을 넘지 못한다(Nx가 감추던 사실).
- **JS/TS 내부** — ① pnpm non-hoisted 구조가 "미선언 의존성 import"를 공짜로 차단 ② `package.json`의 `exports`로 공개 API만 노출(deep import 차단, `index.ts` 규칙의 OSS 네이티브) ③ 태그 방향 규칙은 **Sheriff(`@softarc/sheriff`)** 또는 **dependency-cruiser**로(= `enforce-module-boundaries` 도구 독립 대체재).
- **Python 내부** — **import-linter**(Seddonym): layers(상위→하위만)·forbidden·independence 계약을 CI에서 강제, containers로 반복 패턴 적용. Python 쪽 `enforce-module-boundaries` 등가물.
- **Python ↔ React 사이** — import 린트가 성립 안 함(JS가 Python을 import 못 함). 실체는 **런타임 API 계약**: 서버가 OpenAPI 내보내고 클라 타입을 거기서 생성 → codegen + typecheck를 CI 게이트로("생성 타입 최신인가"). 서버 내부 이름이 클라 코드에 등장하면 그 자체가 위반 신호.
- **소유권(횡단)** — CODEOWNERS 경로별 승인자(도구 무관, 항상 유효).
- **CI 배선(affected 대체)** — JS는 `pnpm --filter '...[origin/main]'`, Python은 경로 필터. 작으면 전부 돌리고, 느려질 때 도입.

## 2단 boundary 설계 (`web/` + `api/` 실제 구조)
경계가 2단이다. Level A(앱 간)와 Level B(앱 내부)를 섞지 않는다.
- **Level A — 앱 간 (`web/` ↔ `api/`)**: import 경계가 아니라 API 계약(OpenAPI + 타입 생성). `web/`은 `api/`의 `core`/`services`를 절대 모르고 생성된 타입만 안다.
- **Level B — `api/` 내부**: **바깥은 헥사고날, 안은 vertical slice**로 확정.
  - 방향(아래로만): `routes/app` → `services/*`(기능 슬라이스) → `llm/`(**경계층으로 확정** — provider 지식을 가두는 어댑터, `core`만 알고 소비자를 역참조 안 함) → `core/`(도메인 primitive, 아무도 위로 import 안 함).
  - import-linter 계약 2개: **layers**(`routes`>`services`>`llm`>`core`) + **independence**(`services/*` 서로 직접 import 금지 → 공유는 `core`로 내림).
- **Level B — `web/` 내부**: **Feature-Sliced Design(FSD) + Steiger 린터** 채택. FSD가 "상위 층 import 금지 + 같은 층 slice 간 cross-import 금지 + 모든 slice 공개 API 필수"를 표준화하고 Steiger가 검사 → Sheriff 수동 배선 대신 규칙+도구를 한 번에.
- **`service`는 양쪽에서 다른 뜻**(React=API/기능 슬라이스, Python=비즈니스 로직 레이어) → 각 앱에서 "레이어인지 슬라이스인지" 먼저 못박아야 규칙이 명확.

## 에이전트 워크플로우 적합성 + 아키텍처 대안
- **왜 맞나.** agent-friendly 코드베이스 요건(locality·small blast radius·boundary integrity·공개 API·ownership-aligned)이 이 설계와 그대로 일치. 기계 검증 경계(import-linter/Steiger) = "done=eval 게이트", 계약만 로드 = 컨텍스트 절감, 경로 소유권 = 병렬 worktree 태스크 스코프.
- **한 곳 조정(순수 수평 레이어의 약점).** 수평 레이어링은 기능 하나가 `core`+`services`+`routes`를 다 건드려 **feature blast radius가 큼** → 병렬 에이전트에 불리. 그래서 `services/`를 기술 레이어가 아니라 **기능 수직 슬라이스**로 두어 "태스크=슬라이스=소유 경계 하나"로 만든다(위 확정 반영).
- **아키텍처 좌표.** Layered/N-tier(기본형·blast radius 큼) → **Hexagonal/Ports&Adapters**(`llm` 경계층 시점에 이미 진입) → **Vertical Slice**(`services`에 적용) → **FSD**(`web` 정답, Steiger로 강제) → **Modular Monolith/Bounded Context**(더 커지면 진화) → Event-driven/CQRS(현재 과함).
- **과설계 경고.** 최소만: api는 `core`/`llm` 경계 + services 슬라이스 독립, web은 FSD 얕게. Bounded Context·CQRS·전면 port 추상화는 규모가 요구할 때. 헥사고날도 `llm`처럼 **변동성 높은 경계만** 어댑터로(원칙: 필요 증명 전 선구축 금지).
