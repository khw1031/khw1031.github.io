## TL;DR
- Idea → PRD → Harness → Spec → Plan & Tasks → workspace & worktree → 구현이라는 선형 리스트는 골격은 맞지만 세 가지가 어긋나 있다: ① harness·workspace는 "단계"가 아니라 기반층, ② worktree는 구현 루프 내부 요소, ③ eval 게이트와 역방향 엣지가 빠져 있다.

## 큰 그림
```
Idea ─→ (PRD) ─→ Spec ─→ Plan & Tasks ─→ [구현 루프] ─→ Review/Eval ─→ Release
                  ↑                        태스크별:            │
                  │                        worktree→구현→검증    │
                  └──── 발견된 요구·실패·피드백 ←────────────────┘

workspace + harness = 위 전체가 올라가는 기반(1회 설치)
```

## 핵심
- **Harness는 단계가 아니라 런타임이다.** normalpowers 같은 하네스는 Spec → Plan → Tasks → 구현 → eval을 실행하고 기록하는 기반이지, 파이프라인 중간의 한 공정이 아니다. 순서상 "Spec 전에 하네스 설치"는 맞지만(locked spec이 하네스 안에 등록되므로), 레포에 1회 설치되는 인프라로 분리해서 봐야 한다.
- **workspace & worktree는 위치가 다르다.** workspace(레포 스캐폴드·환경)는 하네스가 설치될 대상이므로 하네스보다 먼저다. worktree는 별도 단계가 아니라 구현 루프 내부의 태스크별 반복 요소 — 태스크 하나마다 worktree 할당 → 구현 → 검증 → 머지 → 정리가 돈다.
- **PRD와 Spec의 경계를 의식적으로 정해야 한다.** PRD는 왜/누구/성공 기준(제품 언어), Spec은 정확한 동작과 수용 기준(검증 가능한 계약). 1인 + 에이전트 워크플로우에서는 idea 노트(근거·방법론 포함)가 PRD 역할을 흡수하는 경우가 많다. 기본값을 "PRD 생략하고 idea → Spec 직행" 또는 "PRD를 Spec의 서문 섹션으로 흡수" 중 하나로 정해 이중 갱신 비용을 없앤다.
- **워크플로우를 규정하는 것은 게이트와 역방향 엣지다.** 순방향 노드만으로는 루프/그래프가 아니다:
  - eval 게이트 — "done"은 기계 검증(테스트/lint/visual approval) 통과로만 성립 (normalpowers의 eval-passed 이벤트).
  - 리뷰 — 코드 리뷰 + spec 대비 conformance 체크.
  - 역방향 엣지 — 구현 중 발견된 요구사항 → Spec 개정 또는 discovered-from 태스크 등록 / eval 실패 → 태스크 재진입 / 사용 피드백 → idea·PRD 회귀.
  - (선택) 배포/릴리스 — "실제 서비스"까지라면 릴리스와 운영 관측이 종착 노드.
