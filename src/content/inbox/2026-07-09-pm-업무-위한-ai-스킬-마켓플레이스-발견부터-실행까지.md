## TL;DR
- PM Skills Marketplace는 68개 스킬과 42개 명령을 9개 플러그인으로 묶어, AI 어시스턴트가 제품 관리의 전 과정(발견→전략→실행→출시→성장)을 구조화된 프레임워크로 안내하도록 만든 오픈소스 도구다.

## 큰 그림
```
PM Skills Marketplace (전체 구조)
│
├─ Plugin (설치 단위, 9개 도메인)
│   ├─ pm-product-discovery    (13 스킬, 5 명령)
│   ├─ pm-product-strategy     (12 스킬, 5 명령)
│   ├─ pm-execution            (16 스킬, 11 명령)
│   ├─ pm-market-research      (7 스킬, 3 명령)
│   ├─ pm-data-analytics       (3 스킬, 3 명령)
│   ├─ pm-go-to-market         (6 스킬, 3 명령)
│   ├─ pm-marketing-growth     (5 스킬, 2 명령)
│   ├─ pm-toolkit              (4 스킬, 5 명령)
│   └─ pm-ai-shipping          (2 스킬, 5 명령)
│
├─ Command (사용자 트리거, /slash-command)
│   └─ 여러 Skill을 체이닝하여 end-to-end 워크플로우 실행
│       예: /discover → brainstorm → identify → prioritize → experiment
│
└─ Skill (기본 빌딩 블록)
    └─ 단일 PM 태스크에 대한 도메인 지식 + 프레임워크 + 가이드
        예: opportunity-solution-tree, prioritization-frameworks
```

## 핵심
- **Skill은 지식의 원자 단위**다. 각 Skill은 Teresa Torres의 Opportunity Solution Tree나 Alberto Savoia의 Pretotype 같은 검증된 PM 프레임워크를 AI가 이해할 수 있는 구조로 인코딩한다. 사용자가 명시적으로 호출하지 않아도 대화 맥락에 따라 자동으로 로드되며, 필요시 `/plugin:skill` 형태로 강제 로드할 수 있다.

- **Command는 Skill을 엮어 워크플로우를 만든다**. 예를 들어 `/discover` 명령은 brainstorm-ideas → identify-assumptions → prioritize-assumptions → brainstorm-experiments라는 4단계 체인을 실행한다. 각 Command가 끝나면 관련 다음 Command를 제안하여 PM 업무의 자연스러운 흐름을 유지한다.

- **Plugin은 도메인별로 Skill과 Command를 패키징한다**. 9개 플러그인은 제품 수명주기의 각 단계(발견, 전략, 실행, 시장조사, 데이터 분석, 출시, 성장, 유틸리티, AI 코드 출시)를 담당하며, 한 번에 전체를 설치하거나 필요한 것만 선택 설치할 수 있다.

- **멀티 플랫폼 호환성을 추구한다**. Claude Code와 Cowork가 1차 대상이지만, Codex CLI, Gemini CLI, Cursor, OpenCode, Kiro에서도 Skill 파일을 복사해 사용할 수 있다(명령은 Claude 전용). 이는 PM 프레임워크의 보편적 가치를 플랫폼 종속 없이 제공하려는 설계 의도를 보여준다.

## 깊이
- **[Skill 체이닝의 힘]** `/discover` 같은 Command는 단순 자동화가 아니라 PM 사고 프로세스를 강제한다. 아이디어 브레인스토밍 → 가정 식별 → 우선순위 결정 → 실험 설계라는 순서는 Teresa Torres의 Continuous Discovery Habits에서 나온 구조로, AI가 "빠른 답변"이 아닌 "올바른 질문 순서"를 안내하게 만든다(저자 주장).

- **[AI Shipping Kit의 차별점]** pm-ai-shipping 플러그인은 "AI가 작성한 코드의 의도 문서화"라는 독특한 문제를 다룬다. AI 에이전트가 빠르게 코드를 생성하지만 의도(intent)에 대한 기록을 남기지 않는 문제를 해결하기 위해, `/ship-check` 명령은 아키텍처 문서화 → 보안/성능 감사 → 테스트 커버리지 매핑 → 출시 패킷 컴파일을 수행한다. 이는 "AI가 만든 코드"와 "검증된 코드" 사이의 간극을 메우려는 시도다(저자 주장, 검증 필요).

- **[프레임워크 선택의 폭]** 전략 플러그인만 해도 Lean Canvas, Business Model Canvas, Startup Canvas 세 가지를 제공하며, 우선순위 프레임워크는 Opportunity Score, ICE, RICE, MoSCoW, Kano 등 9가지를 참조용으로 포함한다. 이는 "하나의 정답"이 아니라 상황에 맞는 도구 선택을 지원하려는 설계다.

- **[의도 vs 구현 갭 분석]** `intended-vs-implemented` 스킬은 문서화된 의도와 실제 코드 동작 사이의 불일치를 찾는 방법론을 제공한다. 이는 전통적인 코드 리뷰가 놓치는 "문서와 코드의 괴리"를 체계적으로 발견하려는 시도로, 특히 AI 생성 코드에서 중요한 문제다(저자 주장).

## 용어 풀이
- **Skill** — AI 에이전트에게 특정 PM 업무에 대한 도메인 지식과 프레임워크를 제공하는 기본 단위 / 비유: 요리책의 개별 레시피 카드 / 비유가 깨지는 지점: 레시피는 사람이 읽고 실행하지만, Skill은 AI가 자동으로 로드하고 적용한다.

- **Command** — 사용자가 `/slash-command`로 호출하는 end-to-end 워크플로우 / 비유: 레스토랑의 코스 요리 (여러 요리가 순서대로 나옴) / 비유가 깨지는 지점: 코스 요리는 고정되어 있지만, Command는 상황에 따라 다른 Skill 조합을 사용할 수 있다.

- **Plugin** — 관련 Skill과 Command를 묶은 설치 가능한 패키지 / 비유: 스마트폰 앱 스토어의 앱 / 비유가 깨지는 지점: 앱은 독립적이지만, Plugin 간 Skill 공유가 가능하다(예: prioritization-frameworks는 여러 Command에서 사용).

- **Opportunity Solution Tree (OST)** — Teresa Torres가 제안한 시각적 프레임워크로, 목표(outcome) → 기회(opportunity) → 해결책(solution) → 실험(experiment)으로 이어지는 트리 구조 / 비유: 문제 해결을 위한 마인드맵 / 비유가 깨지는 지점: 마인드맵은 자유롭지만, OST는 엄격한 계층과 검증 단계를 요구한다.

- **Pretotype** — Alberto Savoia가 제안한 개념으로, 제품을 만들기 전에 최소 비용으로 시장 반응을 테스트하는 방법 / 비유: 영화 예고편 (본편 제작 전 관객 반응 확인) / 비유가 깨지는 지점: 예고편은 본편의 일부지만, Pretotype은 완전히 다른 형태일 수 있다(예: Wizard of Oz 테스트).

- **Vibe-coded app** — AI 에이전트가 빠르게 생성했지만 의도와 설계 결정이 문서화되지 않은 애플리케이션 / 비유: 속기사에게 받아쓴 회의록 (내용은 있지만 맥락과 의도가 빠짐) / 비유가 깨지는 지점: 속기사는 의도를 파악할 수 있지만, AI는 의도를 갖지 않는다.

```
/ discover [아이디어]
    │
    ├─ 1. brainstorm-ideas

    │      └─ 위험 가정 식별 (Value, Usability, Viability, Feasibility)

    │      └─ Impact × Risk 매트릭스로 우선순위 결정
    │
    └─ 4. brainstorm-experiments
           └─ 상위 가정을 검증할 실험 설계
```

## 핵심 시사점 / 판단
- **(저자 주장)** "Generic AI gives you text. PM Skills Marketplace gives you structure." — ==일반 AI는 텍스트를 생성하지만, 이 도구는 구조화된 프레임워크를 제공하여 더 나은 제품 의사결정을 이끌어낸다==는 주장. 이는 프롬프트 엔지니어링의 진화 형태로, "무엇을 물을 것인가"를 넘어 "어떤 순서로 사고할 것인가"를 가이드한다.

- **(저자 주장)** AI Shipping Kit는 "AI가 빠르게 코드를 작성하지만 의도를 기록하지 않는다"는 문제를 해결한다고 주장. 이는 현실적인 문제 인식이지만, 문서화의 자동화 품질과 유지보수 비용에 대한 검증이 필요하다.

- **(검증 필요·불확실)** 원문은 "68 PM skills and 42 chained workflows"를 주장하지만, 개별 Skill의 실제 효과성(예: 사용자가 더 나은 의사결정을 내리는가?)에 대한 실증 데이터는 제시되지 않았다. 프레임워크의 존재와 실제 활용 효과는 다른 문제다.

- **(사실)** MIT 라이선스 오픈소스이며, Claude Cowork, Claude Code CLI, Codex CLI에서 네이티브 설치를 지원한다. 다른 AI 도구(Gemini, Cursor 등)에서는 Skill 파일 복사 방식으로 제한적 호환성을 제공한다.

- **(검증 필요)** Windows 환경에서 Cowork VM 불안정 문제에 대한 workaround를 제공하지만, 이는 Anthropic 측의 공식 해결책이 아닌 커뮤니티 기반 임시 조치로 보인다(관련 이슈 #27010 참조).

## 확인 질문
- **Q1 (전이)**: 이 스킬들을 실제로 사용하는 PM 팀은 얼마나 되며, 어떤 조직에서 어떤 효과를 측정했는가? (원문에 없음 — 활용 사례와 효과 측정 데이터 부재)

- **Q2 (왜·어떻게)**: AI가 프레임워크를 "강제"하는 것이 인간의 비판적 사고를 약화시키지 않는가? 예를 들어, AI가 자동으로 가정을 식별해주면 PM이 스스로 질문하는 능력이 퇴화할 위험은 없는가?

- **Q3 (경계)**: 이 도구들이 다루지 않는 PM 업무 영역은 무엇인가? 예를 들어, 조직 내 정치적 역학관계, 팀 동기부여, 문화적 맥락 같은 "소프트 스킬" 영역은 의도적으로 배제된 것인가, 아니면 한계인가?
