## TL;DR
- LLM 서비스 도메인은 표면적으로 **RAG·에이전트·프롬프트·특정 모델/프레임워크**를 다루지만, 그 아래 변하지 않는 기반은 ==**"유한한 파라미터·문맥·자원 위에서, 조건부 확률 분포를 데이터로 학습하고, 외부 메모리·도구로 확장하며, 측정 신호로 최적화한다"**==는 오래된 원리들이다.
- 판정은 binary(불변/트렌드)가 아니라 **3단계 내구성 등급**으로 한다: ==T1(수학·물리적 불변) / T2(오래 검증됐으나 반증 가능한 공학 원리) / T3(현재 지배적 트렌드)==.
- 불확실하면 **상향이 아니라 하향**(T2 + 반증조건)한다 — "영원불변" 선언은 T1(수학·물리)에만 국한.

> `bedrock` 스킬로 "LLM Service" 도메인을 스코프로 분석. 특정 문서가 아니라 도메인 키워드 입력이었으므로, T3 표면 목록은 이 분야에서 통용되는 일반적 트렌드에서 도출.

## 주제 추출 (도메인 표면 = T3 후보)

모델 아키텍처(Transformer/attention) · 프롬프트 엔지니어링 · RAG(임베딩·벡터 검색) · 에이전트/tool-calling · 파인튜닝·RLHF·정렬 · 서빙 인프라(스트리밍·배칭·KV 캐시·양자화) · 컨텍스트 윈도우 관리 · 평가(eval·LLM-as-judge) · 프레임워크(LangChain·LlamaIndex·vLLM)

## 기반 원리 맵

| 원리 | 등급 | 왜 이 등급인가 / 반증 조건 |
|---|---|---|
| 선형대수·확률·최적화(경사하강) | **T1** | 기술 무관 substrate. LLM은 결국 행렬연산 + gradient 최적화. 반증 대상 아님 |
| 조건부 확률 — 출력은 `P(다음 토큰 \| 문맥)` | **T1** | 프롬프팅·컨텍스트의 수학적 정체. Bayes/조건부 분포 |
| 계산 복잡도 — attention은 문맥에 대해 O(n²) | **T1** | 문맥이 왜 희소 자원인가의 근거. 알고리즘이 개선돼도 "비용은 문맥에 따라 증가"는 유지 |
| 정보이론 — 유한 파라미터 = 유한 정보 | **T1** | RAG·외부 메모리가 왜 필요한가의 뿌리 |
| 표현 학습 > 수작업 피처 (데이터가 충분할 때) | **T2** | 반증: 소량 데이터·형식검증 도메인에선 기호/수작업이 이김 (실제로 그런 영역 존재) |
| 파라미터 메모리 ↔ 외부 검색 메모리 분리 | **T2** | 메모리 계층·IR의 재현. 반증: 무한·무비용 컨텍스트가 오면 분리가 무의미해짐 |
| 분포 의미론 — 의미 ≈ 벡터 공간의 기하적 근접 | **T2** | Firth(1957). 반증: 기하적 근접이 의미 유사도와 체계적으로 어긋나는 표현법이 우세해질 때 |
| 처리량·지연·비용·품질은 상충 (공짜 없음) | **T2** | 큐잉이론·Amdahl. 반증: 네 축을 동시에 개선하는 방법이 등장하면 (지금껏 없음) |
| 전이학습 — 범용 사전학습 후 목표에 적응 | **T2** | 반증: scratch 학습이 파인튜닝을 압도하는 체제로 회귀 |
| 측정 없이 개선 없음 + Goodhart 경계 | **T2** | 통계적 추정(표본→참값). 반증 어려움이지만 지표 과최적화 리스크는 상수 |
| 에이전트 = 관찰→결정→행동 루프 | **T2** | 제어이론·MDP. 오래된 추상. 반증: 순차적 의사결정 구조 자체의 폐기 (사실상 T1 인접) |

## 트렌드 → 원리 매핑 (forward / backward)

- **Transformer/attention** → *(forward)* T1 최적화·선형대수 위의 표현학습(T2)의 현재 구현. *(backward)* SSM/Mamba 등으로 교체돼도 "데이터로 표현을 대규모 학습"은 남는다.
- **프롬프트 엔지니어링** → *(forward)* 조건부 분포(T1)를 문맥으로 조작하는 것. *(backward)* 특정 프롬프트 트릭은 모델 버전과 함께 사라지지만 "출력은 문맥의 조건부 분포"는 불변.
- **RAG / 임베딩 / 벡터 DB** → *(forward)* 파라미터-외부 메모리 분리(T2) + 분포 의미론(T2) + 최근접 이웃(T1). *(backward)* RAG 패턴은 장문맥·메모리 아키텍처에 흡수될 수 있으나 "유한 작업 문맥으로 관련성 순 외부 지식을 가져온다"(= 수십 년 된 IR·캐싱)는 유지.
- **에이전트/tool-calling** → *(forward)* 관찰→결정→행동 루프(T2) + 유한 추론자가 도구로 오프로드. *(backward)* 프레임워크는 빠르게 죽지만 에이전트 추상은 남는다.
- **파인튜닝/RLHF** → *(forward)* 전이학습(T2) + 보상/선호 신호로의 최적화(T1). *(backward)* RLHF는 이미 DPO 등으로 교체 중 — "선호를 최적화 신호로 정렬"은 유지.
- **서빙(스트리밍·배칭·KV 캐시·양자화)** → *(forward)* 고전 시스템 공학: 캐싱=메모이제이션, 배칭=처리량·지연 트레이드, 양자화=정밀도·비용 트레이드(모두 T1/T2). *(backward)* 기법은 바뀌어도 상충 구조는 상수.
- **컨텍스트 윈도우 관리** → *(forward)* 유한 작업 메모리 → 우선순위화(T1, 페이징/캐싱 재현). *(backward)* 윈도우가 커져도 "문맥은 희소·우선순위 자원"은 유지.
- **프레임워크(LangChain 등)** → *(forward)* 위 개념들의 접착제. *(backward)* 라이브러리는 빠르게 사라지고 개념(검색·tool-calling·템플릿)만 남는다 — 여기에 자기 코드를 결합하지 마라.

## 유행 오인 경보 ⚠️

이 도메인에서 트렌드를 원리처럼 말하는 흔한 지점 — bedrock이 막으려는 바로 그 오류:

1. **"RAG가 LLM에 지식을 주는 방법이다"** — RAG는 *하나의 구현*. 불변은 "외부 메모리 + 검색"이고, RAG 패턴 자체는 이미 장문맥 압력을 받는 중. RAG를 영구 기반으로 취급 = 가짜 불변.
2. **"프롬프트 엔지니어링은 지속 가능한 스킬"** — 모델 버전 의존성이 극도로 높음. 불변은 트릭이 아니라 조건부 분포 조작.
3. **"에이전트가 미래다"** — 에이전트 *추상*은 수십 년 됐고 durable하지만, 지금의 에이전트-프레임워크 붐은 T3. 둘을 뒤섞으면 오인.
4. **특정 모델/프로바이더를 영구 기반으로 가정** — 모델 세대는 가장 빨리 바뀌는 T3.
5. **스케일링 법칙을 T1으로 격상** — 경험적(Kaplan→Chinchilla로 이미 수정됨) → T2, 반증 가능. "영원한 법칙"이라 말하면 과잉 확신.
