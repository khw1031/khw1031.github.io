## TL;DR
- 생성형 UI의 핵심은 AI가 제로에서 UI 코드를 짜는 것이 아니라, 디자이너가 미리 정의한 `elastic primitives`를 규칙에 따라 조합·데이터 주입하는 `text-to-hydration`으로 전환해야 한다는 것이다.

## 핵심
생성형 UI가 주류로 진입하면서, UI는 고정된 사이트맵 대신 사용자의 의도에 따라 실시간으로 조립되는 유동적 작업 공간으로 변화한다. 그러나 AI에게 빈 캔버스에서 코드를 직접 생성시키는 `text-to-code` 방식은 30초 이상의 로딩과 깨진 레이아웃을 만들어내 UX를 파괴한다. 대안은 `text-to-hydration`이다—AI가 할 일은 제로에서 만드는 것이 아니라, 디자이너가 미리 정의한 `elastic primitives`(탄성 프리미티브) 중 적합한 것을 선택하고, 실시간 데이터를 채워 넣는 것이다.

이 전환은 디자이너의 업무 자체를 뒤집는다. 더 이상 수십 개의 정적 페이지 템플릿을 Figma에서 그리는 것이 아니라, 개별 컴포넌트(메트릭 카드, 데이터 테이블, 오디오 플레이어 등)를 완벽하게 설계하고, 그 컴포넌트가 어떻게 늘어나고·줄어들고·다른 컴포넌트와 겹치지 않을지에 대한 **auto-layout 제약·반응형 행동·공간 가드레일**을 명시적으로 정의해야 한다. AI는 "취향"이나 "상식"이 없는 혼돈의 사용자이므로, 시각적 위계를 해치지 않도록 프로그램적 가드레일을 컴포넌트 API 스키마에 박아 넣어야만, 사람이 감시하지 않아도 일관된 UX가 유지된다.

## 깊이
- [탄성 프리미티브—`elastic primitives`]
  컴포넌트 하나하나가 고정 픽셀이 아니라, 문맥에 따라 크기·레이아웃·표현이 변하는 유연한 부품. 비유하자면 레고 블록—각 블록은 고정된 형태를 가지지만 조립 방식은 무궁무진하다. **비유가 깨지는 지점**: 레고는 물리적 결합만 가능하지만, 탄성 프리미티브는 데이터 흐름·상태 상속·전역 변수에 따라 시각·행동 모두 동적으로 변한다. 단순 크기 조절이 아닌 "시각적 압축률" 같은 고급 속성을 정의해야 한다.

- [`text-to-hydration` vs `text-to-code`]
  ```
  ┌────────────────────┬────────────────────────┬────────────────────────┐
  │ 차원               │ text-to-code           │ text-to-hydration      │
  ├────────────────────┼────────────────────────┼────────────────────────┤
  │ 출발점             │ 빈 캔버스              │ 기존 프리미티브 라이브러리│
  │ AI의 역할          │ 코드·CSS·데이터 구조 생성│ 선택·배치·데이터 주입   │
  │ 속도               │ 느림 (30초+ 로딩)       │ 밀리초 단위             │
  │ 안정성             │ 깨진 레이아웃, 모바일   │ 검증된 컴포넌트 재사용  │
  │                    │ 호환 문제 다발          │                         │
  └────────────────────┴────────────────────────┴────────────────────────┘
  ```
  저자의 **주장**: text-to-hydration이 생성형 UI의 유일한 실용적 경로다. **검증 필요**: Vercel AI SDK 등 일부 도구에서 유사 접근이 쓰이고 있으나, 산업 전반의 표준으로 정착했는지는 원문에 없음.

- [AI를 위한 기계 가독형 메타데이터 설계]
  디자이너가 Figma에 쓰던 "부드러운 문장" 설명을, AI가 파싱할 수 있는 구조화된 제약 조건으로 전환해야 한다. 예: "이 컨테이너는 시각적으로 X% 이상 압축되면 자동 접힘", "3개 이상의 데이터 스트림 비교 시에만 복잡한 분석 차트 표시". **비유**: 교통 신호등—신호등이 없어도 "적당히" 양보하는 운전자(개발자)는 있지만, AI는 신호등(명시적 규칙) 없으면 충돌한다.

- [`coarse-grained design`(거친 설계)와 코드 기반 설계]
  Figma에서 픽셀 하나를 다듬는 미세 설계가 아니라, AI 도구와 함께 시스템 전체를 조 단위(scale)로 다룬다. Builder나 Claude Code 같은 도구로 실제 코드 프리미티브를 실시간으로 테스트하며, 화면을 리사이즈하고 규칙을 대화로 수정하는 방식으로 "살아있는 Storybook" 환경에서 검증한다. **저자 주장**: Figma는 초기 컨셉에는 여전히 최적이나, 동적 데이터와 기계 가독형 제약을 모방하려면 너무 취약하다. **불확실**: Figma의 MCP 서버 등 새 기능이 이 한계를 실제로 극복하는지 장기 검증되지 않음.

- [Agent-Native Framework (Builder.io OSS)]
  인간 UI와 AI 에이전트가 단일 공유 데이터베이스 상태와 액션 모델 아래 통합된 아키텍처. 사람이 클릭하는 모든 컴포넌트가 에이전트가 실행할 수 있는 동일한 tool call에 바인딩되어 있어, AI가 느린 코드를 생성할 필요 없이 기존 프리미티브를 하이드레이션한다. **저자 주장**: 에이전트 세계의 성패는 LLM의 지능이 아니라 인간 중심 제품 설계에 달렸다. **불확실**: 오픈소스 초기 단계로, 실제 프로덕션 사례·확장성 검증은 원문에 없음.

## 용어 풀이
- **Generative UI (GenUI)** — 런타임에 AI 에이전트가 UI 일부를 동적으로 생성·선택·렌더링하는 디자인 패턴. / 비유: 맞춤형 재단사—사용자의 주문에 맞춰 매번 다른 옷을 짓는다. / 깨지는 지점: 재단사는 원단(프리미티브) 없이도 천을 재단할 수 있지만, GenUI는 정의된 컴포넌트 없으면 실패한다.
- **`text-to-hydration`** — 텍스트 프롬프트로 기존 UI 컴포넌트를 선택하고 데이터로 채워 넣는 방식. / 비유: 조립 가구—부품은 이미 있고, 설명서(프롬프트)대로 조립만 한다. / 깨지는 지점: 조립 가구는 고정 부품 수지만, 하이드레이션은 데이터 양에 따라 컴포넌트가 동적으로 분기·생략될 수 있다.
- **`elastic primitives`** — 탄성 프리미티브. 문맥에 따라 크기·레이아웃·표현이 변하는 설계 기본 단위. / 비유: 용수철—압축·신장해도 본질 형태를 잃지 않는다. / 깨지는 지점: 용수철은 물리 법칙만 따르지만, 프리미티브는 디자이너가 정의한 시각·기능 제약(가드레일)을 동시에 만족해야 한다.
- **`coarse-grained design`** — 거친 입자도 설계. 개별 픽셀 대신 시스템 전체를 덩어리 단위로 다루는 접근. / 비유: 도시 계획—건물 하나 창문 위치보다 블록 전체 용도 구역을 먼저 결정한다. / 깨지는 지점: 도시 계획은 수십 년 단위로 고정되지만, 거친 설계는 실시간 AI 피드백으로 분 단위에 수정될 수 있다.
- **Agent-Native Architecture** — 인간 인터페이스와 AI 에이전트가 동일한 상태·액션 모델을 공유하는 아키텍처. / 비유: 운전석과 부스위치—같은 핸들·페달을 조종하지만 역할이 다르다. / 깨지는 지점: 실제 운전에서는 물리적 컨트롤이 하나지만, 에이전트 네이티브에서는 UI 바인딩과 tool call이 논리적으로 동일할 뿐 구현 계층이 다를 수 있다.

## 핵심 시사점 / 판단
- **(저자 주장)** 생성형 UI의 성공은 LLM 성능이 아니라, 디자이너가 얼마나 정교한 탄성 프리미티브와 기계 가독형 규칙을 설계하느냐에 달려 있다.
- **(저자 주장)** Figma만으로는 동적 데이터와 AI 제약을 다루기 부족하므로, 디자이너는 코드 기반 도구로 직접 설계해야 한다.
- **(검증 필요·불확실)** text-to-hydration이 text-to-code를 완전히 대체할 산업 표준이 될지, 실제 대규모 프로덕션에서 성능·UX가 검증되었는지는 원문에 근거가 부족하다.
- **(검증 필요·불확실)** Agent-Native Framework의 실제 프로덕션 적용 사례, 확장성, 보안·상태 일관성 검증은 원문에 없음.
- **(사실)** Vercel AI SDK 등 기존 도구에서 AI에게 raw HTML 대신 미리 정의된 React 컴포넌트 중 선택 후 데이터 하이드레이션하도록 하는 접근은 존재한다.
