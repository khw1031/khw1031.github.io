## TL;DR
- Flint는 AI agent가 간단한 선언적 spec만으로 46종 차트를 자동 생성·렌더링하는 시각화 중간 언어다(저자 주장).

## 큰 그림

```
┌─────────────────────────────────────────────────────────────────┐
│                        Flint Architecture                        │
└─────────────────────────────────────────────────────────────────┘

[Input: Flint Spec]                    [Compiler]              [Output]
┌──────────────────┐                   ┌────────────┐          ┌──────────┐
│ • data           │                   │ semantic   │          │ Vega-Lite│
│ • semantic_types │ ──(추론/최적화)──▶ │ inference  │ ────────▶│ ECharts  │
│ • chart_spec     │                   │ + layout   │          │ Chart.js │
│   (chartType,    │                   │   engine   │          └──────────┘
│    encodings)    │                   └────────────┘
└──────────────────┘                          │
                                              ▼
                                    [자동 도출되는 것들]
                                    • scale, axis, parsing
                                    • color scheme, formatting
                                    • sizing, spacing, layout
```

## 핵심

Flint는 **"AI가 차트를 만들기 쉽게"** 하는 것에 초점을 맞춘 중간 언어(intermediate language)다(저자 주장). 기존 시각화 라이브러리(Vega-Lite, ECharts, Chart.js 등)는 scale, axis, spacing, layout 같은 저수준 파라미터를 직접 지정해야 해서 AI agent가 정확히 생성하기 어렵다(저자 주장). Flint는 데이터, 시맨틱 타입(의미 유형), 차트 타입, 인코딩만 선언하면 컴파일러가 나머지 저수준 설정을 자동 도출한다(사실 — 원문 스펙 예시 확인).

이 구조의 핵심은 **의미(semantic)에서 구현(implementation)으로의 분리**다. 예를 들어 `semantic_types`에서 `"period": "YearMonth"`라고 선언하면, 컴파일러가 시간 파싱 방식, 축 포맷, 도메인 범위를 자동으로 계산한다(사실 — 원문 Heatmap 예시). 사용자는 "언제부터 언제까지"를 지정할 필요가 없다.

## 깊이

- **[시맨틱 타입 → 저수준 추론]** Flint는 `Rank`, `YearMonth`, `Delta`, `Temperature`, `Profit` 등 의미 유형을 정의하고, 이를 바탕으로 파싱 방식·스케일 타입·축 포맷·색상표(d diverging color scheme의 중간값 포함)를 자동 결정한다(사실 — 원문 예시). 예를 들어 `Profit`(손익) 타입은 양수/음수를 구분하는 diverging 색상표(`redblue`)와 `domainMid: 0`을 자동 적용한다(사실 — Heatmap 컴파일 결과).

- **[탄성 레이아웃 모델]** 차트 수가 늘어나면 캔버스를 확장하고 band width를 줄여 밀집 버전으로도 캔버스에 맞춘다(저자 주장). 원문은 이를 "스프링이 확장 가능한 컨테이너에 settles하는 방식"에 비유한다(원문 비유). 22×3 grouped bar chart에서도 레이아웃이 깨지지 않는다는 것이 주장이다(검증 필요 — 실제 대규모 데이터에서의 동작은 원문에 benchmark 없음).

- **[차트 타입 전환의 용이성]** Faceted Bar Chart를 Pyramid Chart로 바꾸려면 `chartType`만 변경하면 되고, 컴파일러가 인코딩 선택을 저수준 설정으로 전파(cascade)한다(저자 주장). 이는 AI agent가 차트 디자인을 반복 시행착오(iterate)할 때 특히 유용하다는 주장이다(저자 주장 — 근거 제시 없음).

- **[멀티 백엔드]** Vega-Lite에 없는 sunburst 차트를 ECharts로 렌더링할 수 있다(사실 — 원문 예시). 46개 차트 타입 × 3개 백엔드 = 83개 예제가 gallery에 있다는 수치다(사실 — 원문 수치). 단, 백엔드별 커버리지 균등 여부는 **원문에 없음**.

## 용어 풀이

- **Intermediate Language (중간 언어)** — 소스 코드와 타깃 코드 사이의 추상적 표현 / 비유: "번역기가 이해하는 중간 노트" / 비유가 깨지는 지점: Flint는 컴파일 대상이 3개(Vega-Lite, ECharts, Chart.js)이므로 단일 타깃이 아닌 다중 타깃 중간 표현이다.

- **Semantic Types (의미 유형)** — 데이터 필드의 통계적 타입(숫자, 문자열)이 아닌 도메인 의미(순위, 온도, 손익) / 비유: "엑셀 셀 서식에서 '통화'를 선택하는 것" / 비유가 깨지는 지점: 엑셀은 표시 형식만 바꾸지만, Flint는 스케일·색상·레이아웃까지 추론한다.

- **Elastic Layout Model (탄성 레이아웃 모델)** — 차트 요소 수가 변할 때 스프링처럼 유연하게 크기와 간격을 조정하는 레이아웃 엔진 / 비유: "CSS Flexbox가 차트 내부에 적용된 것" / 비유가 깨지는 지점: Flexbox는 1차원/2차원 컨테이너 제이지만, Flint는 banking principles(시각 인지 원리)까지 반영한다고 주장한다(저자 주장 — 구체 알고리즘은 원문에 없음).

- **Banking Principles** — 시각화에서 데이터 패턴을 가장 잘 인지할 수 있도록 차트 종횡비를 설정하는 원리(Cleveland & Tufte의 시각 인지 연구 기반) / **원문에 구체 알고리즘 없음**.

## 시각 자료

| Flint Spec 단계 | 사용자가 지정 | 컴파일러가 자동 도출 |
|---|---|---|
| data | ✅ 원본 데이터 | — |
| semantic_types | ✅ 의미 유형 | → parsing, scale type |
| chart_spec.chartType | ✅ 차트 타입 | → mark type, layout |
| chart_spec.encodings | ✅ 필드-채널 매핑 | → axis config, domain |
| chart_spec.chartProperties | ✅ (선택) 색상표 등 | → scheme, domainMid |
| (미지정) | — | → size, spacing, step, width |

```
Flint Spec (단순)                    Compiled Spec (복잡)
─────────────────                    ──────────────────────
semantic_types: {                    encoding.x: {
  period: "YearMonth"        ──▶       type: "temporal",
}                                      scale: { nice: false,
                                                 domain: [...] }
                                     }
                                     mark: { type: "rect", width: 47 }
                                     height: { step: 29 }
```

## 핵심 시사점 / 판단

- **(저자 주장)** AI agent가 차트를 생성할 때 저수준 파라미터를 맞추는 것은 실패하기 쉬우므로, 의미 수준에서 선언하고 컴파일러가 채우는 방식이 더 신뢰할 수 있다.
- **(저자 주장)** 46개 차트 타입과 3개 백엔드 지원으로 실무 적용 범위가 넓다.
- **(검증 필요·불확실)** "AI agent가 reliably(안정적으로) 생성한다"는 주장의 정량적 근거(성공률, 오류율)는 원문에 없다.
- **(검증 필요·불확실)** 탄성 레이아웃 모델이 대규모 데이터(수천 행)나 복잡한 차트(faceting 다중 중첩)에서 어떻게 동작하는지 원문에 benchmark 없음.
- **(원문에 없음)** Flint spec 작성 자체를 AI agent가 얼마나 정확하게 하는지(즉, spec 생성 정확도)에 대한 평가는 없다. MCP server가 이 문제를 완화할 수 있으나 구체 메커니즘은 원문에 없음.

## 확인 질문

- **Q1 (전이)**: 이 "시맨틱 타입 → 저수준 추론" 패턴을 차트 외 영역(예: 대시보드 레이아웃, 리포트 자동 생성)에도 적용할 수 있을까? 어떤 추가 추론 엔진이 필요할까?
