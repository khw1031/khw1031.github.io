## TL;DR
- ==흡수(관찰→기술→복제) 가능한 것이라면 그건 애초에 moat가 아니다.== 패턴 추출은 남의 해자를 뺏는 게 아니라, 해자가 *어디에 진짜 있는지* 드러낼 뿐이다.
- LLM은 **명시 가능한** 패턴/로직/flow의 복제 비용을 사실상 0으로 만든다. 그래서 방어력은 "패턴"에서 "패턴이 올라탄 substrate(축적·유통·속도)"로 이동한다.
- 초안은 **2축**(명시가능성 × 재구성비용)이었으나, 전략경영·지식경영 문헌 검증 결과 두 개의 큰 구멍이 드러났다 → **3축 + 2플래그**로 개정.

## 큰 그림

```
  임의의 인공물(패턴 · 코드 · 아키텍처 · 비즈니스 모델)
                     │
        ┌────────────┴────────────┐
        ▼                         ▼
 [명시 가능 표면]           [관찰로 못 가져오는 기저]
  flow·구조·API·가격          축적 상태 · 인과 · 보완자산
        │                         │
   복제 비용 ≈ 0             복제 비용 高 or 불가 = MOAT
        │                         │
        └──── 껍질/기저 분리 엔진 ────┘
                     │
      ┌──────────────┼──────────────┐
      ▼              ▼              ▼
  [code 렌즈]   [architecture 렌즈]  [business 렌즈]
   잔차=상속리스크  잔차=숨은 "왜"      잔차=방어력
   verb=재구축     verb=재구축         verb=경쟁/방어
```

- **엔진 1개 + 렌즈 N개**: "명시/비명시 분리 → 축 배치"는 도메인 무관한 정보이론적 절차(재구성 비용 = 명시가능성의 함수). 도메인마다 바뀌는 건 *기저에 뭐가 들어가는지의 목록(residual taxonomy)*과 *목적(verb)*뿐.
- 렌즈는 **스킬 내부에서 관리**한다(사용자 결정). ESLint/clang-tidy가 core 엔진 + pluggable ruleset인 것과 동형.

## 핵심

- **출발 관찰**: Anthropic의 advisor tool 패턴은 본질이 2단 추론 flow다. flow만 정확히 기술하면 LLM으로 등가 복제가 된다. 하지만 복제해도 Anthropic은 아무것도 잃지 않는다 — advisor의 진짜 moat는 flow가 아니라 그 자리에 꽂히는 **Opus급 모델 품질 + 데이터·컴퓨트·유통**이기 때문. flow는 얇은 껍질.
- **핵심 명제(초안 A)**: "명시 가능하면 leak된다." → **문헌 검증 후 틀린 것으로 판명**. causal ambiguity와 complementary assets가 반례를 제공한다: ==다 보여주고도 못 베끼고(인과 불투명), 다 베끼고도 못 버는(보완자산 부재)== 경우가 실재한다.
- **개정 명제**: "관찰 가능 **+** 인과 투명 **+** 재구성 쌈 **+** 보완자산 게이트 없음 **+** 대체 불필요, 일 때만 leak."
- **엔진의 산출물 = moat map**: "복제했지만 여전히 못 가진 것"을 매번 강제로 적게 만드는 것이 핵심 가치. 껍질만 베끼고 이겼다는 착각을 차단.
- **명칭 주의**: 초안의 "Moat 추출"은 자기모순이었다. ==해자는 정의상 추출(흡수)되지 않는다== — 추출되는 건 껍질(패턴)이고, 해자에 대해선 *위치를 지도로 그릴(map)* 뿐이다. 그래서 이름을 **껍질/기저 분리(shell–substrate separation)**로 교정. 두 대상에 두 동사: 껍질→흡수, 기저→위치추적.

## 깊이

- **[핵심-가지 1] 초안 2축 모델 (검증 대상)**
  - 축1 **명시가능성**: 관찰로 기술 가능한가.
  - 축2 **재구성비용**: spec이 있어도 실제로 쌓는 비용(역사·튜닝·데이터).
  - Moat = 명시가능성 낮거나 재구성비용 높은 것. 무방비 = 명시 쉬움 + 재구성 쌈(좌상단).
  - 이 골격 자체는 문헌이 뒷받침한다(아래). 문제는 *충분히 세분화되지 않았다*는 것.

- **[핵심-가지 2] 개선점 ① — 보완자산은 '축'이 아니라 '조건부 곱셈항'** ★가장 큰 교정 (Teece 1986 전문 정독)
  - Teece(1986): **이윤 = f(전유가능성 체제 × 보완자산)**. 전유가능성 체제 = (a)기술의 성격(tacit/codified) × (b)법적 보호. tight면 혁신자가 **시간을 벌어** 보완자산을 확보 → 게이트 사실상 무력. **weak일 때만** 승부가 보완자산 접근 조건으로 넘어감.
  - ==따라서 보완자산은 단일 게이트(있다/없다)가 아니라, 전유가능성이 약할 때만 켜지는 **조건부 곱셈항**.== 초안이 이걸 독립 축으로 둔 게 과단순화.
  - 게이트의 하위 구조 4개(무시하면 오진): ①자산 유형(generic/specialized/cospecialized) ②의존 방향(uni/bi — cospecialized일수록 파트너의 can-but-won't가 will로 뒤집힘) ③접근 거버넌스(소유/계약/**유도** — IBM PC "induced contractual"은 소유 없이 게이트 통과) ④공급 대체성(competitive supply면 혁신자 유리, airtight monopoly면 자산소유자가 이윤 독식).
  - **누락 축 발견 — 지배적 설계(dominant design) 타이밍**: "preparadigmatic phase엔 complementary assets가 loom large하지 않는다." 지배적 설계 이전엔 게이트가 안 켜지고 승부는 *설계 선택*(de Havilland Comet은 보완자산이 아니라 설계 오판으로 죽음). → 게이트 활성화의 **전제 플래그**("이 시장은 paradigmatic인가") 필요.
  - **재구성비용은 절대값이 아니라 모방자 대비 상대값**("no point building a specialized asset if imitators can do it faster and cheaper").
  - Google "We Have No Moat" 메모("ecosystem이 해자")는 정확히 **weak appropriability → 보완자산 곱셈항 활성화**의 사례.

- **[핵심-가지 3] 개선점 ② — 관찰/인과 분리는 옳으나, 세 원천을 인과축에만 넣으면 category error** ★ (Reed & DeFillippi 1990 — 원문 미접근·2차 근사)
  - 분리 자체는 정당: R&D "having information ≠ understanding it"(p.94), Ryall의 월마트 예("관행은 다 보여도 무엇이 저비용을 만드는지는 불투명") → **관찰가능성 ≠ 인과투명성** 확증.
  - 그러나 R&D의 세 원천(tacitness·complexity·specificity)은 *축이 아니라 축에 점수를 넣는 입력변수*이고, **여러 축에 가중 분산**되며 서로 증폭(specificity↑→tacitness↑):
    - **tacitness** → 주로 재구성비용(알아도 재경험 필요)
    - **complexity** → 주로 인과투명성(어떤 조합이 원인인지 불명)
    - **specificity** → 주로 재구성/전개비용(전용자산 재배치 불가)
  - → 셋을 인과투명성 축에 몰아넣으면 "관찰·인과 다 투명한데 왜 못 베끼지"(tacit skill·전용 co-specialized 자산)를 놓침. ==**재구성비용 축은 반드시 별도 유지.**==
  - **causal ambiguity의 역설(필수 반영)**: 외부자가 원인을 모르면 *내부자도 완전힌 모른다* → 인과투명성 낮음은 해자인 동시에 **사내 이전·스케일·재투자의 취약성**을 예측. 인과투명성은 절대속성이 아니라 **관찰자 상대적(insider vs outsider)**이며 외부불투명↑ ↔ 내부이전성↓ 트레이드오프. (Ryall 2009: causal ambiguity는 우위 창출의 필요조건이나 지속의 충분조건은 아님.)
  - (계보) Lippman-Rumelt "Uncertain Imitability"(1982)가 원형, R&D가 세 원천을 부여, Barney(1991)가 inimitability 원천으로 채택. Rivkin(2000): 조합 상호작용이 크면 spec이 있어도 탐색이 intractable → complexity의 재구성비용 로딩 근거.

- **[핵심-가지 4] 개선점 ③ — 모방(imitation)만 보고 대체(substitution)를 놓침**
  - VRIN의 마지막 N = **Non-substitutable**. 복제엔 안 뚫려도 *다른 방식으로 같은 benefit을 내는 대체재*엔 뚫린다.
  - LLM 시대엔 이게 결정타: advisor 껍질을 복제당하는 것보다, **모델이 알아서 advisor 없이 잘해버리는 것**이 더 큰 위협.

- **[핵심-가지 5] 보너스 개선 — counter-positioning & verb 3종**
  - Helmer counter-positioning: 복제할 수 있지만 *기존 사업을 해쳐서 안 하는* 해자(incumbent 한정). 두 축은 모방자의 *능력*만 보는데 이건 *의지* 문제 → **"can-but-won't" 플래그**.
  - verb는 3개: **rebuild**(재구축) / **compete-against**(경쟁) / **defend-own**(내 해자 자가진단). VRIO·Teece 자체가 자가진단 도구.

## 개정 모델 v2 (원문 2편 정독 반영)

축을 늘리면 체크리스트 지옥이 되므로 **얇은 진단 전면 + 두꺼운 채점 백엔드** 2층으로 분리. 렌즈는 백엔드를 도메인별로 채운다.

**[진단 전면] 사용자가 실제 답하는 질문 (얇게)**
```
겉이 보이나?              관찰가능성
뭐가 원인인지 아나?        인과투명성 (누구 기준? insider/outsider)
알아도 다시 쌓기 비싼가?   재구성비용 (모방자 대비 상대값)
이 시장 지배적설계 섰나?   ← 보완자산 게이트 활성화 전제 (Teece)
누가 보완자산을 쥐었나?     weak일 때만: 유형·의존방향·접근거버넌스·공급대체성
우회 경로 있나?           대체가능성 (기술대체 vs 자산공급대체 — 반대방향)
복제 가능하나 안 하나?     can-but-won't (counter-positioning)
```

**[채점 백엔드] 입력변수 → 축 가중 로딩 (두껍게)**
```
입력변수      관찰가능성  인과투명성  재구성비용
 tacitness       중          중         主
 complexity      중          主         중
 specificity     고          약         主
        (연속척도 + 상호작용항, 단순합산 금지)

보완자산  = f(전유가능성) 조건부 곱셈항, dominant-design 전제 뒤에서만 활성
인과투명성 = 관찰자 상대적, 외부불투명↑ ↔ 내부이전성↓ 트레이드오프(역설)
재구성비용 = 자사 대비 모방자 상대우위
```

## 비유

- **빙산**: 물 위(명시 표면, flow·가격·API)는 누구나 스케치할 수 있다. 물 아래(인과·축적·보완자산)는 스케치해도 못 건진다. 초안은 "물 위를 그릴 수 있으면 배를 뺏은 것"이라 착각했다. — **깨지는 지점**: 빙산은 물 위/아래가 이분법이지만, 실제 moat는 "물 위인데도 못 건지는"(causal ambiguity) 회색지대가 있다.

## 곁가지

- **advisor 스킬이 이미 이 절차의 실증**: SKILL.md=절차 소유, triage 규칙=trigger·상한 소유. 흡수한 것 = flow, 못 흡수한 것 = Opus 백엔드 품질. → 개정 모델로 재-진단해볼 자기사례.
- **defend-own verb의 함의**: 분리 엔진을 남에게 겨누는 만큼, 내 것 중 "명시 가능한 껍질"에만 기대는 부분을 같은 도구로 점검해야 한다.
- **얄팍한 클리셰 함정**: 모든 걸 다 돌리면 결론이 매번 "결국 데이터+유통+속도"로 수렴해 사고정지. 판별력은 일반명제가 아니라 *도메인별 잔차 목록의 구체성*에서 나온다.

## 연결

- (내부) `advisor` 스킬 / advisor-triage 규칙 — 이 프레임워크의 씨앗이 된 사례.
- (내부) skill-manager — engine+lens 구조를 실제 스킬로 스캐폴드할 도구.
- (개념) 자원기반관점(RBV) ↔ Porter의 포지셔닝 관점: moat를 자원 내부에서 볼 것인가 산업구조에서 볼 것인가.

## 내 관점

- (초안 대비) 가장 값진 교정은 **"명시 가능 = 방어 불가"라는 등치를 깬 것.** causal ambiguity 하나만으로도 초안의 핵심 명제가 흔들린다 — 스킬을 짓기 *전에* 조사를 돌린 판단이 옳았음.
- (정독 검증 결론) 3축(관찰·인과·재구성)은 Teece의 appropriability를 **오히려 더 세밀하게 분해**해 잘 버텼다. 반면 **보완자산을 단일 게이트로 접은 것**(①)과 **세 원천을 인과축에 몰아넣은 것**(②)이 두 실질 오류였고, 둘 다 원문이 교정해줌.
- (핵심 긴장) 문헌은 정합적일수록 모델을 무겁게 만든다 — 다 넣으면 "얄팍한 클리셰"의 반대쪽 함정(체크리스트 지옥). → **얇은 진단 전면 + 두꺼운 채점 백엔드** 2층 설계(위 v2)로 해소.
- (남은 부채) R&D 원문 미접근 — 세 원천의 원저 정의 문장은 2차 근사. 스킬 스캐폴드와 병행해 AOM 원문 1부 확보 권장.
