## TL;DR
- AI 코딩 도구의 진짜 병목은 코딩 능력이 아니라 **요구사항 정의와 조직 맥락(Context) 축적**이며, 도구→프로세스→조직 문화까지 바꿔야 AX가 완성된다.

## 큰 그림
```
[AI 등장] Claude Opus 4.1 충격
    │
    ▼
[3가지 벤치마크]
 ├─① 개발자동화 (PR 180건 → 70% 일치)
 ├─② 레거시 마이그레이션 (기존 화면=완벽한 Spec)
 └─③ 신규 서비스 (Spec→PRD→구현 원스톱)
    │
    ▼
[전사 AX 선언] ── QuickWin 챌린지 (28건, 83% 절감)
    │
    ▼
[확산과 허들]
 ├─ Windows/cmd 장벽 → Antigravity IDE로 우회
 ├─ 비개발자 자동화 폭발 → Obsidian + 자동 sync 플러그인
 └─ 맥락 공유 필요 → Markdown 중심 Second Brain
    │
    ▼
[진짜 문제: 배포·인증·환경]
 └─ 사내 AI 플랫폼 KOCO 구축 (Node.js 내장, OneDrive/SAP 연동)
    │
    ▼
[배포 시스템 완성]
 ├─ EKS + GitLab CI/CD + 보안 스캔 + Slack 승인
 └─ Purple Access (리버스 프록시, 인증·권한·감사 로그)
    │
    ▼
[핵심 통찰] 맥락이 핵심 자산 → 데이터 맵·비정형 데이터 구조화
```

## 핵심
- 저자는 2025년 하반기 Claude Opus 4.1을 접한 뒤 AI가 '보조'에서 '메인'으로 전환됐다고 판단하고, 2026년 초 전사 AX를 선언했다(저자 주장).
- 세 가지 벤치마크(자동화·마이그레이션·신규 개발)에서 공통적으로 드러난 결론은 **"AI 실패의 원인은 AI가 아니라 부실한 요구사항 정의"**라는 점이다.
- 이 통찰은 전사 확산으로 이어졌지만, 비개발자에게 CLI 환경을 가르치는 현실적 허들에 부딪혔고, 이를 IDE 기반 도구(Antigravity)로 우회하며 해결했다.
- 확산이 커질수록 '만든 걸 어떻게 배포·공유하는가'가 다음 병목이 되었고, 이를 해결하기 위해 사내 AI 플랫폼 KOCO와 배포 파이프라인, 인증 프록시(Purple Access)를 구축했다.
- 최종적으로 저자가 도달한 결론은 **"AI 활용도의 상한은 조직이 축적한 맥락(Context)의 양과 품질"**이라는 것이다.

## 깊이
- **[벤치마크①] PR 자동화**: 최근 완료된 PR 180건 중 구현 가능 110건을 AI에게 Asana 태스크만 주고 재작성하게 했더니, 사람 코드와 평균 70% 일치. 100% 일치는 30건(27%), 90% 이상은 38건. 실패 원인의 대부분은 태스크 설명 부실(원문 근거). 이는 "코딩 자동화보다 Spec 품질 관리가 먼저"라는 주장의 근거가 된다.
- **[벤치마크②] 레거시 마이그레이션**: AI가 브라우저를 직접 띄워 CMS 화면·네트워크·컴포넌트를 분석하고 마이그레이션. "기존 페이지 소스코드와 화면 자체가 완벽한 기획서"가 되면 결과가 의도대로 나온다는 점이 핵심(저자 주장). 이후 대규모 마이그레이션 프로젝트를 기존 계획 대비 절반 이하의 리소스로 완료했다고 한다(검증 필요 — 내부 사례이므로 외부 재현 불가).
- **[배포 시스템]**: EKS 위에 GitLab CI/CD를 연결하고, CI 파이프라인 중간에 보안 스캔 단계 삽입 → Slack 알림 → 담당자 수동 승인. 인증은 Cloudflare Access에서 영감을 받은 자체 리버스 프록시 'Purple Access'를 Ingress ALB 뒤에 배치하여 앱 단위가 아닌 프록시 레이어에서 통합 처리(원문 근거).

## 용어 풀이
- **AX (AI Transformation)** — 조직 차원의 AI 전환 / 비유: DX(디지털 전환)의 AI 버전. 깨지는 지점: DX는 도구 도입에 가깝지만, 저자가 말하는 AX는 '맥락 축적과 업무 프로세스 재설계'까지 포함하므로 범위가 더 넓다.
- **Vibe Coding** — 자연어로 의도를 설명하면 AI가 코드를 작성하는 방식 / 비유: "지휘자처럼 지시만 하면 오케스트라가 연주". 깨지는 지점: 복잡한 비즈니스 로직이나 보안·성능 제약이 있는 경우 지시만으로는 부족하며 정확한 Spec이 필요하다.
- **Context (맥락)** — 의사결정 배경, 암묵지, 히스토리를 포함한 조직 지식 / 비유: AI에게 주는 '배경 스토리'. 깨지는 지점: 맥락을 문서로 남기는 것 자체가 비용이며, 오래된 맥락은 오히려 노이즈가 될 수 있다.

## 핵심 시사점 / 판단
- **(저자 주장)** AI 코딩의 병목은 코딩 능력이 아니라 요구사항 정의(Spec)와 조직 맥락이다.
- **(저자 주장)** 비개발자 확산의 핵심 허들은 CLI 환경이며, IDE 기반 도구로 우회 가능하다.
- **(저자 주장)** 맥락(Context)이 핵심 자산이므로 Markdown 중심의 지식 축적 프로세스가 필요하다.
- **(검증 필요·불확실)** "인당 생산성 200% 향상" 목표의 달성 여부는 원문에 수치 근거가 없다.
- **(검증 필요·불확실)** PR 70% 일치율이 실제 프로덕션 품질(버그, 유지보수성)로도 유효한지는 원문에서 다루지 않는다.
