## TL;DR
- Cerebras는 데이터를 원본 위치에 둔 채 하이브리드 검색(lexical + semantic + IDF + time decay)과 LLM 정규화를 결합해 일 15,000건의 내부 질문을 처리하는 지식 베이스를 구축했다.

## 핵심
- **데이터를 옮기지 말고 찾아가라**: Cerebras는 "단일 진실 공급원(single source of truth)"의 환상을 인정하고(저자 주장), Slack·GitHub·Jira 등 정보가 이미 존재하는 곳에서 직접 수집한다. 각 플랫폼은 해당 도메인에 최적화되어 있으므로, 억지로 한 곳에 모으는 대신 리더를 붙이는 방식이다.
- **단일 테이블이 모든 소스를 흡수한다**: 출처와 무관하게 모든 데이터는 동일한 Postgres embeddings 테이블에 `embedding + summary + metadata` 형태로 적재되며, 이 순간부터 동일한 쿼리 인터페이스로 검색된다. 이 설계가 커스텀 소스 플러그인을 단순하게 만든다.
- **Slack은 하이브리드 검색이 필수였다**: 벡터 유사도만으로는 "hey sure mike" 같은 잡음과 진짜 답변을 구분하지 못한다(저자 주장). Full-text(정확 토큰), Embedding(패러프레이즈), IDF(희귀 토큰 가중), Age Decay(신선도) 네 신호를 독립적으로 산출한 뒤 쿼리 시점에 융합한다.
- **쓰레드를 LLM으로 정규화하면 정확도가 오른다**: 원본 대화 전체를 임베딩하는 대신, LLM이 question·summary·resolution·systems·code_refs 구조로 추출한 뒤 이를 임베딩한다. 저자에 따르면 이 정규화 후 정확도가 "유의미하게" 상승했다(저자 주장, 수치 미공개).
- **Bursting으로 쓰레드 요약에 빠진 개별 메시지를 구제한다**: 동일 작성자의 연속 메시지를 하나의 burst로 묶고, 쓰레드 주제를 context로 붙여 별도 임베딩한다. IDF≥4.0, 200자 이상, 리액션 존재 등의 문턱을 넘은 burst만 저장되어 노이즈를 억제한다.
- **코드 임베딩은 언어 인식 재귀 청킹으로 관리한다**: CocoIndex를 사용해 class → method → block 순으로 분할하고, 커밋 시 변경된 청크만 재임베딩한다. "grep is all you need"라는 직관과 달리 Cursor의 사례를 참고해 도입을 결정했다(저자 주장).
- **RRF + Reranker로 최종 순위를 결정한다**: Reciprocal Rank Fusion(`Σ 1/(60+rank)`)으로 여러 검색기의 순위표를 하나로 합치고, 소수 reranker 모델이 0~10점 scoring하여 상위 10개를 남긴다. 이후 이웃 섹션을 붙여 맥락을 복원한다.
- **MCP는 오케스트레이션 없이 원시 도구만 노출한다**: `search_slack`, `search_code`, `who_knows` 등 검색 프리미티브를 LLM-free 도구로 제공하여, Claude Code 같은 외부 에이전트가 자체적으로 오케스트레이션하게 한다.

## 깊이
- **[Slack 정규화]** 쓰레드 원문을 그대로 임베딩하면 대화의 잡음이 벡터 공간을 오염시킨다. LLM이 question/summary/resolution으로 "증류(distillation)"하면, 검색 시 실제 엔지니어의 질문 어휘와 더 잘 일치한다(저자 주장). 이는 Anthropic의 Contextual Retrieval·XML Tags 프랙티스와 맥락을 공유한다(2차).
- **[RRF 융합]** smoothing 상수 k=60은 "한 검색기 1위"보다 "여러 검색기 상위권"을 우대한다. 예시에서 `CKPT_LOADER.CC`는 4개 리스트에 걸쳐 누적 0.0792점으로 단일 리스트 1위를 제쳤다. 이는 검색기 간 상호 보완을 수치로 구현한 것이다.
- **[MCP 설계 철학]** 도구에 LLM을 넣지 않으면 호출 비용이 낮고, 어떤 클라이언트든 재사용할 수 있다(저자 주장). 오케스트레이션 책임은 에이전트 측에 남는다 — 이는 도구-에이전트 분리 원칙의 실례다.

## 용어 풀이
- **Reciprocal Rank Fusion (RRF)** — 여러 순위표를 점수 하나로 합치는 공식 / 비유: 여러 심사위원의 순위 투표에서 "골고루 상위권인 후보"가 이김 / 비유가 깨지는 지점: 가중치와 k 값에 따라 결과가 민감하게 변하므로 "공정함"이 보장되진 않음.
- **Bursting** — 동일 작성자의 연속 메시지를 하나로 묶는 처리 / 비유: 대화 중 한 사람의 "발언권 독점 구간"을 하나의 단락으로 묶기 / 비유가 깨지는 지점: 여러 사람이 주고받는 핵심 논의는 burst로 포착되지 않을 수 있음.
- **IDF (Inverse Document Frequency)** — 전체 문서에서 드물게 등장하는 토큰에 높은 가중치 / 비유: 일상 단어보다 전문 용어가 검색에서 더 "의미 있는" 증거 / 비유가 깨지는 지점: 새로운 프로젝트의 고유 용어는 아직 corpus에 희귀하므로 과도하게 부스트될 수 있음.
- **Socket Mode** — Slack이 WebSocket으로 이벤트를 푸시하는 방식 / 비유: 우체국이 매일 편지를 가져다주는 것(폴링) 대신 실시간 등기 알림 / 비유가 깨지는 지점: 연결이 끊기면 놓친 이벤트를 별도 복구 로직으로 메워야 함.

## 시각 자료
| 검색 기법 | 잘 잡는 것 | 놓치는 것 |
|-----------|-----------|----------|
| Full-text | 에러 문자열, 플래그 이름, 호스트명 | 동의어·패러프레이즈 |
| Embedding | 다른 단어로 쓴 같은 의미 | 정확히 일치하는 토큰 |
| IDF | 희귀 설정값, 전문 용어 | "sounds good" 같은 상투어 |
| Age decay | 최신 답변 우선 | 오래됐지만 여전히 유효한 지식 |

## 핵심 시사점 / 판단
- (저자 주장) 하이브리드 검색이 단일 벡터 검색보다 실전에서 유의미하게 우수하다 — 수치적 근거는 미공개.
- (저자 주장) 쓰레드 원문보다 LLM 정규화 문서의 검색 정확도가 높았다 — 벤치마크 조건 불명.
- (검증 필요·불확실) "매일 15,000 질문" 규모에서 정확도·지연시간·비용 지표는 원문에 없음.
- (저자 주장) MCP에 LLM을 넣지 않으면 클라이언트 재사용성이 높아진다 — 이는 일반적 설계 원칙과 부합.
- (원문에 없음) 인증·인가·감사 계층의 구체적 구현(HRBAC, row-level security 등)은 상세 미공개.
