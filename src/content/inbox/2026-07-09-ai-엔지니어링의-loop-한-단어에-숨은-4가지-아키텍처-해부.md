## TL;DR
- ==AI 엔지니어링에서 "loop"은 실행·작업·제품·시스템 네 층위의 서로 다른 아키텍처를 뭉뚱그린 용어이며, 진짜 엔지니어링 질문은 "어느 루프의 자율성 다이얼을 어디까지 돌릴 것인가"다.==

## 큰 그림

```
┌─────────────────────────────────────────────────────────┐
│  ⑤ Oversight Loop (저자 명명)                          │
│     반복: 목표·예산·작업 선별 │ 종료: 없음(인간 판단)  │
├─────────────────────────────────────────────────────────┤
│  ④ System Loop — autoresearch                          │
│     반복: prompt/harness/eval 자체 │ 종료: eval 통과    │
├─────────────────────────────────────────────────────────┤
│  ③ Product Loop — software factory                     │
│     반복: 코드베이스+백로그 │ 종료: 외부 신호(이슈 등)  │
├─────────────────────────────────────────────────────────┤
│  ② Task Loop — Ralph Loop                              │
│     반복: 단일 산출물 │ 종료: spec 충족 + 테스트 통과   │
├─────────────────────────────────────────────────────────┤
│  ① Execution Loop — agent act-observe                  │
│     반복: 도구 호출-관찰 │ 종료: 환경 피드백/에이전트    │
└─────────────────────────────────────────────────────────┘
```

안쪽 루프는 구체적인 도구 호출부터, 바깥쪽 루프는 추상적 목표 설정까지 담당하며, ==각 루프의 "자율성 다이얼"은 서로 독립적으로 조절할 수 있다.==

## 핵심
- 2026년 6~7월 AI 엔지니어링 커뮤니티에서 "loop"이 유행어로 부상했으나, 저자(Aparna Dhinakaran) 주장에 따르면 ==같은 단어 아래 최소 네 가지 별개 아키텍처가混在하고 있다.==
- 가장 안쪽인 **실행 루프**는 에이전트가 도구 호출→결과 관찰→다음 행동 결정을 반복하는 고리로, 인간은 주로 경계(계획 승인·결과 리뷰)에서만 개입한다.
- 실행 루프가 "에이전트가 스스로 완료 선언"하면 끝나버리는 문제를 해결하기 위해 **작업 루프**(Ralph Loop)가 그 위를 감싼다. 매 반복마다 새 context window를 열어 spec을 처음부터 먹여 context rot을 막는 것이 핵심 설계다.
- 작업 루프가 단일 산출물을 반복한다면, **제품 루프**는 코드베이스 전체와 백로그를 대상으로 삼아 triage→spec→구현→리뷰→배포→모니터링의 전 라이프사이클을 자동화한다(저자 주장: Warp의 Oz, Anthropic 내부 Claude Tag가 사례).
- **시스템 루프**는 한 단계 더 올라가 prompt·harness·eval 자체를 개선 대상으로 삼는다("the loop is the product" — Roland Gavrilesecu 인용).
- 이 네 루프 위에 저자는 **oversight loop**을 명명하는데, 목표 설정·예산 배분·작업 중단 결정을 담당하며 ==인간이 반드시 머물러야 하는 고리==로 위치づ인다.

## 깊이

- **[① 실행 루프 — "끝났다고 말하는" 에이전트 문제]**
  - 에이전트가 스스로 done을 선언해도 실제로는 아닐 수 있어, 바깥 루프(②)가 "에이전트의 말을 믿지 않는" 검증자로 기능한다(저자 주장).

- **[② 작업 루프 — "낭비가 설계"인 이유]**
  - Geoffrey Huntley의 Ralph Loop는 매 반복마다 context를 완전히 새로 여는데, 겉보기엔 비효율적이지만 이것이 장시간 세션에서 발생하는 context rot과 compaction 이벤트를 방지한다는 주장이다.
  - 인간의 역할은 spec 작성 + 완료 판정 + **실패 패턴을 관찰해 재발하지 않도록 수정**(기관차 엔지니어 비유).

- **[③ 제품 루프 — 자율성 도입 경로]**
  - Warp는 저위험 저장소부터 시작해 자동 PR 머지율을 20%→60%로 점진적으로 올리는 경로를 dogfooding 중이라고 저자는 전한다(2차 정보).
  - Anthropic은 내부 제품팀 코드의 65%가 내부 Claude Tag로 생성된다고 발표(저자 인용) — ==검증 필요: 자체 발표 수치==.

- **[④ 시스템 루프 — "실존 증명" 양 끝단]**
  - 최소 사례: Karpathy의 autoresearch(2026.3), ~630줄 Python으로 하룻밤 50개 가설 실험.
  - 제품 사례: Meta Brain2Qwerty v2(2026.6) — 에이전트가 decoding architecture를 반복 수정해 word error rate 개선했으나, ==최종 학습 설정은 인간이 수동 선택==했다는 주의점이 붙는다.

- **[⑤ Oversight Loop — 논쟁의 정점]**
  - "오케스트라 다음에 팩토리를 만들라"(Roland Gavrilesecu) vs "이해를 위임하면 에이전트에게 대체된다"(Geoffrey Litt, Notion) vs "auto는 없다"(Paul Bakaus) — 저자는 이 논쟁을 ==누가 최상위 고리를 운영하는가의 문제==로 재해석한다.
  - Dex Horthy(HumanLayer)는 "Kubernetes의 제어 루프는 결정론적"이라며 열광이 엔지니어링을 앞서고 있다고 경고(저자 인용).
  - 흥미로운 데이터 포인트: Anthropic 내부에서도 Tag 운영팀이 **리뷰와 인간 이해 속도**에서 병목을 보고(저자 주장) — ==인간이 유지한 체크포인트가 이제 제약이 되는 역설==.

- **[Agentic MapReduce — 루프가 아닌 이유]**
  - Cognition의 Devin Security Swarm은 fan-out→aggregate 구조이며, 저자는 "피드백이 없는 루프는 그냥 for문"이라며 루프 분류에서 의도적으로 제외했다.

## 용어 풀이

- **Loop (이 글의 용법)** — 반복 실행 구조 일반. / 비유: 러시아 인형(안쪽 루프가 바깥 루프 안에 중첩). / ==깨지는 지점: 프로그래밍의 for/while과 혼동하면 "피드백 유무"를 놓친다. 이 글의 루프는 반드시 종료 신호(feedback)를 전제한다.==
- **Ralph Loop** — 같은 spec으로 에이전트를 매번 새 context에서 재실행하는 패턴. / 비유: 매번 깨끗한 도마로 요리 다시 시작. / 깨지는 지점: "랑은 낭비"라는 직관에 반하므로, context rot 개념을 모르면 비합리적으로 보인다.
- **Context rot** — 긴 세션에서 컨텍스트가 축적되며 품질이 조용히 저하되는 현상. / 비유: 오래 사용한 프라이팬에 달라붙은 잔여물. / 깨지는 지점: 정량적 정의가 명확하지 않아(원문에 기준 없음) 진단이 주관적일 수 있다.
- **Software factory** — 소프트웨어 개발 전 라이프사이클을 자율적으로 운영하는 시스템. / 비유: 제품 대신 코드를 찍어내는 공장. / 깨지는 지점: "공장" 비유는 인간 역할을 라인 작업자로 축소하는 뉘앙스를 풍겨, 반대진영(Geoffrey Litt 등)이 "우울한 비전"이라 비판.
- **Autoresearch** — 시스템이 자기 자신의 prompt/eval을 개선하는 루프. / 비유: 요리사가 자기 레시피를 매일 고쳐 쓰는 것. / 깨지는 지점: Meta 사례에서도 최종 선택은 인간 — 완전 자동과의 경계가 불분명.
- **Agentic MapReduce** — 여러 에이전트를 병렬로 fan-out해 결과를 집계하는 패턴. / 저자 주장: ==피드백이 없으므로 루프가 아닌 파이프라인==.

## 핵심 시사점 / 판단

- **(저자 주장)** ==루프의 교훈은 추상화 수준을 한 단계 올려 인간 판단을 스택 위쪽으로 미는 것이며, 이제 "어디로 올라갈지" 지도가 생겼다.==
- **(저자 주장)** 자율성은 네 루프 각각에서 독립적으로 조절 가능한 다이얼이므로, "전부 자동 vs 전무"라는 이분법은 잘못된 프레임이다.
- **(검증 필요·불확실)** Anthropic 내부 코드의 65%가 Claude Tag로 생성된다는 수치 — 기업 자체 발표이므로 독립 검증은 원문에 없음.
- **(검증 필요·불확실)** "루프" 유행이 실제 프로덕션 성과로 이어지는지, 컨퍼런스 마지막 날 논쟁에서조차 결론 미결(원문에 "what works in practice" 의문 제기).
- **(저자 주장)** Agentic MapReduce는 루프가 아니라 파이프라인 — 피드백 유무를 분류 기준으로 삼음.
- **(저자 주장)** 루프가 실제로 닫히는지 확인하려면 트랜스криプト 샘플 점검이 아닌, 트레이스 전수 스윕+실패 클러스터링이 필요하며 Arize AX가 그 도구로 언급됨 — ==특정 제품 언급이 있으므로 주의==.
