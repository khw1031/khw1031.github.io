## TL;DR
- GPT-5.6은 프롬프트를 줄일수록(41~66%) 점수가 10~15% 향상되며, 자율성 경계와 문체 지시를 정밀하게 설계하는 것이 핵심이다.

## 큰 그림

```
GPT-5.6 프롬프팅 베스트 프랙티스
│
├── 1. 프롬프트 최소화 ──────────────────────────────────────┐
│   ├── 누적된 레거시 지시 제거 → 모델 기본 동작으로 흡수됨 │
│   ├── 도구(Tool) 설명 단순화 → 선택 일관성 ↑              │
│   └── 예시·스타일 지시 → 필요 시에만 추가                 │
│                                                            │
├── 2. 자율성 & 권한 정의 ───────────────────────────────────┤
│   ├── 읽기/검증 → 자동 수행                               │
│   ├── 로컬 변경 → 확인 없이 수행                          │
│   └── 외부 쓰기/파괴적 작업 → 사용자 확인 필수            │
│                                                            │
└── 3. 문체 & 성격 제어 ─────────────────────────────────────┘
    ├── 기본 응답이 이미 짧아짐 (도입부↓, 추측 분기↓)
    ├── "간결하게" 지시 오히려 역효과 → 우선순위 지시 사용
    └── 친근함 → 구체적 행동 지시로 대체
```

## 핵심

1. **프롬프트는 적을수록 강해진다.** GPT-5.6은 이전 세대가 "기본적으로 수행하게 된" 행동까지 프롬프트로 적어주던 시대의 산물을 제거했을 때 토큰·비용이 크게 줄고 성능 점수는 오히려 올라간다(저자 주장: 내부 평가 기준 토큰 41~66%, 비용 33~67% 감소, 점수 10~15% 향상). 이유는 무거운 프롬프트가 모델에게 불필요한 탐색·반복 검증을 유도하고 누적 컨텍스트를 비대하게 만들기 때문이다.

2. **자율성의 경계를 3단계로 명시한다.** 모델이 "어디까지 혼자 해도 되는지"를 모르면 안전 모드(과도한 확인) 또는 오작동(무단 실행) 양쪽으로 흔들린다. 저자는 읽기 전용 → 로컬 변경 → 외부/파괴적 작업의 3단계 정책을 제안하며, 프롬프트 곳곳에 "먼저 물어보세요"를 반복하면 안전한 작업에서도 불필요한 확인 루프가 발생한다고 지적한다.

3. **문체 제어는 "방향"이 아니라 "우선순위"로 준다.** GPT-5.6은 이미 응답이 압축되어 있어서 "간결하게 답하세요" 같은 포괄 지시는 모델이 결과물 자체를 축소하는 방향으로 작동할 수 있다. 대신 "결론→근거→주의사항→다음 행동" 순으로 제시하고 부차적 세부사항만 줄이라고 지시해야 필요한 내용을 보존한 채 간결해진다.

## 깊이

- **[프롬프트 최소화 — 레거시 지시 제거]** 그동안 에이전트 시스템의 시스템 프롬프트에 누적되어 온 지시(예: "단계별로 생각하세요", "도구를 사용하기 전에 설명하세요" 등) 중 상당수는 GPT-5.6이 별도 지시 없이도 수행하는 기본 동작이 되었다. 따라서 **"모델이 자연스럽게 하지 못하는 행동"**에만 지시를 집중하는 것이 가장 큰 효과를 낸다(저자 주장).

- **[프롬프트 최소화 — 도구 구성]** 작업과 무관한 도구를 노출하거나 도구 설명을 장황하게 쓰면 시작 컨텍스트가 커지고 도구 선택 일관성이 떨어진다. 저자 주장에 따르면 평가에서 구체적 문제가 확인된 경우에만 도구·예시·지시를 추가하는 "점진적 확대"가 정답이다.

- **[자율성 — 3단계 정책 예시]**
  | 단계 | 허용 행위 | 확인 필요 |
  |------|-----------|-----------|
  | 읽기·진단 | 파일 읽기, 로그 확인, 검색, 비파괴적 테스트 | ❌ |
  | 로컬 변경 | 요청 범위 내 코드 수정, 로컬 검증 실행 | ❌ |
  | 외부·파괴적 | 외부 시스템 쓰기, 구매, 범위 확장 | ✅ |

- **[문체 — GPT-5.5 vs 5.6 비교 예시]** 고객 이탈 조사 요청 시, 데이터 미접근 상태에서 5.5는 가능한 데이터셋과 비즈니스 컨텍스트를 나열했지만, 5.6은 대시보드 내보내기나 테이블 접근 권한을 요청하고 최소 필드와 분석 계획을 제시했다(저자 주장). → 5.6이 더 **행동 지향적**이라는 근거로 사용됨.

- **[문체 — "간결하게"의 역설]** "최소한의 텍스트만 사용하세요"는 군더더기 제거를 넘어 모델의 **작업 우선순위 판단** 자체를 바꾼다. 결과적으로 완전한 결과물 대신 더 짧은 대체물을 선택하는 편이 낫다고 판단할 수 있다(저자 주장).

## 용어 풀이

- **시스템 프롬프트 (System Prompt)** — 모델의 역할·제약·행동 규칙을 미리 정의하는 숨겨진 지시문. 비유: 직원의 업무 매뉴얼. 깨지는 지점: 매뉴얼이 너무 길면 직원이 핵심 규칙을 놓치듯, 모델도 긴 시스템 프롬프트에서 우선순위를 잃을 수 있다.

- **컨텍스트 누적 (Accumulated Context)** — 대화 턴이 쌓이면서 프롬프트·이전 응답·도구 결과가 윈도우에 쌓이는 현상. 비유: 책상 위에 서류가 점점 쌓여 중요한 문서가 묻히는 것. 깨지는 지점: 책상 정리(컨텍스트 리셋)가 불가능한 장기 세션에서는 이 가이드만으로 해결되지 않는다.

- **에이전틱 (Agentic)** — 모델이 도구 호출·파일 수정 등 외부 행동을 연쇄적으로 수행하는 패턴. 비유: 지시만 받는 인턴이 아니라 스스로 판단해 행동하는 주니어 사원. 깨지는 지점: 사원의 판단력이 부족하면 자율성이 오작동으로 이어지므로, 3단계 권한 정책이 필수.

## 핵심 시사점 / 판단

- **(저자 주장)** GPT-5.6은 "적게 말할수록 잘하는" 모델이므로, 기존 프롬프트를 과감히 줄이는 리팩토링이 ROI가 높다.
- **(저자 주장)** 코딩 에이전트 분야에서 자율성 3단계 정책을 적용하면 확인 루프 감소와 안전성 확보를 동시에 달성할 수 있다.
- **(검증 필요·불확실)** "내부 평가에서 점수 10~15% 향상"이라는 수치의 평가 벤치마크·조건이 명시되지 않아 범용적 적용 가능성은 불확실하다.
- **(검증 필요·불확실)** 5.5 vs 5.6 문체 비교 예시는 저자의 해석이 개입된 2차 전달이며, 원본 OpenAI 문서의 정확한 문맥 확인이 필요하다.
- **(원문에 없음)** 프롬프트 최소화가 실패하는 경우(예: 도메인 특화 작업, 안전 크리티컬 환경)에 대한 논의는 포함되어 있지 않다.
