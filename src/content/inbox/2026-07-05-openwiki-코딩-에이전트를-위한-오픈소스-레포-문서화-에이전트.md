## TL;DR
- OpenWiki는 코드베이스 위키를 **자동 생성 → 에이전트 지시 파일에 참조 삽입 → GitHub Action으로 지속 갱신**하는 오픈소스 CLI/에이전트로, 코딩 에이전트가 최신 레포 문서를 스스로 찾아 쓰게 한다.

## 핵심
- 코딩 에이전트는 자신이 작업하는 레포의 **핵심 로직 위치, 파일 간 연결, 코드베이스 패턴**을 알 때 더 나은 코드를 작성한다(저자 주장). 그러나 문서는 초기 작성도, 코드 변경 후 갱신도 비용이 커서 대형 레포에서는 빠르게 낡아진다.
- OpenWiki는 이 갱신 문제를 자동화하기 위해 **위키 생성 → 지시 파일 참조 → 지속 갱신**의 3단계 파이프라인을 제공한다. 전체 문서를 `AGENTS.md` 같은 지시 파일에 우겨넣는 대신 **짧은 참조만 삽입**하고, 에이전트가 필요할 때 위키를 직접 찾아가게 설계한 점이 핵심이다.
- 이 설계는 "코딩 에이전트는 이미 지시 파일을 읽는다"는 전제 위에 서 있다. 따라서 추가 워크플로 변경 없이, 참조 삽입 하나만으로 에이전트가 위키를 발견할 수 있다고 저자는 주장한다.

## 깊이
- **[핵심-생성] `openwiki --init` 한 번으로 시작**: npm 전역 설치 후 init을 실행하면 모델 제공자와 API 키를 묻고, 레포를 스캔해 위키를 생성한다. 기본값은 OpenRouter의 오픈 모델이며, OpenAI·Anthropic·Fireworks·Baseten 등으로 전환 가능하다(원문 기준).
- **[핵심-연결] 지시 파일 참조 패턴**: `AGENTS.md`와 `CLAUDE.md`에 wiki 위치와 "언제 사용할지"를 설명하는 짧은 참조를 추가한다. 저자는 대형 레포에서 위키가 수백 파일에 이를 수 있어, 전체를 컨텍스트로 넣으면 **토큰 낭비와 유지보수 어려움**이 발생한다고 설명한다.
  - 비유: 지시 파일은 '목차', 위키는 '본문 책'. 에이전트는 목차에서 장 번호를 보고 필요한 페이지만 펼친다.
  - 비유가 깨지는 지점: 실제 에이전트가 목차를 정확히 해석하고 필요한 페이지만 검색할지는 모델의 지시 추종 능력과 검색 도구에 의존하므로 보장되지 않는다.
- **[핵심-갱신] GitHub Action 기반 증분 업데이트**: 스케줄(예: 1일 1회)로 실행되며, 마지막 실행 이후 커밋을 확인하고 **git diff**로 변경 사항을 파악한 뒤 위키의 관련 부분만 갱신한다.
- **[핵심-인프라] DeepAgents + LangSmith**: OpenWiki는 LangChain의 DeepAgents 위에 빌드되어 LangSmith API 키를 제공하면 에이전트가 문서 생성/갱신 과정에서 어떤 행동을 했는지 추적 가능하다.
- **[핵심-확장] 코드베이스 너머 적용 가능성**: 저자는 코드베이스 문서화가 "첫 번째 유스케이스"이며, 에이전트가 다양한 워크플로에서 **지속적 컨텍스트(durable context)**를 유지하는 패턴으로 확장 가능하다고 주장한다. 현재는 코드베이스에만 집중(원문 기준).

## 용어 풀이
- **AGENTS.md / CLAUDE.md** — 코딩 에이전트에게 레포별 지시를 전달하는 마크다운 파일 / 비유: 신입사원에게 건네는 '팀 온보딩 메모'. 깨지는 지점: 에이전트마다 인식하는 파일명이 다르고(CLAUDE.md는 Anthropic Claude Code 관례), 모든 에이전트가 이 파일을 반드시 읽는 것은 아니다.
- **DeepAgents** — LangChain의 에이전트 빌딩 블록 프레임워크 / 비유: 레고 블록 세트. 깨지는 지점: 레고는 조립이 자유롭지만, DeepAgents는 특정 API와 추상화 계층을 학습해야 하는 비용이 있다.
- **git diff** — 두 커밋 간 변경 사항을 텍스트 수준으로 비교하는 git 명령 / 비유: 두 판본의 '틀린 그림 찾기'. 깨지는 지점: 파일 이동·리네임·의미 변경은 텍스트 diff만으로는 놓칠 수 있다.
- **Durable context** — 에이전트가 세션 간에 유지·재사용할 수 있는 영속적 지식 / 비유: 책상 위에 펼쳐둔 참고서. 깨지는 지점: 책상은 공간이 제한되어 있어(=컨텍스트 윈도우), 너무 많은 참고서를 한 번에 올릴 수 없다.
- **LangSmith tracing** — 에이전트 실행 단계별 입력·출력·도구 호출을 기록하는 관측 도구 / 비유: 블랙박스 비행 기록. 깨지는 지점: 기록 자체가 토큰/비용을 추가로 발생시킬 수 있다(원문에 비용 언급 없음).

## 핵심 시사점 / 판단
- **(저자 주장)** 좋은 문서는 코딩 에이전트의 실수를 줄이고 코드 변경의 품질을 높인다. → 직관적으로 타당하나, "얼마나 줄이는가"에 대한 정량 근거는 원문에 없음.
- **(저자 주장)** 지시 파일에 전체 위키를 넣지 않고 참조만 넣는 방식이 토큰 효율과 유지보수 면에서 우월하다. → 컨텍스트 윈도우 관리 관점에서 합리적이나, 에이전트가 참조를 따라 wiki를 정확히 검색·사용하는지는 구현과 모델에 의존.
- **(검증 필요·불확실)** git diff 기반 증분 갱신이 의미적 변경(리팩터링, 아키텍처 변경)을 정확히 포착해 위키에 반영하는지는 원문에서 검증 데이터가 제시되지 않음.
- **(검증 필요·불확실)** "몇 분 안에 위키 생성"이라는 속도 주장이 대형 레포(수천 파일)에서도 성립하는지 불확실.
- **(원문에 없음)** 위키 페이지의 품질 평가 지표, 환각(hallucination) 방지 메커니즘, 멀티 언어 레포 지원 범위에 대한 언급 없음.
- **(관점)** DeepWiki, AutoWiki 등 기존 도구와의 **정량적 차별점**은 원문에서 제시되지 않으며, "오픈소스 + GitHub Action 자동 갱신 + 지시 파일 참조 패턴"이 주요 차별점으로 보인다.
