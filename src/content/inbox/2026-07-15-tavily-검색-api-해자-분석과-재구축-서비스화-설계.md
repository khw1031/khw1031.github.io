## TL;DR
- **(분석·판단)** Tavily(에이전트용 웹검색·추출 API)는 **대부분 껍질**이다. API 표면·SDK·검색→추출 흐름은 관찰 가능하고 교체비용이 낮아 며칠~수 주면 복제된다. 실제로 신뢰할 만한 대안이 이미 5~7개다.
- **(분석·판단)** 못 베끼는 **기저**는 좁다 — ① 에이전트 질의→소비 텔레메트리 플라이휠, ② Nebius발 원가·유통, ③ 개발자 디폴트 습관. 이 중 진짜 격리 메커니즘은 ①뿐이고, 그마저 "동일한 공개 목적함수에 최적화"로 기능적 동등이 가능해 **얕은 해자**다.
- **(분석·판단)** "동일 로직 구현"은 주말치 작업(껍질)이지만 "서비스화"란 곧 그 기저를 *직접 짓는 일*이라 여기에 진짜 비용이 있다. 범용 Tavily 클론은 원가·유통 열세로 자살행위이고, 이길 자리는 **좁은 버티컬 × 로컬라이제이션 × 구조화 출력**이다.

## 배경 — 어디서 나온 물음인가
LangChain `create_agent`(ReAct 루프) 실습에서 `TavilySearch`+`TavilyExtract`로 한국 주식 리서치 에이전트를 돌리다, 검색 결과가 죄다 영어권 매체로 쏠리는 현상을 봤다. 원인을 파다 보니 두 갈래 — ① 모델이 만든 질의어가 영어일 수 있음(로그로 확인 가능), ② `topic="news"`는 글로벌 매체 쏠림이 있고 이 토픽에선 `country` 부스트도 못 씀. 여기서 "Tavily는 오픈소스인가 / 이걸 그냥 우리가 만들면 안 되나"로 물음이 번졌고, 그걸 shell/substrate로 정리한 것이 이 메모다.

## Moat Map (verb: compete-against, 곁들여 rebuild)

**(a) 껍질 — 흡수 가능(거의 다, 싸게)**
- API 표면·파라미터·출력 스키마, `langchain-tavily` 소스(공개) → 인터페이스는 사실상 공짜.
- "검색→추출" 흐름 자체 → SearXNG + 추출 라이브러리로 주말 프로토타입.
- **낮은 교체비용 자체가 껍질이 얇다는 증거** — `create_agent`의 tools 자리에서 검색 도구를 바꾸는 건 몇 줄.

**(b) 기저 — 복제해도 안 딸려오는 것(왜 버티는지 축과 함께)**

| 기저 | 왜 저항하는가 |
|---|---|
| 에이전트 질의→소비 텔레메트리 플라이휠 | 인과 불투명(외부는 이 피드백 못 봄) + 재구성비용(신규 진입자 콜드스타트). *유일한 진짜 격리 메커니즘* |
| 크롤 신선도 운영 규율 + 롱테일 추출 정제 | 암묵지(인과 불투명) + 지속 운영 비용(일회성 빌드 아님) |
| Nebius 원가 바닥 + 유통 | 보완자산 게이트(co-generic → 특화 전환 중) |
| 개발자 디폴트 습관 | 유통 자산이나 약하고 상쇄 가능 |

**(c) compete-against 잔차(API를 다 따라잡아도 남는 것):** ①텔레메트리 플라이휠 ②Nebius 원가·유통 ③디폴트 습관. 결정적 뉘앙스 — ①의 인과 불투명성은 *문자 그대로의 클론*은 막지만, 경쟁자가 **동일한 공개 목적함수(LLM이 먹기 좋은 결과)**에 최적화하면 내부 리버스 없이 파리티에 도달한다. "베끼기"는 막아도 "따라잡기"는 못 막는 얕은 해자.

## 동일 로직 구현 — 껍질(주말~수 주)
"같은 로직"의 정체는 두 동사: `search(query)→results` + `extract(url)→clean text`.

```
질의 ─▶ [검색 백엔드] ─▶ URL/스니펫 ─▶ [재랭킹] ─▶ [본문 fetch+정제] ─▶ 에이전트용 정돈 결과
        (직접 크롤 X,             (RAG 목적함수로            (boilerplate 제거)
         상류 API/메타검색)         상위 N 재정렬)
```

껍질은 기성품 조립이 정석(직접 인덱스 만들지 말 것): 검색 백엔드(SearXNG 자체호스팅 또는 Bing/Brave Search API) · 재랭킹(크로스인코더 리랭커) · 추출/정제(trafilatura·readability 계열) · API 껍질(FastAPI 한 겹).

> **(검증 필요)** 위 도구 이름은 예시지 "현재 최선"의 검증된 추천이 아니다. 착수 전 각 컴포넌트(특히 리랭커 모델·상류 검색 API 약관/가격)를 그 시점 기준으로 재확인해야 한다 — 이 층은 빠르게 바뀐다.

## 서비스화 = 기저 짓기(진짜 비용)
데모를 서비스로 만드는 작업 목록이 곧 위 "복제 안 됨" 기저 항목과 일치한다(정의상 그렇다):

| 기저 항목 | 서비스화에서 실제로 해야 하는 일 |
|---|---|
| 크롤 신선도 | 재크롤 스케줄·캐시 무효화·상류 레이트리밋/ToS 대응(매일, 일회성 아님) |
| 랭킹 플라이휠 | 질의→어떤 결과가 실제 추출·소비됐나 텔레메트리 → 목적함수 튜닝 루프 |
| 롱테일 추출 품질 | 사이트별 boilerplate·페이월·JS렌더 엣지케이스 누적 처리 |
| 멀티테넌시 | API 키·쿼터·과금·레이트리밋·관측(Tavily 크레딧제가 이 층) |
| 지연 SLA | 병렬 fetch·타임아웃·부분결과 반환 |
| 법무 포지션 | 대규모 크롤 ToS/저작권 리스크 관리 |

핵심: 껍질만으론 데모, 이 표가 서비스. 다 채우면 결국 "또 하나의 Tavily"라 차별화가 없다 → 그래서 아래 개선점이 전략의 본체.

## 개선점 — Tavily 약점 공략
실습에서 *직접 발견한* 구멍으로 공략점을 짠다(일반론 "더 빠르게/싸게" 금지):

- **로컬라이제이션 우선** — Tavily는 영어권 쏠림 + `topic=news`에서 `country` 불가. 언어/지역을 1급 파라미터로 둔 검색 API는 비영어권에서 즉시 우위.
- **버티컬 특화** — 예: 한국 주식 재료·테마 전용. DART·증권사 리포트·네이버 금융을 큐레이션 소스로 고정 + 도메인 튜닝 랭킹. 범용 뉴스검색은 여기서 못 이긴다(더 좁은 목적함수로 플라이휠 공략).
- **구조화 출력** — 정제 텍스트가 아니라 타입드 데이터(엔티티·날짜·티커·감성) 반환 → 에이전트가 파싱 없이 바로 소비.
- **오픈/셀프호스트 옵션** — "닫힌 두뇌 + 낮은 교체비용" 시장 정면 공략. 투명성·온프레미스가 필요한 기업 세그먼트.
- **가격 정면승부 금지** — Nebius 원가 바닥은 못 이긴다. 원가 대신 버티컬 품질로 프리미엄.

## 판단 / 시사점
- **(사실)** Tavily 코어 검색·추출 엔진은 프로프라이어터리 클라우드 SaaS로 소스 비공개·셀프호스팅 불가, API 키 필요. 클라이언트 SDK(`langchain-tavily`, `tavily-python`, `tavily-mcp`)는 오픈소스.
- **(사실)** `country` 파라미터는 `topic="general"`에서만 동작하고 `news`/`finance`에선 못 쓴다. 언어 전용 파라미터는 없고, 언어는 질의문에 실어야 한다.
- **(보도·2차, 검증 필요)** 2026년 2월 Nebius가 Tavily를 인수했다고 다수 2차 매체가 보도 — 1차 확인 필요.
- **(분석·판단)** 소비자(사용자) 관점 함의: 낮은 교체비용은 *이득*이다. Tavily-특화 결합을 깊게 만들지 말고 `create_agent` tools 인터페이스 뒤에 얇게 둬 공급자 교체를 계속 "몇 줄"로 유지하라.
- **(분석·판단)** 만들 거라면 범용 클론이 아니라 좁은 버티컬로. 서비스화 비용의 대부분은 껍질이 아니라 기저에 있음을 전제로 로드맵을 짜라.
