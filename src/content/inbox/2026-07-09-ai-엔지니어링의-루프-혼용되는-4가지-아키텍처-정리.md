## TL;DR
- AI 엔지니어링에서 "루프"는 하나의 개념이 아니라 ==최소 4가지 다른 아키텍처(실행·작업·제품·시스템)를 가리키는 혼용어==이며, 여기에 인간의 목표 설정을 담당하는 Oversight Loop까지 더하면 5층 구조가 된다.

## 큰 그림

```
 ┌──────────────────────────────────────────────────────────────────┐
 │                   ⑤ Oversight Loop (감시 루프)                   │
 │          목표 설정 · 예산 배분 · 작업 선별 │ 종료: 없음(인간)     │
 │                     "The outer loop is agency." — Addy Osmani    │
 └──────────────────────────────┬───────────────────────────────────┘
                                │ 감시·조정
 ┌──────────────────────────────▼───────────────────────────────────┐
 │                   ④ System Loop (autoresearch)                   │
 │       반복 대상: prompt, eval, 모델 선택, harness                │
 │       종료 신호: eval 결과, judge, 인간 질의                      │
 └──────────────────────────────┬───────────────────────────────────┘
                                │ 개선
 ┌──────────────────────────────▼───────────────────────────────────┐
 │              ③ Product Loop (software factory)                   │
 │       반복 대상: 코드베이스 + 백로그                              │
 │       종료 신호: 이슈, 프로덕션 로그, 사용자 피드백               │
 └──────────────────────────────┬───────────────────────────────────┘
                                │ 산출
 ┌──────────────────────────────▼───────────────────────────────────┐
 │                  ② Task Loop (Ralph Loop)                        │
 │       반복 대상: 단일 산출물 (매 반복마다 fresh context)          │
 │       종료 신호: spec 준수 + 테스트 통과                          │
 └──────────────────────────────┬───────────────────────────────────┘
                                │ 수행
 ┌──────────────────────────────▼───────────────────────────────────┐
 │                ① Execution Loop (act-observe)                    │
 │       반복 대상: 단계(step) │ 종료: 환경 피드백 or 에이전트 판    │

   피드백이 되먹이는 루프가 아님 → 어떤 루프 안에서도 배치 가능한 토폴로지.
```

## 핵심

- 2025년 6월, AI 엔지니어링 커뮤니티에서 "루프"가 유행어로 떠올랐지만, 화자마다 지칭하는 아키텍처가 달랐다. 저자는 이 혼란을 **네 층의 루프 + 하나의 감시 루프**로 정리한다.
- 가장 안쪽인 **① Execution Loop**는 ==에이전트가 도구를 호출하고 결과를 관찰하며 다음 행동을 결정하는 act-observe 사이클==이다. 이것은 단일 작업 내의 단계(step)를 반복하며, 테스트 출력이나 API 응답 같은 환경 피드백으로 종료된다. 문제는 에이전트가 "끝났다"고 판단하면 실제로는 아니어도 멈춘다는 점인데, 이 한계를 해결하기 위해 바깥 루프가 등장한다.
- **② Task Loop**(Geoffrey Huntley의 "Ralph Loop")는 동일한 spec을 에이전트에게 **매 반복마다 fresh context window**로 다시 먹인다. 겉보기엔 비효율적이지만, 이것이 바로 핵심이다. 긴 세션에서 발생하는 ==**context rot**(컨텍스트가 서서히 오염되는 현상)과 compaction 이벤트를 원천 차단==하기 때문이다.
- **③ Product Loop**는 소프트웨어 팩토리로, 단일 산출물이 아니라 **코드베이스 전체와 백로그**를 지속적으로 반복한다. Warp의 Zach Lloyd는 이를 triage → spec → 구현 → 리뷰 → 검증 → 배포 → 모니터링의 라이프사이클로 구체화했고, Anthropic은 내부적으로 제품팀 코드의 65%가 Claude Tag로 생성된다고 주장한다(저자 **"주장"** 전달, **검증 필요**).
- **④ System Loop**(autoresearch)는 제품을 만드는 것이 아니라 ==제품을 만드는 시스템 자체를 반복==한다. prompt, eval, 모델 선택을 개선 대상으로 삼으며, Meta의 Brain2Qwerty v2가 대표 사례이나, 최종 훈련 설정은 여전히 인간이 선택했다는 점이 중요하다.
- 저자는 swyx의 다이어그램에서 "???? loop"로 비어 있던 최상위를 **⑤ Oversight Loop**로 명명한다. 목표 설정, 예산 배분, 작업 선별이 이루어지는 곳이며, Addy Osmani의 말처럼 =="안쪽 루프는 능력(capability)이고, 바깥쪽 루프는 주체성(agency)"==이다.

## 깊이

- **[① 실행 루프 — 토큰 루프와의 관계]** swyx의 스택에는 token loop가 실행 루프보다 더 안쪽에 존재하지만, 아무도 이것을 설계하지 않는다 — 모델 내부 동작이기 때문이다(**원문에 근거 있음**). 따라서 엔지니어가 설계 가능한 가장 안쪽 루프는 실행 루프다.
- **[② 작업 루프 — "낭비가 핵심"]** Ralph Loop는 매 반복마다 전체 spec을 다시 주입하는데, 저자는 이를 "겉보기 낭비가 의도"라고 설명한다. 비유하면, 매번 깨끗한 도마에서 요리하는 것과 같다 — 이전 요리의 잔여물이 다음 요리를 오염시키는 것을 막기 위해서다. 이 비유는 컨텍스트가 단순한 "오염"이 아니라 구조적 compaction으로 정보 손실이 발생할 수 있다는 점에서는 불완전하다.
- **[② 작업 루프 — 기관사 역할]** Geoffrey Huntley는 인간의 역할을 "기관사(locomotive engineer)"에 비유한다: 기차를 레일 위에 유지하는 것이 온전한 직무. 즉, 루프를 지켜보고 실패 패턴을 발견하여 다시는 재발하지 않도록 수정하는 것(**원문 주장**).
- **[③ 제품 루프 — 실제 수치 주장들]** Warp는 자동 PR 머지율을 20%에서 60%로 올려간다고 주장. Anthropic은 제품팀 코드의 65%가 Claude Tag로 생성된다고 주장. Mike Krieger는 이를 "fix this bug"가 아니라 "이 코드베이스 영역을 책임져라" 수준의 위임으로 설명(**모두 저자 전달, 2차 자료**).
- **[④ 시스템 루프 — 규모 양극단]** 최소 사례: Karpathy의 autoresearch(2026.3) — 약 630줄 Python, GPU 1개로 하룻밤 50개 hypothesis-edit-evaluate 실험. 최대 사례: Meta Brain2Qwerty v2(2026.6) — 에이전트가 디코딩 아키텍처를 반복 수정하여 word error rate 개선. 단, **최종 훈련 설정은 인간이 수동 선택**(**원문 기재**).
- **[⑤ 감시 루프 — 자율성 논쟁]** AIEWF 폐막 토론에서 두 진영이 충돌. Zach·Roland 진영: "자율성 dial을 올리되, 체크포인트를 의도적으로 배치하라. 오케스트라(지휘자 있는 시스템)를 먼저 만들고 그다음 팩토리로 가라." 반대 진영: Geoffrey Litt(Notion) — "이해를 위임하는 자는 에이전트에 의해 대체된다." Paul Bakaus — **"There is no auto, and there will be no auto."**(**원문 직접 인용**).
- **[자율성 dial — 개념]** 저자는 자율성이 네 루프 각각에 ==**독립적으로 존재하는 dial**==이라고 설명한다. 예: 실행 루프는 완전 자율, 제품 루프는 강력 감시 — 이런 조합이 가능하다. 원문에 수치 스케일이나 구체적 dial 설정값은 제시되어 있지 않다(**원문에 없음**).
- **[Agentic MapReduce — 루프 아님]** Cognition의 Devin Security Swarm은 저장소 전체에 병렬 bounded agent를 배치하고 결과를 집계하는데, 이를 루프라 부르는 경우가 있다. 저자는 이것이 dispatch → gather → validate **파이프라인**이며, 피드백이 다음 사이클로 되먹이지 않으므로 =="루프 없는 for문"에 불과==하다고 주장.

## 용어 풀이

- **Execution Loop (실행 루프)** — 에이전트의 act-observe 사이클. 비유: 요리사가 재료 썰기 → 볶기 → 맛보기 → 간 맞추기를 반복하는 한 번의 조리 과정. 비유가 깨지는 지점: 요리사는 "다 됐다"를 맛으로 판단하지만, 에이전트는 스스로 "완료"를 선언할 수 있어 틀릴 수 있다.
- **Task Loop / Ralph Loop (작업 루프)** — 같은 spec으로 에이전트를 처음부터 다시 실행하는 루프. 비유: 시험지를 매번 깨끗한 책상에서 다시 푸는 것, 이전 풀이의 오답이 보이지 않게. 비유가 깨지는 지점: 실제 시험은 시간이 한정되지만 Ralph Loop는 통과할 때까지 무한 반복 가능하다.
- **Context rot** — 긴 에이전트 세션에서 컨텍스트가 점진적으로 오염되거나 compaction으로 정보가 손실되는 현상. 비유: 오래된 화이트보드에 계속 써서 이전 글씨가 겹쳐 보이는 상태.
- **Product Loop / Software Factory (제품 루프)** — 코드베이스 전체와 백로그를 지속 관리하는 루프. 비유: 공장의 생산 라인 — 한 제품이 아니라 라인 자체를 운영하는 것.
- **System Loop / Autoresearch (시스템 루프)** — 시스템을 만드는 시스템을 개선하는 루프. 비유: 요리사가 레시피를 개선하는 연구. 비유가 깨지는 지점: 연구는 보통 인간이 하지만, 여기서는 에이전트가 가설-실험-평가 사이클을 자동화한다.
- **Oversight Loop (감시 루프)** — 목표 설정, 예산 배분, 작업 선별을 담당하는 최상위 루프. 비유: 회사의 이사회 또는 경영진.
- **Agentic MapReduce** — 여러 bounded agent를 병렬 배치하고 결과를 집계하는 패턴. 루프가 아니라 fan-out 토폴로지.

## 시각 자료

| 루프 | 반복 대상 | 종료 신호 | 인간 역할 | 대표 사례 |
|------|-----------|-----------|-----------|-----------|
| ① Execution | 단계(step) | 환경 피드백 / 에이전트 판단 | 루프 경계에서 승인·리뷰 | 일반 코딩 에이전트 |
| ② Task (Ralph) | 단일 산출물 | spec 준수 + 테스트 통과 | spec 작성, 실패 패턴 수정 | Geoffrey Huntley의 Ralph Loop |
| ③ Product | 코드베이스 + 백로그 | 이슈, 로그, 사용자 피드백 | 라이프사이클 구간별 설정 가능 | Warp Oz, Anthropic Claude Tag |
| ④ System | prompt, eval, 모델 | eval, judge, ask-a-human | 최종 체크포인트 유지 | Karpathy autoresearch, Meta Brain2Qwerty v2 |
| ⑤ Oversight | 목표·예산·선별 | 없음 (원문에 "none") | **인간이 거주** | 조직 리더십 |

**자율성 dial 개념:** 원문에 따르면 자율성은 각 루프에 **독립적으로 존재하는 dial**이다. 특정 루프 하나에 완전 자율을 주면서 다른 루프에는 강한 인간 감독을 둘 수 있다. 저자는 "흥미로운 공학 질문은 어느 진영이 이기느냐가 아니라, 각 dial을 올바르게 설정하기 위해 어떤 정보가 필요한가"라고 결론 짓는다. 구체적인 dial 수치나 스케일은 원문에 제시되지 않았다.

## 핵심 시사점 / 판단

- **(저자 주장)** "루프"라는 단어는 2025년 6월 AI 엔지니어링 커뮤니티의 hype cycle 정점에 있으며, 최소 4가지 다른 아키텍처가 이 한 단어 뒤에 숨어 있다.
- **(저자 주장)** 네 루프의 공통 교훈은 "인간이 추상화를 올리고, 판단을 스택 위쪽으로 밀어 올린다"는 것이다. 추상화를 올리지 말고, 어느 루프에서 올릴지를 선택해야 한다(Dex Horthy의 조언 참고).
- **(저자 주장)** ==루프에 종료 신호를 이름 붙이는 것과 실제로 그 신호를 배선(wire in)하는 것은 다르다== — 신호 없는 루프는 수렴하지 않고 외부에 의해 강제 중단될 때까지 달린다.
- **(검증 필요·불확실)** Anthropic 제품팀 코드의 65%가 Claude Tag로 생성된다는 수치는 Anthropic 내부 주장 전달로, 독립 검증 없음.
- **(검증 필요·불확실)** Warp의 자동 PR 머지율 20%→60% 목표 또한 회사 측 dogfooding 설명 기반.
- **(저자 주장)** "루프 without 피드백은 for문일 뿐" — Agentic MapReduce를 루프로 분류하는 것에 대한 저자의 반대 입장.
- **(불확실)** Geoffrey Litt의 "이해를 위임하는 자는 대체된다" 주장과 Paul Bakaus의 "no auto" 주장은 철학적 입장 표명으로, 실증적 근거는 원문에 제시되지 않음.
