## TL;DR
- AI는 조직의 강점과 약점을 모두 증폭한다. 사용자 문제 해결에 집중하지 못하는 팀은 AI로 더 빠르게 잘못된 방향으로 간다.

## 핵심
- Google과 Microsoft가 AI 생성 코드 비율을 발표했지만, 정의가 달라 정확한 비교는 불가능하다(불확실). 그러나 AI가 작성하는 코드 총량이 증가하는 방향은 명확하다.
- DORA 2025 보고서(약 5,000명 조사)에 따르면 AI는 도구 자체가 아닌 조직 시스템에서 투자 효과가 발생한다. Faros AI 분석(1만 명+)에서는 개인 PR 머지가 98% 늘었어도 조직 지표는 정체되어, 개인 속도 향상이 팀 경계에서 소실되고 있음을 시사한다.
- DORA가 식별한 7가지 역량 중 '사용자 중심 포커스'만 효과가 부호 반전된다. 이 역량이 약한 팀은 AI 도입 시 오히려 성능이 하락한다. AI가 '잘못된 방향으로 더 빠르게 가는 엔진'이 되기 때문이다(저자 주장).
- 인간이 모든 코드를 작성하던 시절에는 코드 작성 과정 자체가 사용자 이해의 필터 역할을 했다. AI 시대에는 이 필터가 사라지므로, 개발자는 '사용자의 대리인(proxy for the user)'으로 역할 전환이 필요하다(저자 주장).
- Algomatic은 PdM과 엔지니어가 AI로 프론트엔드 모킹을 만들고, 이를 사양으로 확정하는 방식을 실천한다. Anthropic Claude Code 팀도 PRD 대신 수십 개의 작동 프로토타입을 만든다고 Boris Cherny가 발언했다(2차).

## 깊이
- [사용자 포커스 부호 반전] 사용자 중심성이 약한 팀이 AI를 도입하면, 사용자 요구와 무관한 기능을 고속 양산하는 '피처 팩토리'가 된다. 대시보드에 속도와 배포 빈도만 표시된다면, 사용자는 이미 보이지 않는 상태다(저자 주장).
- [모킹이 사양이 되는 메커니즘] 문서로 합의하면 인식 차이가 발생하지만, 작동하는 모킹을 만지며 "이게 아니다/이쪽에 가깝다"를 반복하면 오차가 줄어든다. 구현 비용이 낮아진 결과, 사양 확정 최적 수단이 '문서 작성'에서 '작동하는 것 체험'으로 역전됐다(저자 주장).
- [실제 성과] Algomatic에서 반년 예정 대규모 개발이 4개월로 단축. 구현 단계만 보면 4개월→2개월로 정확히 절반 감소. 모킹이 사양으로 합의되어 프론트는 품질 정비만, 백엔드는 AI가 사양대로 생성하면 되므로 이해 시간이 거의 불필요했다(저자 주장, 검증 필요).

## 용어 풀이
- **DORA** — Google Cloud 산하 개발 생산성 연구 프로그램. / 비유: 소프트웨어 개발의 건강검진 기관. / 깨지는 점: 연구 결과이지 모든 조직에 적용되는 법칙은 아님.
- **Forward Deployed Engineer (FDE)** — 고객 현장에 상주하며 요구사항을 시스템으로 만드는 엔지니어. / 비유: 고객사 내부의 개발자. / 깨지는 점: Palantir/OpenAI 등 특정 기업 모델이며 일반화 가능 여부는 불확실.
- **PRD (Product Requirements Document)** — 제품 요구사항 명세서. / 비유: 건축 설계도. / 깨지는 점: 문서가 실제 사용자 경험을 보장하지는 않음.

## 핵심 시사점 / 판단
- (저자 주장) AI 시대 개발자의 핵심 역할은 코드 작성이 아닌 '사용자 대리인'으로서 AI 생성 방향 조정이다.
- (저자 주장) 모킹을 사양으로 만들면 후반부 재작업이 줄어들어 총 개발 기간이 단축된다.
- (검증 필요) Algomatic의 50% 단축 사례는 단일 사례이므로 일반화 가능성은 불확실.
- (불확실) DORA의 7가지 역량 중 부호 반전이 사용자 포커스에서만 발생하는지 추가 검증 필요.
