## TL;DR
- 채팅박스는 "가장 싸게 출시할 수 있었던" 기본값이었지만, 사용자가 원하는 것을 이미 아는 작업에는 세금(tax)이 된다. 2026년 승리하는 제품은 AI를 프롬프트 뒤가 아니라 **동사·캔버스·위임·배경 포착** 뒤에 둔다.

## 핵심

채팅박스가 AI의 기본 인터페이스가 된 이유는 **모델이 이미 만들어낸 형태**였기 때문이다. 언어 모델은 토큰을 내보내고, 그 토큰을 말풍선에 넣는 것이 가장 쉬운 포장법이었다. ChatGPT가 2022년 말 그 형태를 "미래"처럼 보이게 만들었고, 이후 모든 제품이 그 뒤를 따랐다.

그러나 2026년의 조건은 다르다. 모델 비용은 100분의 1로 떨어졌고, 제품 팀은 3년간 사용자가 실제로 어떤 작업을 하는지 배웠다. 그럼에도 대부분의 AI 제품 첫 화면은 여전히 빈 텍스트 박스다.

==저자는 이것이 기술적 이유가 아니라 조직적 이유라고 주장한다.== 채팅은 임원의 "AI 넣어달라"는 요구에 가장 안전하게 답할 수 있는 형태이며, 참여 지표(메시지 수, 세션 길이)를 부풀리고, 실패 시 사용자가 "잘못 물어봤다"고 자책하게 만들어 제품 책임이 모호해지기 때문이다.

Apple WWDC 2026은 이 모순을 압축해서 보여준다. 같은 키노트에서 Siri 챗봇 앱(채팅)과 스크린샷 기반 일정 추출 도구(ambient)를 동시에 발표했다. 저자의 해석에 따르면, 전자는 "추가할 수 있었던 것"이고 후자가 "실제 제품 작업"이다.

## 깊이

### [채팅이 맞는 영역과 아닌 영역]
- **채팅이 맞는 경우**: 사용자가 아직 무엇을 원하는지 모르는 단계 — 학습, 낯선 주제의 초안, 질문을 맴도는 탐색 단계. 이때 오고감 자체가 가치다.
- **채팅이 틀린 경우**: 사용자가 정확히 무엇을 원하는지 아는 단계. 빈 텍스트 박스는 매번 부과되는 세금이 된다. 저자는 ChatGPT의 2022년 데모(날짜를 물었는데 5문단 에세이를 반환한 사례)를 근거로 제시한다.

### [Verb Surface — 동사 표면]
- 사용자가 이미 선택한 객체(텍스트, 코드 블록, 화면 요소) 위에 AI를 **동사**로 겹치는 방식. 맥락은 이미 제품에 있으므로 프롬프트 엔지니어링이 불필요하다.
- 대표 사례: Cursor의 cmd-K 인라인 편집, GitHub Copilot, Notion의 슬래시 명령, Linear의 AI 하위 이슈, Apple Shortcuts.
- 제거하는 것: **설정 비용(setup)**.

### [Generative Canvas — 생성 캔버스]
- AI 출력이 "읽고 해석해서 복사해야 하는 단락"이 아니라 **직접 조작 가능한 산출물**이 되는 방식.
- 대표 사례: v0(작동하는 컴포넌트), NotebookLM Audio Overviews(오디오 파일), Claude Artifacts, ChatGPT Canvas(문서). 채팅이 존재하더라도 그것은 수정용 사이드 채널일 뿐, 캔버스가 곧 제품이다.
- 제거하는 것: **번역 비용(translation)** — AI 출력을 실제 작업물로 옮기는 과정.

### [Delegated Agent — 위임 에이전트]
- 사용자가 중간 수준의 추상화로 작업을 설명하고 넘기면, 에이전트가 중간 단계를 모두 수행하고 결과만 보고하는 방식.
- 저자는 "긴 응답을 가진 채팅"으로 오분류되기 쉽지만 본질적으로 다르다고 강조한다.
- 대표 사례: Claude Code(터미널 상주, 파일 읽고 코드 편집하고 테스트 실행), OpenAI Codex(클라우드 샌드박스에서 비동기 수행 후 PR 반환), OpenClaw(Claude Code 위에 구축된 전문 에이전트 OS).
- 제거하는 것: **감독 비용(supervision)**.

### [Ambient Capture — 배경 포착]
- 사용자가 AI를 호출하지 않는다. AI가 사용자가 **이미 하고 있던 일**을 관찰하고 부산물로 산출물을 생성한다.
- 대표 사례: Granola·Circleback(회의 녹음→노트 자동 생성), WWDC 스크린샷 도구(화면 관찰→일정 추출).
- 최신 형태: **상시 구동 로컬 에이전트** — 사용자 기기에서 백그라운드로 돌아가며 캘린더, 메시지, 화면 활동을 감시하고 중요한 것이 나타나면 적절한 앱으로 라우팅.
- 제거하는 것: **프롬프트 자체**.

### [채팅에서 벗어나지 못하는 조직적 이유]
- **정치적 안전성**: "Verb surface를 만들자"는 말은 "어떤 동사가 어떤 사용자에게 중요한지 합의하자"는 전략 작업을 요구한다. 합의가 안 되면 기본값인 채팅으로 회귀한다.
- **지표 착시**: 메시지 수, 세션 길이, 일일 활성 대화 수 — 모두 대시보드에서 긍정적으로 보인다. 채팅 세금은 대시보드에 직접 나타나지 않는다.
- **책임 모호성**: 채팅이 실패하면 사용자가 "잘못 물었다"고 생각한다. Verb surface가 실패하면 제품을 탓한다. 채팅은 구체적 약속을 하지 않는 안전한 선택이다.

## 용어 풀이

- **Chat tax (채팅 세금)** — 사용자가 이미 제품에 알려준 정보를 다시 입력하거나, 긴 응답에서 필요한 부분을 찾아내야 하는 추가 노력. 비유: 세금은 도로를 쓰려면 매번 내야 하는 통행료. / 비유가 깨지는 점: 실제 세금은 사회적 합의 하에 부과되지만, 채팅 세금은 제품의 설계 선택으로 사용자가 강제 부담한다.

- **Verb surface (동사 표면)** — AI가 "무엇이든 물어보세요"가 아니라 "이 동사를 적용하세요"로 작동하는 인터페이스. 비유: 만능 도마 대신 이미 썰린 재료에 '볶기/찌기' 버튼만 누르는 주방. / 비유가 깨지는 점: 주방 버튼은 고정된 기능이지만, Verb surface의 동사는 맥락에 따라 다르게 해석될 수 있다.

- **Generative canvas (생성 캔버스)** — AI 출력이 텍스트 설명이 아니라 직접 편집 가능한 작업물 자체인 인터페이스. 비유: 건축가를 설명하는 편지 대신 바로 수정 가능한 3D 모델. / 비유가 깨지는 점: 3D 모델은 구조적 제약을 가지지만, 캔버스 출력물은 형식이 자유로워 품질 편차가 크다.

- **Ambient capture (배경 포착)** — 사용자가 아무것도 요청하지 않았는데 AI가 관찰만으로 유용한 부산물을 만드는 방식. 비유: 비서 없이도 CCTV가 자동으로 출입 기록을 남기는 것. / 비유가 깨지는 점: CCTV는 수동 기록이지만, ambient capture은 판단(무엇이 중요한지)을 포함하므로 오류 가능성이 있다.

- **Delegation vs. Chat 혼동** — 긴 응답 채팅과 위임 에이전트를 구분하는 핵심은 사용자가 중간 단계에 관여하는지(loop 여부)다. 위임은 "결과만 알려줘"이고, 채팅은 "매 단계 확인해줘"다.

## 시각 자료

| 패턴 | 제거하는 비용 | 사용자 역할 | AI 역할 | 대표 제품 |
|------|-------------|------------|---------|----------|
| Chat | (없음 — 기준선) | 프롬프트 작성 | 응답 생성 | ChatGPT, Claude |
| Verb Surface | 설정 비용 | 객체 선택 + 동사 지정 | 선택된 맥락에 동사 적용 | Cursor cmd-K, Copilot |
| Generative Canvas | 번역 비용 | 산출물 직접 편집 | 산출물 생성 + 사이드 채널 수정 | v0, Canvas, Artifacts |
| Delegated Agent | 감독 비용 | 추상적 작업 위임 | 파일 읽기/편집/테스트/PR | Claude Code, Codex |
| Ambient Capture | 프롬프트 자체 | (호출 안 함) | 관찰 → 부산물 생성 | Granola, WWDC 스크린샷 |

```
사용자 인지 상태에 따른 인터페이스 선택:

  "무엇을 원하는지 모르겠다"  ──→  Chat (탐색)
  "무엇을 원하는지 안다"      ──→  Verb Surface / Canvas (실행)
  "어떻게 하는지도 안다"      ──→  Delegated Agent (위임)
  "요청할 필요도 없다"        ──→  Ambient Capture (자동)
```

## 핵심 시사점 / 판단

- **(저자 주장)** 2026년 AI 제품 조직에서 가장 가치 있는 인력은 "어떤 AI 기능에 동사를 주고, 캔버스를 주고, 위임을 주고, 배경에 둘지"를 결정할 수 있는 사람이다. 이 역할에는 아직 이름이 없다.
- **(저자 주장)** Frontier 모델은 수렴 중이며, 5~8GB 이하 소형 로컬 모델도 전용 작업에서는 대형 범용 모델과 대등하다. 따라서 모델 선택보다 **인터페이스 선택**이 차별화 포인트다.
- **(저자 주장)** 3가지 진단 질문 — ① 사용자가 이미 알려진 정보를 10번 중 3번 이상 반복하는가 ② 상위 10개 유스케이스 중 채팅박스를 제거해도 살아남는 것이 있는가 ③ 신규 사용자가 한 문장도 타이핑하지 않고 90초 안에 실제 작업을 완료할 수 있는가 — 중 2개 이상이 적색이면, 프롬프트 템플릿이 아니라 인터페이스 자체를 바꿔야 한다.
- **(검증 필요·불확실)** "소형 로컬 모델이 대형 범용 모델과 대등하다"는 주장은 작업 범위를 명시하지 않아 일반화하기 어렵다. 원문에 구체적 벤치마크 없음.
- **(검증 필요·불확실)** WWDC 2026 스크린샷 도구가 실제로 ambient capture 패턴으로 출시되었는지, 데모 수준인지 원문만으로 확인 불가.

## 확인 질문

- **Q1 (전이)**: 우리 제품에 이미 존재하는 상위 10개 유스케이스 중, 채팅박스를 제거해도 기능적으로 성립하는 것은 몇 개인가? 그 작업들은 4가지 패턴 중 어디에 배치해야 하는가?
- **Q2 (왜·어떻게)**: 채팅이 조직적으로 "가장 안전한 선택"이 되는 구조(지표 착시, 책임 모호성, 전략 회피)를 우리 팀에서 깨려면 어떤 의사결정 프로세스가 필요한가?
- **Q3 (경계)**: Ambient Capture 패턴의 상시 구동 로컬 에이전트는 프라이버시·배터리·오류 라우팅 문제를 어떻게 해결하는가? 원문은 이 한계를 다루지 않는데, 실제 출시 시 가장 큰 걸림돌은 무엇인가?
