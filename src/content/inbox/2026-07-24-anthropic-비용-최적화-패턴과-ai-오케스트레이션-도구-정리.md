## TL;DR
- (저자 주장) 고성능 LLM(Fable 5)을 '모두'가 아닌 '관리자'로만 배치하면 품질 92%를 비용 63%로 달성 가능하며, 개인 개발자는 Orca로 이를 멀티에이전트 관제탑으로 확장한다.

## 큰 그림
```
AI 비용·오케스트레이션 (스레드 전체)
│
├─ [핵심1] Anthropic 비용 최적화 패턴 ──┐
│   ├─ Advisor 패턴 (감독:Fable5 + 실행:Sonnet5)
│   ├─ Orchestrator 패턴 (Fable5=관리자, 툴 없음)
│   └─ 기본기: prompt caching 우선
│
├─ [핵심2] Orca 개인 오케스트레이션 ──┐
│   ├─ Master Orchestrator 에이전트
│   ├─ Worker 발행 (Grok, Codex 등)
│   └─ 병렬 워크플로우 → 1개월→2주 (저자 경험)
│
└─ [기타 언급]
    ├─ CocoIndex (오픈소스, 상세 불명)
    ├─ Apple HIG (디자인 가이드라인)
    └─ AI 시대 멀티태스킹 작업 스타일
```

## 핵심
- 스레드는 크게 두 축으로 구성된다. 첫 번째 축은 **Anthropic 비공개 웨비나에서 발표된**(저자 전언) Fable 5 비용 최적화 전략이고, 두 번째 축은 저자가 **직접 구축한** Orca 기반 멀티에이전트 관제탑 경험이다.
- Anthropic 패턴의 선택 기준은 단순하다. 작업이 **쪼개지지 않으면 advisor**, **쪼개지면 orchestrator**다. 이 기준 아래에서 두 패턴은 상호 배타적이라기보다 작업의 분해 가능성(granularity)에 따라 선택되는 분기점으로 이해된다.
- 저자는 Anthropic 패턴을 '이론'으로 소개하는 데 그치지 않고, Orca를 통해 orchestrator 패턴을 **개인 워크플로우에서 실제로 구현**했다고 주장한다. Master 에이전트가 위임·검증·스코프 규율을 담당하고 Worker가 실행하는 구조는 Anthropic orchestrator 패턴의 확장판으로 읽힌다.

## 깊이
- **[Advisor 패턴 — 주니어 옆 시니어]** Fable 5가 '감독'하고 Sonnet 5가 '실행'한다. 도입 장벽이 낮은데, 별도 아키텍처 변경 없이 기존 Messages API 호출에 **서버사이드 툴 하나만 추가**하면 되기 때문이다. (저자 전언, 2차) 단, 호출 빈도 튜닝이 필요한데 — 너무 잦으면 비용 이점이 사라지고, 너무 드물면 품질이 떨어진다. **이 지점은 평가를 돌리며 맞춰야 한다**고 강조된다. (원문에 평가 구체 방법은 없음)
- **[Orchestrator 패턴 — Fable 5를 관리자로만]** 방향이 반대다. 사용자 요청이 Fable 5로 **먼저** 들어가고, 발표자가 강조한 운영 규율은 **오케스트레이터에 툴을 주지 않는 것**이다. (저자 전언, 2차) 즉 Fable 5는 판단만 하고 실행은 하위 에이전트가 담당한다.
- **[Orca 구현 사례]** 저자는 Orca의 핵심을 '병렬 테스트'가 아닌 **안정적인 오케스트레이션 인터페이스**로 본다. (저자 주관 판단) Master Orchestrator가 역할 규율·위임·제안-승인-실행·회수 원칙·검증·스코프 규율을 하네싱하고, Grok/Codex 등을 'Worker'로 발행해 토큰을 최적화한다. 저자 주장에 따르면 명세 없이 시작한 작업을 AI 단독 기준 1개월에서 **2주로 단축**했다. (**불확실**: 재현 조건·작업 규모 미상)
- **[기본기 우선 원칙]** 발표자는 두 패턴보다 **prompt caching이 먼저**라고 단호히 말했다고 전한다. 비용 절감 폭이 가장 큰 순서대로 정리하라고 했으나, 구체적인 순서 목록은 원문에 포함되지 않았다. (**원문에 없음**)

## 용어 풀이
- **Advisor 패턴** — 고성능 모델이 판단만 하고 실행은 저비용 모델이 하는 구조 / 비유: 수술은 레지던트가 하고 교수님이 옆에서 거수만 드는 수술실 / 비유가 깨지는 지점: 실제 수술과 달리 LLM은 호출 지연·비용이 매번 발생하므로 '옆에 앉힌다'는 비유보다 'on-call 자문'이 더 정확할 수 있다.
- **Orchestrator 패턴** — 고성능 모델이 요청을 받아 하위 에이전트에 분배하는 구조 / 비유: PM이 티켓을 나눠주는 것 / 비유가 깨지는 지점: PM은 사람 판단이지만, LLM 오케스트레이터는 툴을 주지 않아야 한다는 역설적 제약이 있다(판단만 하고 실행하지 말라는 규율).
- **Prompt caching** — 반복되는 프롬프트 앞부분을 캐시해 재호출 시 비용을 줄이는 기법 / 비유: 이미 인쇄된 교재를 매번 복사하지 않는 것 / 비유가 깨지는 지점: 캐시 TTL·일치율에 따라 실제 절감률이 크게 달라진다.
- **Worker 발행** — 코드 레벨에서 특정 모델을 특정 역할로 강제 배정하는 것 / 원문에 구체 구현 상세는 없다.

## 핵심 시사점 / 판단
- (저자 주장) 고성능 LLM의 경제적 사용은 '더 많이 쓰는 것'이 아니라 **어디에 배치하느냐**의 문제다. 두 패턴 모두 Fable 5를 실행자가 아닌 판단자로 둔다는 공통점이 있다.
- (저자 주장) Orca 같은 도구를 쓰면 개인 개발자도 Anthropic이 제안한 orchestrator 패턴을 **멀티에이전트 관제탑**으로 확장할 수 있으며, 이는 'AI로 1개월 걸릴 작업을 2주에' 끝내는 생산성으로 이어진다.
- (검증 필요·불확실) "성능의 92%를 63% 비용으로"라는 수치는 웨비나 발표자 주장이며, 어떤 벤치마크·워크로드 기준인지 원문에 없다. 재현 가능성이 불확실하다.
- (검증 필요·불확실) Orca의 병렬 실행 성능 및 안정성은 저자의 주관적 경험이며, 공식 문서의 설명과 저자의 해석이 명시적으로 불일치한다고 저자 자신이 언급한다.
