## TL;DR
- TypeScript 7.0은 Go로 재작성된 네이티브 컴파일러로, 풀 빌드 기준 8~12배 속도 향상과 메모리 절감을 제공하지만 아직 프로그래밍 API가 없어 일부 에코시스템(Vue, Svelte, Angular 템플릿 등)은 과도기적 제약이 있다.

## 핵심

TypeScript 7.0의 가장 큰 변화는 **컴파일러 전체를 Go로 네이티브 포팅**했다는 점이다. 기존 TypeScript 컴파일러는 JavaScript로 작성되어 V8 엔진 위에서 동작했는데, 이를 Go로 재작성함으로써 네이티브 코드 실행 속도, 공유 메모리 멀티스레딩, 그리고 파싱·타입 체킹·이밋(emit) 단계의 병렬화를 얻을 수 있게 되었다.

이 속도 향상은 단순히 `tsc` 빌드 시간만 줄여주는 것이 아니라, **개발 경험 전체의 피드백 루프를 단축**시킨다. 에디터에서 파일을 열 때 첫 오류가 뜨기까지의 시간, 자동 완성, find-all-references, 진단(diagnostic) 갱신 등 언어 서버가 관여하는 모든 상호작용이 빨라진다. 실제 ==VS Code 코드베이스에서 첫 오류 표시까지 17.5초에서 1.3초로 줄었다==는 것이 저자의 보고다.

속도만큼 중요한 것이 **생태계 연속성**이다. TypeScript 7.0은 아직 프로그래밍 API를 제공하지 않기 때문에 typescript-eslint 같은 도구나 Vue/Svelte/Astro 같은 임베디드 언어 도구들은 여전히 TypeScript 6.0의 API에 의존해야 한다. 이를 위해 `@typescript/typescript6` 호환 패키지를 통해 6.0과 7.0을 사이드바이사이드로 설치하고, `tsc`는 7.0을 쓰되 API 의존 도구는 6.0을 쓰도록 구성할 수 있게 했다.

## 깊이

- **[네이티브 포팅] Go 재작성의 설계 원칙**: 원문은 "원본 코드베이스의 구조와 로직을 최대한 충실히 유지하면서 새 코드를 작성했다"고 밝힌다. 이는 두 컴파일러 간의 결과 일관성과 호환성을 유지하기 위한 선택이었다. 즉, 알고리즘을 새로 발명한 것이 아니라 **기존 로직을 Go라는 더 빠른 실행 환경으로 이식**한 것이 핵심이다.

- **[병렬화] 타입 체커 워커 모델**: 파싱과 이밋은 파일 간 독립성이 높아 자연스럽게 병렬화되지만, 타입 체킹은 파일 간 복잡한 의존 관계가 존재한다. TypeScript 7.0은 **고정 수의 타입 체커 워커**(기본 4개)를 만들고, 각 워커가 독립적인 "세계관(view of the world)"을 갖도록 했다. 일부 공통 작업이 중복되지만, 동일한 입력에 대해 항상 동일한 파일 분할과 결과를 보장한다. `--checkers` 플래그로 이 수를 조절할 수 있으며, `--checkers 8`로 설정 시 vscode 빌드가 10.6초에서 7.51초로 추가 단축되었다고 보고된다.

- **[병렬화] --builders와 곱셈 효과**: `--builders`는 `--build` 모드에서 프로젝트 레퍼런스 빌드를 병렬화한다. `--checkers 4 --builders 4`면 최대 16개의 타입 체커가 동시에 돌 수 있어 메모리 부담이 급증할 수 있다. 원문은 이 균형 찾기가 머신과 코드베이스에 따라 다르다고 강조한다.

- **[Watch 모드] Parcel watcher의 Go 포팅**: 기존 Go 표준 라이브러리에는 파일 감시 API가 없었고, 서드파티 라이브러리도 안정성·크로스플랫폼 지원에 문제가 있었다. 폴링 기반 해결책은 `node_modules`가 많은 대규모 프로젝트에서 CPU 비용이 너무 컸다. 그래서 VS Code에서 이미 검증된 **Parcel bundler의 C++ 파일 감시기를 Go로 포팅**하되, 최소한의 어셈블리 심만 남겨 C++ 도구체인 의존을 피했다.

- **[호환성] 5.x → 6.0 → 7.0의 연쇄 변경**: TypeScript 7.0은 6.0의 새 기본값을 그대로 계승하고, 6.0에서 deprecated였던 항목들을 **하드 에러**로 격상했다. 주요 변경:
  - `strict` 기본 `true`
  - `module` 기본 `esnext`
  - `target: es5`, `moduleResolution: node/node10`, `baseUrl` 등 더 이상 미지원
  - `esModuleInterop`, `allowSyntheticDefaultImports`를 `false`로 설정 불가
  - `rootDir` 기본 `./`, `types` 기본 `[]` (명시 나열 필요)

- **[호환성] 템플릿 리터럴 타입의 유니코드 처리**: 이전에는 이모지 ``를 UTF-16 서러게이트 쌍(`\ud83d`, `\ude00`)으로 분리했는데, 7.0からは 하나의 유니코드 코드 포인트로 취급한다. 이는 `for...of`나 `[...str]`의 동작과 직관을 맞추는 방향이지만, UTF-16 코드 단위를 전제로 한 문자열 `Length` 유틸리티 등에는 **브레이킹 체인지**다.

- **[JS 지원] JavaScript 분석의 TypeScript 파일 분석과의 일치**: `@enum` 특수 인식 제거, postfix `!` 미지원, Closure 스타일 함수 문법 미지원 등, 느슨하게 작성된 JSDoc 코드베이스에 대한 타협을 줄이고 `.ts` 파일 분석과 일관성을 맞췄다.

- **[에디터] 임베디드 언어의 과도기적 제약**: Vue, MDX, Astro, Svelte, Angular 템플릿은 TypeScript 7.0의 언어 서버를 아직 사용하지 못한다. 이유는 **안정적 프로그래밍 API가 없기 때문**이며, Volar 같은 도구가 TypeScript를 자체 컴파일러/언어 서비스에 임베드할 수 없어서다. Angular 프로젝트는 CLI에서 `tsc`(7.0)로 빠른 오류 검출을 하고, 에디터에서는 6.0을 쓰는 하이브리드 구성이 권장된다.

## 용어 풀이

- **네이티브 포팅 (native port)** — 기존 언어(여기선 JavaScript)로 쓰인 프로그램을 다른 언어(Go)로 다시 작성하면서, 해당 언어의 실행 환경(네이티브 머신 코드)이 제공하는 이점을 활용하는 것. / 비유: 번역가가 원문의 의미를 그대로 살리면서 더 빠르게 읽히는 언어로 다시 쓰는 것. / 비유가 깨지는 지점: 단순 번역이 아니라 메모리 모델, 스레딩 모델 등 런타임 특성이 근본적으로 달라지므로 "동일한 결과" 보장이 훨씬 어렵다.

- **공유 메모리 멀티스레딩 (shared memory multithreading)** — 여러 스레드가 같은 메모리 공간을 읽고 쓰며 작업을 나누는 방식. / 비유: 여러 요리사가 같은 주방(메모리)에서 각자 요리를 하되, 같은 재료 저장소를 공유하는 것. / 비유가 깨지는 지점: 실제 구현에서는 타입 체커 워커가 "독립된 세계관"을 가져 일부 작업을 중복하므로, 완전한 공유 메모리 모델보다는 제한된 공유 모델에 가깝다.

- **LSP (Language Server Protocol)** — 에디터와 언어 서비스 간 통신 규약. 에디터가 "이 파일에서 자동 완성 보여줘"라고 요청하면 언어 서버가 응답하는 구조. / 비유: 레스토랑에서 웨이터(에디터)와 주방(언어 서버) 사이의 주문서 포맷을 표준화한 것. / 비유가 깨지는 지점: LSP는 프로토콜일 뿐 구현체가 아니며, TypeScript 7.0의 LSP 구현은 멀티스레드를 활용해 동시 요청을 처리한다는 점에서 기존과 차별화된다.

- **타입 체커 워커 (type-checker workers)** — TypeScript 7.0이 타입 검사를 병렬로 수행하기 위해 생성하는 고정 수의 작업 단위. / 비유: 시험지 채점관을 여러 명 두고, 각 채점관이 독립적으로 채점하되 같은 채점 기준을 공유하는 것. / 비유가 깨지는 지점: 채점관들이 완전히 독립적이지 않고 일부 공통 기준(글로벌 타입 정보)을 중복 계산한다는 점이 실제 워커 모델과 다르다.

- **프로젝트 레퍼런스 (project references)** — TypeScript의 `--build` 모드에서 여러 하위 프로젝트를 의존성 그래프에 따라 빌드하는仕組み. / 비유: 큰 건물을 지을 때 하청 업체들(하위 프로젝트) 간의 공사 순서를 조율하는 것. / 비유가 깨지는 지점: `--isolatedDeclarations`를 쓰면 의존성 그래프의 병목을 일부 우회할 수 있어, 단순한 순차적 조율보다 유연하다.

## 핵심 시사점 / 판단

- **(저자 주장)** TypeScript 7.0은 8~12배의 빌드 속도 향상과 6~26%의 메모리 절감을 제공하며, 이는 네이티브 Go 포팅과 공유 메모리 멀티스레딩의 결과다. 벤치마크는 5개 오픈소스 코드베이스 기준으로 제시되었다.

- **(저자 주장)** VS Code, Slack, Vanta, Canva, Microsoft 내부 팀(Loop, Office, PowerBI, Teams, Xbox) 등 대규모 코드베이스에서의 사전 테스트를 거쳐 프로덕션 준비가 완료되었다. 언어 서버 크래시가 60% 이상, 실패 명령이 80% 이상 감소했다는 데이터가 제시되었다.

- **(저자 주장)** Slack의 경우 CI 타입 체킹 시간이 7.5분에서 1.25분으로 줄었고, 머지 큐 시간의 40%가 절감되었다. Microsoft News Services 팀은 월 400시간의 CI 대기 시간을 절감했다.

- **(검증 필요·불확실)** 벤치마크가 특정 하드웨어 환경에서 수행되었는지, 그리고 `--checkers` 수를 늘렸을 때 메모리 증가폭이 코드베이스별로 어떻게 달라지는지에 대한 체계적 데이터는 원문에 상세히 나와 있지 않다. `--checkers 8` 수치가 같은 머신에서의 결과라고만 언급된다.

- **(검증 필요·불확실)** "순서 의존적 결과(order-dependent results)"가 드물게 발생할 수 있다고 언급되나, 구체적 시나리오나 재현 조건은 원문에 없음. 팀 재량으로 `--checkers` 수를 고정하라고만 권고한다.

- **(사실)** TypeScript 7.0은 프로그래밍 API를 포함하지 않고 출시되었으며, API는 7.1에서 제공될 예정이다. 이는 typescript-eslint, Volar 등 API 의존 도구들에게 실질적 제약이다.

- **(사실)** `target: es5`, `moduleResolution: node/node10`, `baseUrl`, `module: amd/umd/systemjs` 등 다수의 옵션이 7.0에서 하드 에러로 전환되었다. 이는 6.0 마이그레이션을 선행하지 않은 프로젝트에게 추가 작업을 요구한다.
