## TL;DR
- omp는 Pi(코딩 에이전트)의 포크로, 100k+ LoC Rust 네이티브 엔진 위에 LSP·DAP·서브에이전트·이미지 기반 컴팩션(snapcompact)·해시 기반 편집(hashline) 등 16개 기능을 "배터리 동봉" 형태로 묶은 코딩 에이전트 하니스다(저자 주장).

## 큰 그림
```
                         ┌──────────── omp (harness) ────────────┐
                         │   Pi 포크 + 16 모듈, MIT, 40+ LLM 공급 │
                         └──────────────┬────────────────────────┘
          ┌─────────────┬───────────────┼───────────────┬─────────────┐
    [코딩 인텔리전스]  [에이전트 오케]  [컨텍스트 관리]  [통합·확장]  [네이티브 엔진]
     ├─ LSP (14 op)    ├─ subagents    ├─ snapcompact  ├─ ACP/Zed    └─ Rust 100k+ LoC
     ├─ DAP (28 op)    ├─ advisor      ├─ TTSR         ├─ GitHub FS     ├─ shell (brush)
     ├─ hashline       ├─ collab(E2E)  ├─ mnemopi      ├─ web/14 prov   ├─ coreutils
     └─ ast_edit       ├─ review       └─ compaction   ├─ vault://      ├─ walker/grep
                       └─ orchestrate     4 strategies  └─ discovery     └─ minimizer
```

## 핵심
- omp는 "IDE가 이미 아는 것을 에이전트도 알게 한다"는 설계 철학을 가진다: LSP 연산(`willRenameFiles`)과 DAP 세션(lldb, dlv, debugpy)을 도구 표면으로 노출해, 기존 에이전트가 `print` 디버깅에 머무는 지점을 실제 stepping·frame inspection으로 넘어간다(저자 주장).
- 편집 경로는 두 축으로 최적화된다. **hashline**은 내용 해시(xxHash32) 앵커로 편집 위치를 지목해 "문자열 불일치 루프"를 제거하고, **ast_edit**은 tree-sitter 구조Rewrite를 preview→resolve 2단계로 커밋해 부분 적용을 막는다.
- 컨텍스트 창 관리는 LLM 호출 없이 해결하는 방향을 택했다: **snapcompact**는 이력을 픽셀 폰트 PNG로 렌더링해 비전 모델이 거의 그대로 읽어내도록 하고, **TTSR**(time-traveling stream rules)은 정규식 매칭 전까지 토큰 비용 0으로 규칙을 주입한다(저자 주장).
- 에이전트 간 협업은 **subagents**(작업트리 격리, 스키마 검증 yield)와 **collab**(AES-256-GCM, 릴레이가 평문을 보지 못함) 두 층위에서 이루어지며, **advisor**는 별도 모델이 매 턴을 검토해 aside/concern/blocker를 주입한다.
- 하부 엔진은 ripgrep·glob·find·bash(brush)를 프로세스 내부에 링크해 fork-exec 왕복을 제거하고, 1000ms TTL 공유 캐시로 grep/glob/LSP가 같은 트리 탐색을 재사용한다(저자 주장).

## 깊이
- **[snapcompact]** — 기존 하니스가 요약을 위해 별도 LLM 턴을 소비하는 것과 달리, 1568×1568 프레임에 6×10 픽셀 폰트로 ~40k 문자(~10k 토큰)를 담아 Anthropic 픽셀 공식 기준 3,279 이미지 토큰으로 읽힌다(저자 주장). Qwen2.5-VL-7B에서 cross-carrier retrieval 12/12, 질문 유사도 기하 r=0.94라는 **화이트박스 증거**가 제시되나, 외부 재현 여부는 **불확실**. 비전 미지원 모델은 `context-full`로 폴백.
- **[hashline]** — 모델이 줄 번호 대신 4-hex 태그를 가리켜 편집. "Grok 4 Fast가 동일 작업에서 출력 토큰 61% 절감, 성공률 6.7%→68.3%"이라는 벤치가 제시되나(저자 주장), 16모델×180태스크×3런 조건 외 환경은 **원문에 상세 없음**. 오래된 파일 편집 시 앵커가 발산하면 패치를 거부해 손상을 막는다.
- **[collab]** — `<room>#<key>` 형식에서 `#` 이후(URL fragment)만 클라이언트가 보유해 릴레이 서버가 키를 보지 못한다. full 링크는 32바이트 room key + 16바이트 write token을 담아 게스트가 프롬프트·중단·서브에이전트 조작 가능, view-only 링크는 읽기만. 브라우저 게스트는 `collab-web` 정적 클라이언트로 동일 링크 접속.

## 용어 풀이
- **harness** — 에이전트를 감싸 도구·모델·세션을 관리하는 "운반대". 비유: 자동차 섀시. 깨지는 점: 섀시와 달리 하니스는 LLM 프로바이더·규칙·메모리까지 포함한다.
- **TTSR (time-traveling stream rules)** — 모델 출력이 스트리밍되다 정규식에 걸리면 **그 지점에서** 중단·재시도하는 규칙. 비유: 녹화본을 되감아 다시 찍기. 깨지는 점: 실제 시간 여행이 아니라 토큰 스트림 중단·재주입이다.
- **snapcompact** — 이력 텍스트를 픽셀 PNG로 "스냅샷"해 비전 입력으로 압축. 비유: 책을 스캔해 이미지로 보관. 깨지는 점: OCR이 아니라 모델이 픽셀을 직접 토큰화하므로 텍스트 모델에선 작동 안 함.
- **hashline** — 줄 번호 대신 내용 해시로 편집 위치 특정. 비유: 페이지 번호 대신 문장 첫 소절로 찾기. 깨지는 점: 해시 충돌 시 4-hex로는 구별 불가(실제로는 파일 스냅샷과 조합).
- **mnemopi** — SQLite + 벡터 임베딩 + 그래프 기반 로컬 메모리. retain/recall/reflect/memory_edit 네 연산.

## 핵심 시사점 / 판단
- **(저자 주장)** "가장 유능한 에이전트 표면"·"첫 시도에 편집 성공"·"모델이 눈치채기 전에 검색 완료" 등 성능 표현은 마케팅 문구에 가깝고 독립 검증 없음.
- **(저자 주장)** snapcompact가 "입력 비용의 약 1/3"이라는 수치는 Anthropic 픽셀 공식에 의존하므로 타 프로바이더에선 성립하지 않을 수 있음.
- **(검증 필요·불확실)** hashline 벤치(16모델×180태스크)의 재현 조건, collab 릴레이(`my.omp.sh`)의 운영·프라이버시 정책, mnemopi 임베딩 모델의 정확도는 원문에서 불충분.
- **(사실)** Pi(`badlogic/pi-mono`) 포크, MIT, GitHub 공개(`can1357/oh-my-pi`), bun ≥ 1.3.14 요구, Windows 네이티브(WSL 아님) 지원은 저장소로 확인 가능한 사실성 주장.
