## TL;DR
- Linear의 속도는 단일 기술이 아니라 **로컬 우선 동기화 + 공격적 로딩 최적화 + 키보드/GPU 중심 UX** 세 축이 맞물린 시스템 속성이다.

## 큰 그림
```
Linear의 속도 시스템 (3축)
│
├─① 첫 로드 (빠른 paint)
│   ├─ 번들 최소화 ─ Rolldown 전환 → 전송 JS 50%↓, 메모리 70~80%↓
│   ├─ modulepreload ─ entry import 전 병렬 fetch로 폭포 제거
│   ├─ Service Worker ─ ~1,200 자산 precache, 이후 탐색 네트워크 스킵
│   ├─ 인라인 앱 셸 ─ <head>에 CSS/JS 넣어 HTML만으로 껍질 렌더
│   └─ 인증 후검증 ─ localStorage 신호로 먼저 렌더, 401 시 리디렉트
│
├─② 런타임 (동기화 엔진)
│   ├─ IndexedDB ─ 브라우저 내부가 실제 DB
│   ├─ MobX 객체 풀 수화 ─ Issue/Comment는 lazy-hydrate
│   ├─ 낙관적 로컬 우선 쓰기 ─ 네트워크는 확인 단계일 뿐
│   └─ 필드 단위 observable ─ 델타 하나 = 셀 하나만 리렌더
│
└─③ UX (체감 속도)
    ├─ Keyboard-first + ⌘K 커맨드 팔레트 (로컬 풀 검색)
    └─ GPU 애니메이션 ─ transform/opacity만, layout 속성 금지
```

## 핵심
- 전통 CRUD 웹앱은 *클릭 → HTTP → 서버 DB → 응답 → 리페인트*의 수백 ms를 스피너로 채우지만, Linear는 UI가 읽는 진실원을 **브라우저 내부 IndexedDB**로 옮겨 네트워크를 사용자 시야에서 제거한다(저자 주장).
- 변경은 메모리·IndexedDB·서버 전송 큐에 **동시 기록**되고, 서버는 WebSocket 델타로 다른 클라이언트에 배포한다. 즉 서버는 *source of truth*가 아니라 **sync target**이다.
- 첫 로드는 Rolldown 기반 번들러 재작성 → 레거시 브라우저 제거 → 라우트 수준 청크 분할 → `modulepreload`로 병렬화 → 서비스 워커로 precache하는 **누적 최적화 체인**으로 즉각적 paint를 만든다.
- 런타임은 IndexedDB → MobX 객체 풀로 **수화(hydrate)** 되고, 모든 모델 필드가 개별 observable이어서 50개 이슈 업데이트가 목록 전체가 아닌 50개 셀 리렌더로 끝난다.
- 엔지니어링 속도가 단일 상호작용을 빠르게 만든다면, **디자인 속도**는 shortcut·커맨드 팔레트로 *상호작용까지의 경로*를 줄여 체감 속도를 완성한다.

## 깊이
- **[동기화-세 축 결합]** 로컬 DB만 있고 낙관적 쓰기가 없으면 저장 시 스피너가 남고, 낙관적 쓰기만 있고 세밀 observable이 없으면 업데이트마다 jank가 생기며, observable만 있고 로컬 DB가 없으면 초기 로드 대기가 남는다. 세 개가 모두 있어야 "속는 체감"이 성립한다는 점이 Linear 분석의 핵심 통찰(저자 주장).
- **[로딩-modulepreload vs SW]** `modulepreload`는 **지금 필요한 것**을 병렬로 당겨 import 폭포를 막고, 서비스 워커는 **다음에 필요할 것**을 준비한다. 역할이 다르며 둘 다 필요.
- **[인증-localStorage 신호]** `localStorage.ApplicationStore` 존재만으로 로그인 여부를 판단해 먼저 렌더하고, 실제 세션은 쿠키+서버 검증으로 비동기 조정. 실패 시 401이 리디렉트 트리거.

## 용어 풀이
- **IndexedDB** — 브라우저가 제공하는 영속 키-값 저장소 / 비유: "브라우저 안에 심은 소형 DB" / 비유 깨짐: SQL 조인·트랜잭션 격리 수준이 없어 복잡한 쿼리는 앱 레이어에서 해결해야 함.
- **MobX observable** — 값이 바뀌면 구독자에게 자동 알림 / 비유: "스프레드시트의 셀 의존성 그래프" / 비유 깨짐: React의 가상 DOM 재조정이 필요 없어지는 대신, 의존성 추적 실패 시 디버깅이 어려움.
- **수화(hydrate)** — 영속 저장소의 데이터를 메모리 객체로 되살리는 것 / 비유: "말린 해산물에 물 불려 원래 상태로" / 비유 깨짐: Linear는 전체가 아닌 **lazy-hydrate**로 필요한 테이블만 되살림.
- **modulepreload** — `<link rel="modulepreload">`로 ESM 모듈을 실행 전에 미리 받음 / 비유: "식자재를 요리 전에 다 꺼내두기" / 비유 깨짐: `crossorigin` 불일치 시 캐시 재사용이 깨져 이중 fetch 발생.
- **낙관적 업데이트** — 서버 응답 전에 UI를 먼저 바꿈 / 비유: "계산원 영수증 전에 음식 먼저 내오기" / 비유 깨짐: 충돌·롤백 로직이 복잡해지고, TanStack Query/SWR로 대체 가능한 수준이 있다고 원문은 지적.

## 시각 자료
**속도 계층별 비용 분류 (원문 기반)**

| 계층 | 속성 예시 | 파이프라인 비용 | 애니메이션 허용 |
|---|---|---|---|
| Composited | `transform`, `opacity` | GPU 위임, main thread 독립 | ✅ 권장 |
| Paint | `color`, `background-color`, `fill` | 레이아웃 스킵, 픽셀 재드로 | ⚠️ 제한적 |
| Layout | `width`, `height`, `margin`, `top` | 이후 모든 요소 위치 재계산 | ❌ 금지 |

**번들러 변천**: Parcel → Rollup → Vite → **Rolldown** (각 단계마다 JS/CSS↓, DX↑).

## 핵심 시사점 / 판단
- **(저자 주장)** "올바른 아키텍처와 디자인이 있으면 CSR도 즉각적으로 느껴질 수 있다" — SSR/ISR 대세 담론에 대한 반례 제시.
- **(저자 주장)** 대부분의 앱은 자체 동기화 엔진까지 필요 없고, TanStack Query/SWR의 낙관적 업데이트로도 "상당히 가까운" 체감 속도 달성 가능.
- **(검증 필요·불확실)** "전송 코드 50%↓, 콜드 로드 10~30% 개선, 메모리 70~80%↓" 수치는 Linear 블로그 출처로 3자 검증 없음.
- **(원문에 없음)** IndexedDB 용량 한계, 대규모 워크스페이스에서의 실제 메모리 사용량, 경쟁 도구 대비 정량 벤치마크.
