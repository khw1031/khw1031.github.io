## TL;DR
- Postgres는 "작을 땐 다 되던 것"이 규모가 커지면 스키마·인덱스·autovacuum·연결 관리 어디서든 터질 수 있으며, ORM 추상화 너머 SQL을 직접 제어할 줄 알아야 생존한다.

## 큰 그림
```
                    ┌─ 스키마 설계 (변경 불가에 가까움)
                    │   └─ PK(identity/UUID), timestamptz, FK 주의
                    │
  Postgres 생존 ────┼─ 읽기 최적화
                    │   ├─ btree 인덱스 / 복합 인덱스 / ORDER BY 정렬 맞추기
                    │   └─ 쿼리 플래너 + EXPLAIN ANALYZE
                    │
                    ├─ 쓰기 최적화
                    │   ├─ 짧은 트랜잭션, 행 단위 잠금
                    │   ├─ CREATE INDEX CONCURRENTLY
                    │   └─ 배치 처리(≈10x 처리량)
                    │
                    ├─ 운영·내구성
                    │   ├─ 연결 풀링(pgbouncer / pgxpool)
                    │   ├─ autovacuum 튜닝 → dead tuple / TX ID wraparound 방지
                    │   └─ 테이블·인덱스 팽창 관리(pg_repack, REINDEX CONCURRENTLY)
                    │
                    └─ 규모 확장 패턴
                        ├─ FOR UPDATE SKIP LOCKED (작업 큐)
                        ├─ 파티셔닝 (시계열, 과거 데이터)
                        └─ 트리거 + 배치 백필 (대형 테이블 데이터 이동)
```

## 핵심
- 스키마는 한 번 배포하면 바꾸기 어려우므로, **테이블 초안 → 필요한 쿼리 작성 → 스키마 수정**을 반복하며 "어떤 열을 자주 읽고/쓰고/갱신하는가"를 먼저 확정해야 한다. 읽기 위주면 인덱스를, 쓰기 위주면 트랜잭션 길이와 잠금 범위를 중심으로 설계가 갈린다.
- 읽기 성능은 "인덱스 단일 행 탐색 vs 순차 스캔"이라는 두 경로로 단순화할 수 있고, 2만 행 미만 테이블에서는 순차 스캔도 충분히 빠르다(저자 주장). 복합 인덱스는 필터 열 뒤에 `ORDER BY` 열을 정렬 방향까지 맞춰 배치하는 것이 경험칙이다.
- 쓰기는 트랜잭션을 짧게, 잠금을 좁게 유지하는 것이 기본이며, `CREATE INDEX`는 대형 테이블에서 쓰기를 전면 차단하므로 반드시 `CONCURRENTLY`를 붙여야 한다. Hatchet 측정에서 배치 처리는 처리량을 약 10배 끌어올렸다.
- 고빈도 쓰기 환경에서는 기본 autovacuum 설정이 dead tuple 회수를 따라가지 못하고, 방치하면 transaction ID wraparound로 대규모 다운타임이 발생할 수 있다(저자 경험). 이는 스키마가 아니라 **운영 설정** 문제다.
- 규모가 커지면 ORM 추상화 밖에서 `FOR UPDATE SKIP LOCKED`로 작업 큐를 만들고, 파티셔닝으로 autovacuum 부하를 분산하며, 트리거+배치 백필로 대형 테이블 데이터를 이동하는 패턴이 필요하다.

## 깊이
- **[쿼리 플래너]** 인덱스가 있어도 플래너가 순차 스캔을 선택할 수 있다. 인덱스는 heap과 별도 저장되므로, 인덱스에서 찾은 행을 heap에서 다시 읽는 비용이 순차 스캔보다 클 수 있기 때문. 해결은 `EXPLAIN ANALYZE`로 예측과 실측을 비교하고, `pg_stats` 통계가 최신인지(`ANALYZE`/autovacuum 빈도) 확인하는 것이다.
- **[autovacuum과 wraparound]** tuple은 행의 한 버전이고, 갱신/삭제 시에도 선행 트랜잭션이 끝날 때까지 구버전이 남아 dead tuple이 된다. `pg_stat_activity`에서 autovacuum이 1시간 이상 돌면(저자 기준) 설정 조정이 필요하다. TX ID를 다 쓰면 wraparound로 DB가 정지할 수 있다.
- **[FOR UPDATE SKIP LOCKED]** 선택한 행을 현재 트랜잭션이 예약하되 다른 세션은 건너뛰게 한다. 이를 `CTE + UPDATE ... RETURNING`과 조합하면 외부 메시지 브로커 없이 Postgres만으로 동시 안전한 작업 큐를 구현할 수 있다.

## 용어 풀이
- **dead tuple** — 행의 "폐기된 구버전" / 비유: 책의 이전 개정판이 서가에 남은 것 / 비유가 깨지는 지점: 실제로는 특정 트랜잭션이 아직 읽을 수 있어 즉시 삭제되지 않음.
- **transaction ID wraparound** — TX ID 공간(약 20억)이 소진되어 DB가 안전을 위해 멈추는 현상 / 비유: 계기판 숫자가 한 바퀴 돌아 0으로 돌아오기 전에 점검해야 하는 것.
- **순차 스캔(seq scan)** — 인덱스 없이 테이블 전체를 훑는 것 / 비유: 색인 없이 책장을 첫 장부터 넘기기 / 2만 행 이하면 메모리 캐시로 충분히 빠를 수 있음(저자 주장).
- **btree** — 정렬된 트리 구조 인덱스 / 비유: 사전의 색인 카드 / 탐색 시간 ≈ log(n).

## 시각 자료
```
쓰기 흐름과 병목 지점 매핑
─────────────────────────────────────────────────────────
App ──► Connection Pool ──► Transaction ──► Row Lock
          │                      │                │
     pg_bouncer/pgxpool     짧게 유지         필요한 행만
          │                      │                │
          ▼                      ▼                ▼
     connection storm 방지   외부 호출 금지    경합 최소화
                                     │
                                     ▼
                         autovacuum이 dead tuple 회수
                         (실패 시 → bloat → wraparound)
```

| 주제 | 핵심 도구/명령 | 비고 |
|------|---------------|------|
| 인덱스 생성 | `CREATE INDEX CONCURRENTLY` | 쓰기 차단 방지 |
| 느린 쿼리 분석 | `EXPLAIN ANALYZE` + explain.dalibo.com | 프로덕션 주의 |
| 통계 확인 | `pg_stats`, `ANALYZE` | autovacuum 빈도와 연동 |
| 동시 큐 | `FOR UPDATE SKIP LOCKED` | 외부 브로커 대체 가능 |
| 팽창 제거 | `pg_repack`, `REINDEX CONCURRENTLY` | `VACUUM FULL`은 비권장 |
| 대량 이동 | 트리거 + 배치 백필 | 단일 TX 장시간 실행 회피 |

## 핵심 시사점 / 판단
- **(저자 주장)** ORM은 소규모엔 충분하지만 규모가 커지면 SQL 직접 제어가 불가피하다. Prisma TypedSQL, sqlc 등으로 절충 가능.
- **(저자 주장)** 2만 행 미만 테이블은 순차 스캔으로도 충분하며,过早한 인덱스 추가는 오버헤드만 키운다.
- **(저자 측정)** 배치 처리는 처리량을 약 10배 높임. 단일 행 insert의 왕복/잠금 오버헤드가 주요 병목.
- **(검증 필요·불확실)** Postgres 19의 `REPACK...CONCURRENTLY`는 아직 Hatchet도 시험 전이라 안정성 미확인.
- **(원문에 없음)** 구체적인 autovacuum 파라미터 수치, 파티션 개수 기준, 하드웨어 사양별 임계값은 제시되지 않음.
