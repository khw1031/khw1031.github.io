## TL;DR
- nose는 코드를 의미 중간 언어(IL)로 낮춰 외형이 달라도 같은 계산을 하는 코드(Type-4 clone)를 탐지하고, 그 등가성을 "증명"했다고 주장하는 Rust 단일 바이너리 도구다(pre-1.0).

## 핵심

==nose의 핵심 아이디어는 코드를 token 단위가 아니라 의미 중간 언어(IL)로 "낮추면"(lowering), 표면적으로 다르게 보이는 코드도 같은 계산으로 판별할 수 있다는 것이다.== 기존 copy-paste 탐지기(예: token run 비교)는 변수명 변경(Type-2)까진 잡지만, 루프를 `all()`로 바꾼 경우나 De Morgan 법칙으로 조건을 뒤집은 경우(Type-4)는 놓친다. nose는 이 격차를 **proof-carrying frontier packet**이라는 추적 가능한 증명 조건 묶음으로 메우려 한다.

도구는 단일 Rust 바이너리로 배포되어 런타임·서비스·네트워크 의존성이 없고, `.gitignore`를 기본 존중한다. 세 채널(syntax·semantic·near)을 기본 병렬 실행하며, CI에서는 `--mode syntax`로 copy-paste만 게이트하는 등 용도별 분리가 가능하다.

## 깊이

### [의미 IL과 lowering] 어떻게 "다른 모양의 같은 코드"를 잡는가
- **세부**: 원문의 예시에서 `all(x != 0 and x != 1 for x in xs)`와 `for` 루프+`if`+early return은 token 단위로 완전히 다르다. nose는 양쪽을 의미 IL로 낮춘 뒤, **loop ↔ all() 변환**과 **De Morgan 법칙**을 proof packet(`python-loop-demorgan-all-2026-07-07`)으로 추적하여 등가를 "증명"한다.
- **한계 지점**: 이 증명은 **pure scalar comparison**과 **boolean-only** 조건에서만 성립한다. side-effect가 있거나, 단락 평가(short-circuit) 순서가 다르면 깨질 수 있으며, 저자는 이를 proof conditions에 명시한다고 **주장**한다.

### [proof-carrying frontier packet] "증명"의 구조
- **세부**: 각 Type-4 매치는 어떤 증명 조건(loop 조건, vacuous truth, short-circuit, boolean-only 등)을 만족했는지 **packet 이름과 날짜**로 추적 가능하다. 탐지기에 admission되기 전에 조건이 감사(auditable)된다는 구조.
- **검증 필요**: "equal fingerprint ⟹ equal behavior, never a false equivalence"라는 저자 주장은 매우 강력하다. 그러나 이 "증명"이 형식 검증(formal verification) 수준인지, 휴리스틱+조건부 등가성 검사 수준인지는 원문에서 **명확히 구분되지 않는다(불확실)**.

### [CSS/HTML 등가성] 선언형 언어까지 확장
- **세부**: CSS는 computed-style 등가, HTML은 rendered-DOM 등가를 기준으로 삼는다고 **주장**. Vue/Svelte의 `<script>`/`<style>`/markup 영역도 분리 분석.
- **검증 필요**: "rendered-DOM equivalence"가 어떤 렌더링 엔진·버전을 기준으로 하는지 원문에 **없음**. 브라우저 간 차이, shadow DOM, dynamic binding 등은 다루는지 불확실.

### [CI 통합과 게이트] 실무 파이프라인 삽입
- **세부**: `--fail-on any`로 copy-paste 탐지 시 빌드 실패, `base=origin/main`으로 PR에서 한쪽만 수정된 복제본 감지, SARIF 출력 지원. `--format json`은 "versioned machine-readable contract"로 설계.
- **시사점**: CI 게이트는 syntax 채널(copy-paste)만 쓰는 것을 권장하여, semantic 채널의 오탐 리스크를 실무적으로 우회하는 구조로 읽힌다.

## 용어 풀이

| 용어 | 쉬운 말 / 비유 | 비유가 깨지는 지점 |
|------|---------------|-------------------|
| **Type-4 clone** | 같은 계산을 다른 방식으로 짠 코드. "같은 레시피를 다른 조리도구로 요리" | 계산이 "같다"의 기준이 언어·맥락마다 다르며, nose는 자체 IL 기준 등가만 판정 |
| **의미 IL (Intermediate Language)** | 코드를 "의미만 남긴 설계도"로 변환한 중간 표현 | IL 변환 과정에서 손실되는 정보가 있을 수 있으며(주석, 스타일, 일부 side-effect), 이 손실이 등가성 판단에 미치는 영향은 원문에 **없음** |
| **proof-carrying frontier packet** | 탐지기가 "왜 이게 같은지"를 추적 가능한 증명 카드 묶음 | "proof"라는 용어가 형식 검증(formal proof)을 암시하지만, 실제 구현 수준은 원문만으로 **불확실** |
| **lowering** | 고수준 코드를 저수준 표현으로 "내리는" 컴파일러 용어 | 컴파일러 lowering은 보통 의미 보존이 보장되지만, nose의 lowering이 의미 보존을 보장하는지는 **검증 필요** |
| **bounded (in bounded type-4)** | "제한된 범위 안에서만" Type-4 탐지가 가능하다는 자기 한정 | 모든 Type-4 clone을 잡는 것이 아니라, 증명 가능한 특정 패턴만 잡는다는 의미 |

## 시각 자료

### Clone Type 분류와 nose 채널 매핑

```
Type-1 (exact)     → syntax 채널    "토씨까지 같은 복붙"
Type-2 (renamed)   → syntax 채널    "변수명만 바꾼 복붙"
Type-3 (modified)  → near 채널      "일부 수정한 유사 복제"  (fuzzy)
Type-4 (semantic)  → semantic 채널  "같은 계산, 다른 코드"   (proof-carrying)
```

### query 필터 조합 매트릭스

| 목적 | 명령 예시 | 채널 |
|------|----------|------|
| 전체 탐색·트리아지 | `nose query src` | all (syntax+semantic+near) |
| CI 게이트 (copy-paste만) | `nose query src --mode syntax --min-size 80 --fail-on any` | syntax |
| PR diff 검사 | `nose query src base=origin/main` | all |
| 정확 등가만 필터 | `nose query src witness=exact` | semantic |
| 마크다운 중복 | `nose query docs` | (문서 전용) |

## 핵심 시사점 / 판단

- **(저자 주장)** "equal fingerprint ⟹ equal behavior, never a false equivalence" — 오탐이 구조적으로 불가능하다는 매우 강력한 주장. proof-carrying packet 설계로 이를 뒷받침한다고 하지만, **형식 검증 수준인지는 원문에서 확인 불가**.
- **(저자 주장)** "bounded"라는 자기 한정은 정직하다. 모든 Type-4를 잡는 것이 아니라, 증명 조건이 준비된 특정 패턴(frontier packet이 존재하는 경우)만 탐지한다는 의미.
- **(검증 필요·불확실)** CSS computed-style / HTML rendered-DOM 등가의 기준 엔진·환경이 명시되지 않음.
- **(검증 필요·불확실)** Pre-1.0이므로 JSON contract·proof packet 체계가 변경될 수 있음. 벤치마크 수치도 wiki에 있으나 원문 입력에는 **포함되지 않음**.
- **(사실)** MIT 라이선스, Rust 단일 바이너리, Homebrew/curl 설치 지원, SARIF 출력 지원은 README에 명시된 사항.
