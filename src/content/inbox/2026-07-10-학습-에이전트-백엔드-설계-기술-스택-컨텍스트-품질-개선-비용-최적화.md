## TL;DR
- 인프랩은 학습 에이전트 백엔드로 Spring MVC + Virtual Thread를 선택하고, 컨텍스트·Tool 설계와 LLM-as-a-Judge 품질 워크플로우, 명시적 프롬프트 캐싱을 통해 ==답변 품질과 비용 효율을 동시에 확보했다==.

## 큰 그림

```
[학습 에이전트 백엔드 설계 전체 흐름]

  ① 기술 스택 선정
  ├─ Spring MVC + Virtual Thread ← 채택
  ├─ Spring Webflux             ← 기각 (학습 곡선)

  ├─ 동적 컨텍스트: 유저 메모리, 진도, 화면 캡처
  └─ Tool: 요약노트 조회, 화면 캡처, 키워드 검색, 학습내역 조회

  └─ 과적합 방지: 반복 패턴 확인 후 수정

  └─ 명시적 캐싱 → TTL 제어, 90% 할인, 손익분기점 분석
```

## 핵심

학습 에이전트는 LLM 채팅 위에서 동작하므로, **하나의 요청이 수십 초간 연결을 유지하는 SSE 스트리밍 트래픽**을 안정적으로 처리해야 한다. 전통적인 쓰레드 풀 모델로는 쓰레드 고갈이 발생하기 때문에, 인프랩은 ==Virtual Thread를 활용해 동기 코드 스타일을 유지하면서도 높은 동시성을 확보하는== Spring MVC + Virtual Thread 조합을 선택했다.(저자 주장)

서버가 정해지자 다음 과제는 **에이전트가 무엇을 보고 판단할 것인가**였다. 커리큘럼·자막·진도 같은 정적 컨텍스트를 기본으로 깔되, 컨텍스트가 너무 많으면 **Lost in the Middle** 현상과 토큰 비용 증가가 발생하므로 필요할 때만 동적으로 로드하는 Tool 구조를 도입했다. 특히 비디오 화면 캡처는 이미지 분석 전용 서브에이전트로 분리해 환각과 토큰 낭비를 동시에 줄였다.

답변 품질은 사람이 정한 **Golden Dataset**을 기준으로 LLM이 LLM을 평가하는 워크플로우로 개선했다. 이 과정에서 단일 피드백에 과민 반응하는 **과적합** 문제가 발생했고, ==여러 피드백에서 반복되는 패턴만 반영하도록 수정해 해결했다==. 마지막으로 비용은 에이전트 역할 분리로 시스템 프롬프트를 최소화하고, 암시적 캐싱의 한계를 확인한 뒤 **명시적 캐싱**으로 전환해 제어 가능한 비용 절감을 달성했다.

## 깊이

### [기술 스택 — Virtual Thread 선택의 근거]
- Virtual Thread는 블로킹 I/O 발생 시 캐리어 쓰레드에서 자동 unmount되어, 동기 코드 스타일 그대로 높은 동시성을 확보한다. Java 25(JEP 491)에서 `synchronized` 블록 내 pinning 이슈가 상당 부분 해결되었다고 원문은 밝혔다. (저자 주장)
- Webflux는 SSE/스트리밍에 최적화되어 있지만 **리액티브 전염(Reactive Contamination)** — 체인 중 한 곳이라도 블로킹되면 이벤트 루프 전체가 멈추는 위험 — 과 팀 학습 곡선이 부담이었다.
- Nest.js는 AI/LLM 생태계가 풍부하지만 CPU 바운드 작업 시 싱글 스레드 이벤트 루프가 블로킹되는 한계가 있다.

### [컨텍스트 설계 — Tool로 동적 로드하는 이유]
- 컨텍스트가 많다고 무조건 품질이 오르지 않는다. 일정 임계치 이상에서는 **Lost in the Middle**(긴 컨텍스트 중간 정보를 놓치는 현상)이 강화되고, 입력 토큰 비용만 증가한다.
- 따라서 항상 넣는 기본 컨텍스트(커리큘럼, 페르소나, 자막)와 Tool 호출 시에만 로드하는 동적 컨텍스트(요약노트, 화면 캡처, 키워드 검색 결과)를 분리했다.

### [화면 캡처 — 서브에이전트 분리의 이중 효과]
- 이미지 분석은 토큰 소비가 크다. 본체 에이전트에 이미지 컨텍스트까지 넣으면 전체 컨텍스트가 비대해져 비용과 환각이 동시에 증가한다.
- 서브에이전트로 분리하면: (1) 이미지 분석에 필요한 최소 컨텍스트만 전달 → **환각 감소**, (2) 본체는 이미지 토큰을 받지 않음 → **비용 절감**.

### [LLM-as-a-Judge 워크플로우 — 과적합과의 싸움]
- Golden Dataset(~40개 Q&A)을 정답지로 두고, 평가관 에이전트가 점수·피드백을 생성, 조교 에이전트가 프롬프트를 개선하는 루프를 돌렸다.
- 문제: 단일 피드백에 바로 반응하면 프롬프트가 특정 케이스에 과적합된다.
- 해결: **여러 피드백에서 동일 문제가 반복될 때만** 프롬프트를 수정하도록 조건을 강화했다.
- 저자 주장: "모든 과정을 LLM이 처리하는 것보다, 사람이 개입하여 개선안을 함께 검수했을 때 답변 품질이 더 많이 향상되었다."

### [비용 — 암시적 캐싱의 실전 한계와 명시적 캐싱 전환]
- 암시적 캐싱: 공식 문서는 "prefix를 동일하게, 짧은 시간에 반복하면 캐싱된다"고 설명했지만, 실제로는 시스템 프롬프트 전체가 호출 간 **완전히 동일**해야 캐싱이 동작했다. TTL도 명시되지 않아 효과 측정이 어려웠다. (불확실 — 특정 벤더/Gemini 기준 경험)
- 명시적 캐싱: 개발자가 직접 캐시 객체를 생성·관리하며 TTL을 제어할 수 있고, 캐시 적중 시 90% 할인이 적용된다. 단, 캐시 보관 비용이 별도로 과금되므로 트래픽이 적으면 손해다.
- 손익분기점 예시(gemini-3.1-flash-lite 기준): 30,000 토큰 캐시 시 시간당 약 4.44회 이상 요청해야 이익.

### [ADR 문서와 코딩 에이전트 협업]
- 기술 결정 근거를 ADR(Architectural Decision Records)로 문서화하고, `CLAUDE.md`, `AGENTS.md` 등 AI 컨텍스트 파일에 링크로 연결했다. 코딩 에이전트가 팀의 의사결정 맥락을 이해한 채 작업할 수 있게 하기 위함이다. (저자 주장 — 효과 정량 데이터는 원문에 없음)

## 용어 풀이

- **Virtual Thread** — JVM이 관리하는 경량 스레드. / 비유: 플랫폼 쓰레드라는 '큰 책상'을 여러 Virtual Thread가 돌아가며 쓰는 '시간 공유 데스크'. / 비유가 깨지는 지점: `synchronized` 블록이나 JNI 호출 시에는 여전히 책상(pinning)을 차지하고 놓지 않는다(Java 25에서 상당 부분 해소).

- **Pinning** — Virtual Thread가 캐리어 쓰레드에서 분리되지 못해 OS 쓰레드를 계속 점유하는 현상. / 비유: 공유 데스크에 앉아 일하다가 전화(블로킹 I/O)가 왔는데, 자리에서 안 일어나고 계속 앉아있는 상태.

- **Lost in the Middle** — LLM이 긴 컨텍스트의 앞·뒤는 잘 활용하지만 중간 정보는 놓치는 현상. / 비유: 두꺼운 책의 첫 장과 마지막 장은 기억나는데, 중간 챕터는 흐릿한 것. / 비유가 깨지는 지점: 인간은 의식적 재독이 가능하지만 LLM은 어텐션 메커니즘에 의존하므로 구조적 한계다.

- **LLM-as-a-Judge** — LLM의 출력 품질을 다른 LLM이 평가하는 방법론. / 비유: 학생이 쓴 에세이를 다른 AI 조교가 채점하는 것. / 비유가 깨지는 지점: 평가관 LLM 자체의 편향·환각이 채점에 그대로 반영될 수 있다.

- **암시적 캐싱 vs 명시적 캐싱** — 암시적은 API가 자동 감지, 명시적은 개발자가 직접 생성·관리. / 비유: 암시적은 서점에서 "자주 보는 코너"를 알아서 만들어주는 것, 명시적은 내가 직접 사물함을 빌려 책을 넣어두는 것. / 비유가 깨지는 지점: 명시적 캐싱은 사물함 대여료(보관 비용)가 별도로 든다.

- **Reactive Contamination** — 리액티브 체인 중 한 지점의 블로킹이 이벤트 루프 전체를 멈추게 하는 현상. / 비유: 릴레이 달리기에서 한 주자가 멈추면 팀 전체 기록이 망가지는 것.

## 핵심 시사점 / 판단

- **(저자 주장)** Virtual Thread + Spring MVC는 리액티브 프로그래밍 없이도 LLM 스트리밍 트래픽을 감당할 수 있는 실용적 선택지다. 단, Java 25의 pinning 해결이 전제.
- **(저자 주장)** 컨텍스트는 "많을수록 좋다"가 아니라 "적시에 적절히"가 핵심이다. Lost in the Middle과 토큰 비용이 상한선을 만든다.
- **(저자 주장)** LLM-as-a-Judge 워크플로우는 효과적이지만, 과적합 방지(반복 패턴 확인 후 수정)와 사람 검수 개입이 품질 향상의 핵심이다.
- **(검증 필요·불확실)** 암시적 캐싱이 "prefix 동일" 조건에서만 동작했다는 경험은 Gemini 특정 버전 기준일 수 있으며, 다른 벤더·버전에서는 다를 수 있다.
- **(저자 주장)** 명시적 캐싱은 트래픽이 충분한 서비스에서만 비용 이익이 발생한다. 손익분기점 계산이 선행되어야 한다.
- **(원문에 없음)** Virtual Thread 채택 후 실제 쓰레드 사용량, 레이턴시, 처리량 등 운영 지표는 제시되지 않았다.
- **(원문에 없음)** Golden Dataset 40개로 전체 답변 품질을 대표할 수 있는지, 과적합 방지 후 일반화 성능은 어떠했는지에 대한 정량 데이터는 없다.
