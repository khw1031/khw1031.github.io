## TL;DR
- 2026년 6월 AI 엔지니어링 커뮤니티에서 "루프"가 유행어로 떠올랐지만, 사실 최소 4가지 다른 아키텍처가 이 한 단어 뒤에 숨어 있으며, 진짜 설계 질문은 "루프를 쓸 것인가"가 아니라 **각 루프의 자율성 다이얼을 어디까지 올릴 것인가**다.

## 핵심
- 저자(Aparna Dhinakaran)에 **따르면**, 2026년 6월 "루프"는 AI 엔지니어링 컨퍼런스(AI Engineer World's Fair)와 블로그들(Peter Steinberger, Boris Cherny, Addy Osmani, swyx, LangChain)에서 동시에 유행했지만, 화자들은 서로 다른 것을 가리키고 있었다.
- 저자는 이 혼란을 정리하기 위해 루프를 **안쪽→바깥쪽**으로 네 층(실행 → 작업 → 제품 → 시스템)에 나누고, 가장 바깥에 인간이 거주해야 하는 **다섯 번째 루프(oversight loop)** 를 추가한다.
- 네 루프는 각각 **반복 대상(iteratee)** 과 **종료 신호(exit condition)** 가 다르며, 이 두 가지를 명시하지 않으면 루프가 아니라 "멈춤 조건 없는 for문"이 된다는 것이 저자의 핵심 **주장**이다.
- 자율성은 네 루프에서 **독립적인 다이얼**이다: 실행 루프를 완전 자율로 돌리면서 제품 루프는 강력히 감독할 수 있으며, 역도 성립한다.

## 깊이

### [1. Execution Loop] 에이전트의 내부 사이클
- 도구 호출 → 결과 관찰 → 다음 행동 결정 → 반복. 한 **작업 내부의 단계**를 반복하며, 테스트 출력·API 응답·파일 내용 같은 **환경 피드백**으로 종료한다.
- 문제: 에이전트가 "다 했다"고 스스로 선언하면 실제로 끝난 게 아니어도 루프가 멈춘다. 그래서 현장에서는 이 루프를 **검증 루프로 감싸는** 패턴이 등장했다(저자 **주장**).
- swyx의 계층에는 이 아래에 "token loop"가 있지만, 이는 모델 내부 동작이라 인간이 설계하는 영역이 아니다.

### [2. Task Loop] Ralph Loop — 같은 spec으로 처음부터 다시
- Geoffrey Huntley가 명명한 패턴. **매 반복마다 컨텍스트 윈도우를 완전히 새로 할당**하여 동일한 spec을 다시 투입한다.
- "낭비가 핵심이다": 긴 세션에서 발생하는 **context rot**(컨텍스트가 서서히 오염되는 현상)과 **compaction event**(요약 과정에서 정보 손실)를 구조적으로 회피한다.
- 종료 조건은 **spec 준수 + 테스트 통과**. 인간은 spec을 쓰고, done을 판정하며, 반복되는 **실패 패턴을 찾아 영구 수정**하는 역할(기관사 비유)을 맡는다.
- Allie Howe(Keycard)가 AI Engineer World's Fair 메인 무대에서 "everything is a ralph loop"라고 인용한 것은 **사실**(컨퍼런스 발언).

### [3. Product Loop] 소프트웨어 팩토리
- Factory(Tereza Tížková)는 "소프트웨어 개발의 전체 라이프사이클을 자율적으로 돌리는 것"으로 정의했고, Warp의 Zach Lloyd는 이를 **triage → spec → 구현 → 리뷰 → 검증 → 배포 → 모니터링**으로 구체화했다(인터뷰 발언, **사실**).
- Warp의 내부 실험: 자체 오픈소스 레포를 Oz(팩토리 플랫폼)에 넣고 자동 PR 병합률을 **20%에서 60%로 점진적으로 올리는 중**(저자 보고, 2차).
- Anthropic 내부 실험: 제품 팀 코드의 **65%가 내부 Claude Tag**로 생성되며, Mike Krieger는 이를 "버그 고쳐라"가 아니라 "이 코드베이스 영역을 책임져라" 식의 **위임·능동형**으로 운영한다고 발언(**사실**, World's Fair 연설).
- 종료 신호가 코드베이스 **바깥**(이슈, 프로덕션 로그, 사용자 피드백)에서 온다는 점에서 앞의 두 루프와 질적으로 다르다.

### [4. System Loop] 자동연구(autoresearch)
- Roland Gavrilescu(Introspection): "내부 루프는 사용자 대상 업무, **외부 루프는 그 내부 루프를 연구·유지**한다." 반복 대상은 프롬프트, 하네스, 모델 선택, **eval 자체**.
- 미니멀 사례: Andrej Karpathy의 autoresearch(2026.3) — GPU 1대에서 하룻밤에 50개의 가설-수정-평가 실험을 돌린 ~630줄 Python(저자 보고, 2차).
- 실제품 사례: Meta Brain2Qwerty v2(2026.6) — 에이전트가 디코딩 아키텍처를 반복 수정해 word error rate을 크게 개선. **단, 최종 학습 설정은 사람이 수동 선택**(Meta의 caveat, 저자 인용).
- 저자 **주장**: 플래그십 사례에서도 인간이 마지막 체크포인트를 지키고 있다는 점이 이 루프의 현실적 한계다.

### [5. Oversight Loop] 저자가 이름 붙인 최상위 루프
- swyx의 다이어그램에서 "???? loop"로 비어 있던 최외곽 링. 저자는 이를 **oversight loop**(목표 설정, 예산 배분, 작업 도태)로 명명하고 **"인간이 거주해야 하는 유일한 링"** 이라고 주장.
- Addy Osmani의 무대 발언 "내부 루프는 capability, 외부 루프는 agency"를 근거로, agency가 바로 이 oversight loop의 소유물이라고 연결한다.

### [Agentic MapReduce] 루프가 아닌 pipeline
- Cognition의 Devin Security Swarm: 레포 전역에 **병렬 bounded agent**를 fan-out → 결과 집계. 이 패턴을 "루프"라 부르기도 하지만 저자는 **루프가 아니라고 주장**.
- 이유: dispatch → gather → validate는 **feedback이 다음 사이클로 돌아가지 않는 pipeline**이며, "피드백 없는 루프는 그냥 for문"이다. fan-out은 네 루프 **내부에서** 사용할 수 있는 **토폴로지**일 뿐.

## 용어 풀이

- **Loop (이 글에서)** — "조건이 충족될 때까지 반복하는 자율 사이클" / 비유: 온도계의 신호를 받아 보일러가 켜지고 꺼지는 것 / 비유가 깨지는 지점: 온도계 루프는 목표치가 고정되어 있지만, 여기서 말하는 루프는 목표(eval) 자체가 시스템 루프에 의해 수정될 수 있다.
- **Context rot** — 긴 세션에서 에이전트의 컨텍스트가 서서히 오염·퇴화되는 현상 / 비유: 오랫동안 같은 화이트보드를 지우지 않고 덧쓰면 earlier 내용이 번져 보이는 것 / 비유가 깨지는 지점: LLM의 "오염"은 시각적 잔상이 아니라 attention 분포 변화이므로 눈에 보이지 않는다.
- **Compaction event** — 컨텍스트 윈도우 한계로 인한 자동 요약·압축 과정에서 정보가 손실되는 사건 / 비유: 긴 회의를 3줄 요약하면 뉘앙스가 사라지는 것 / 비유가 깨지는 지점: LLM compaction은 요약 품질이 들쑥날쑥하며, 어떤 정보가 사라졌는지 **사후 검증이 어렵다**.
- **Ralph Loop** — 같은 spec으로 매번 fresh context에서 에이전트를 재시작하는 패턴 / 비유: 시험지를 매번 새 종이로 다시 풀게 하되, 정답 기준은 동일하게 유지 / 비유가 깨지는 지점: 학생은 이전 시도의 기억이 남지만, Ralph Loop는 의도적으로 기억을 지운다.
- **Software factory** — 코드베이스 전체 라이프사이클을 자율 루프로 운영하는 체제 / 비유: 자동차 공장 — 원자재(이슈)가 들어가면 완성차(PR/배포)가 나옴 / 비유가 깨지는 지점: 자동차 공장은 설계도가 안정적이지만, 소프트웨어 팩토리는 요구사항이 계속 바뀐다.
- **Autoresearch** — 시스템이 자기 자신의 프롬프트·eval·모델 선택을 반복 개선하는 외부 루프 / 비유: 연구실에서 자신의 실험 방법을 매일 개선하는 박사과정 / 비유가 깨지는 지점: 박사과정은 지도교수가 있지만, autoresearch는 eval이 그 역할을 대신해야 한다.
- **Oversight loop** — 목표 설정·예산 배분·작업 도태를 담당하는 최상위 인간 루프 / 비유: 오케스트라 지휘자 — 악기를 연주하지 않지만 어떤 곡을 연주할지 결정 / 비유가 깨지는 지점: 지휘자는 실시간으로 연주에 개입하지만, oversight loop는 대부분 **비동기·정책적**으로 작동한다.

## 핵심 시사점 / 판단

- **(저자 주장)** "루프를 쓸 것인가 말 것인가"는 잘못된 질문이다. 이미 쓰고 있으며, 진짜 질문은 **각 루프의 자율성 다이얼을 어디까지 올릴 것인가, 그리고 그 다이얼을 올리는 데 필요한 정보는 무엇인가**다.
- **(저자 주장)** 루프의 본질적 교훈은 "추상화 단계 위로 올라가서 인간 판단을 더 높은 루프로 밀어 올리는 것"이며, 네 루프를 구분하면 **어디로 올라가야 할지**가 보인다.
- **(저자 주장)** AIEWF의 가장 날 선 논쟁은 사실 "팩토리의 다이얼을 어디까지 올릴 것인가"에 대한 것이었다: Zach Lloyd·Roland Gavrilescu는 점진적 상향을, Geoffrey Litt·Paul Bakaus는 "이해를 위임하면 대체된다", "auto는 없다"며 반대.
- **(사실·컨퍼런스 발언)** Anthropic 내부 Tag 팀조차 **리뷰와 인간의 개념화 능력이 병목**이라고 보고 — 인간이 지켜둔 체크포인트가 이제 제약이 되고 있다.
- **(검증 필요·불확실)** Warp의 자동 PR 병합률 20%→60% 목표, Anthropic 65% 코드 생성 수치 — 원문에 숫자는 있으나 **독립 검증은 원문에 없음**.
- **(검증 필요·불확실)** "피드백 없는 루프는 for문"이라는 저자 정의는 개념적으로 깔끔하지만, 실제 파이프라인의 집계 결과가 다음 dispatch 전략에 영향을 주면 루프 경계가 모호해질 수 있다(원문에 없는 반론).
