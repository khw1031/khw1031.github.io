## TL;DR
- Claude Code 팀은 코딩 에이전트의 "루프"를 **4가지 기준(trigger, stop 조건, 사용 primitive, 적합한 작업 유형)** 에 따라 분류하며, 가장 단순한 루프부터 시작해 필요할 때만 복잡한 패턴을 쓰라고 권한다.

## 핵심

Claude Code 팀은 "루프 설계(designing loops)"를 프롬프트 엔지니어링의 다음 단계로 제시한다. 그들의 정의에 따르면 루프란 **에이전트가 종료 조건(stop condition)을 만족할 때까지 작업 사이클을 반복하는 것**이다. 이때 루프는 단일 개념이 아니라 **4가지 독립 기준** — 어떻게 시작되는가(trigger), 어떻게 끝나는가(stop), 어떤 Claude Code primitive를 사용하는가, 어떤 작업 유형에 맞는가 — 에 따라 분류되며, 이 기준들의 조합이 4가지 루프 유형(Turn-based, Goal-based, Time-based, Proactive)을 만들어낸다.

가장 기본인 **Turn-based 루프**는 사용자가 프롬프트를 보내면 에이전트가 컨텍스트 수집 → 실행 → 검증 → 응답하는 "agentic loop"를 한 바퀴 도는 구조다. 여기서 Claude가 스스로 "완료됐다"고 판단하면 멈춘다. 이 루프의 한계는 **무엇이 "충분히 좋은" 결과인지를 Claude가 스스로 결정**해야 한다는 점인데, 이를 보완하는 것이 **Goal-based 루프(/goal)** 이다. /goal은 성공 기준을 명시적으로 정의하고, Claude가 멈추려 할 때마다 **evaluator 모델**이 조건을 점검하여 미달이면 다시 작업하게 보낸다. 테스트 통과 수, 점수 임계값 같은 **결정론적(deterministic) 기준**이 특히 효과적이다.

반복적이지만 입력만 달라지는 작업(예: 매일 Slack 요약, PR 리뷰 대응)에는 **Time-based 루프(/loop, /schedule)** 가 적합하다. /loop은 로컬에서 일정 간격으로 재실행하고, /schedule은 클라우드로 옮겨 지속 실행한다. 마지막으로 **Proactive 루프**는 위 primitive들을 조합하여 **사람 없이도** 돌아가는 장기 자동화 파이프라인을 만든다. 예를 들어 /schedule로 버그 리포트를 정기 수집 → /goal로 완료 조건 설정 → dynamic workflows로 트리아지·수정·리뷰 에이전트를 오케스트레이션 → auto mode로 권한 승인 없이 전자동化する 식이다.

저자는 "모든 작업에 복잡한 루프가 필요한 것은 아니며, 가장 단순한 해결책부터 시작해 선택적으로 패턴을 적용하라"고 강조한다.

## 깊이

### [Turn-based] SKILL.md로 자가 검증 강화
- Turn-based 루프의 약점은 Claude의 자체 검증이 부정확할 수 있다는 것이다. 원문에 따르면 이를 개선하는 방법은 **수동 검증 단계를 SKILL.md 파일로 인코딩**하는 것이다.
- 예시: "좋아요 버튼 만들어줘" → Claude가 코드 읽기 → 수정 → 테스트 실행 → 결과 반환. 이때 SKILL.md에 정량적 체크 항목을 적어두면 Claude가 스스로 더 많이 검증할 수 있다.
- 원문 핵심 인용: *"The more quantitative the checks are, the easier it is for Claude to self-verify."*

### [Goal-based] Evaluator 모델의 역할
- /goal에서 눈여겨볼 구조는 **작업 모델과 평가 모델의 분리**다. Claude가 "다 했다"고 판단해도 evaluator 모델이 조건을 다시 점검하고, 미충족 시 되돌려 보낸다.
- 이는 인간의 "괜찮아" 판단을 외부화한 것으로, **결정론적 기준(test 통과 수, lint 점수)** 일수록 evaluator가 명확히 판정할 수 있다.
- 턴 상한("stop after 5 tries")을 명시하면 무한 루프를 방지할 수 있다.

### [Time-based] 로컬 vs 클라우드 실행
- /loop은 **로컬 머신**에서 돌아가므로 컴퓨터를 끄면 중단된다. /schedule은 이를 **클라우드 routine**으로 승격시킨다.
- 저자 주장: 외부 시스템(CI, PR 리뷰)과 인터페이스할 때, 이벤트 기반 웹훅 대신 **폴링(polling) 간격**으로 체크하는 것이 "단순한 접점"이 될 수 있다.

### [Proactive] Primitive 조합 패턴
- Proactive 루프는 단일 기능이 아니라 여러 primitive의 **합성(composition)** 이다.
  ```
  /schedule (트리거)
    → /goal (완료 조건)
      → dynamic workflows (멀티 에이전트 오케스트레이션)
        → auto mode (권한 자동 승인)
  ```
- 이 패턴은 버그 리포트 처리, 이슈 트리아지, 의존성 업그레이드, 대규모 마이그레이션 등 **잘 정의된 작업 스트림**에 적합하다.
- 원문에 따르면 작은·빠른 모델로 라우팅하고, 판단이 필요한 지점만 고성능 모델을 쓰는 것이 토큰 비용 관리 핵심이다.

### [코드 품질] 시스템적 접근
저자는 루프 출력 품질을 **개별 결과가 아닌 시스템 설계 문제**로 다뤄야 한다고 주장한다:
- 코드베이스를 깨끗이 → Claude가 기존 패턴을 따르므로, 나쁜 패턴이 있으면 그대로 복제된다.
- 두 번째 에이전트로 코드 리뷰 → 메인 에이전트의 **컨텍스트 편향**을 피할 수 있다. `/code-review` 스킬이나 GitHub Code Review 연동 사용.
- 개별 실패를 시스템 개선으로 연결 → "이 결과만 고치지 말고, 이 실패 패턴을 인코딩해 미래 모든 반복을 개선하라."

### [토큰 관리] 구체적 도구
- `/usage`: 스킬·서브에이전트·MCP별 토큰 사용량 분해
- `/goal`(인수 없이): 현재 턴 수·토큰 사용량 표시
- `/workflows`: 각 에이전트의 토큰 사용량 표시 + 중간 중단 가능
- 결정론적 작업은 **스크립트로 치환**: 예) PDF 폼 채우기를 매번 reasoning하지 말고 스크립트를 실행.

## 용어 풀이

- **Agentic loop** — 사용자가 한 번 프롬프트를 보내면 에이전트가 "컨텍스트 수집 → 행동 → 검증 → 응답"을 한 바퀴 도는 기본 단위. 비유하면 "심부름꾼이 갔다 와서 보고하는 한 왕복." 비유가 깨지는 지점: 심부름꾼은 보통 한 번만 가지만, agentic loop는 스스로 부족하다고 판단하면 같은 프롬프트 안에서 여러 번 왕복할 수 있다.

- **Primitive** — Claude Code가 제공하는 기본 실행 단위(/goal, /loop, /schedule, dynamic workflows 등). 비유하면 "레고 블록." 비유가 깨지는 지점: 레고는 수동 조립이지만, primitive 중 일부(auto mode, dynamic workflows)는 스스로 조립 방식을 바꾸는 메타 기능을 가진다.

- **Evaluator model** — Goal-based 루프에서 Claude의 출력이 성공 조건을 만족하는지 판정하는 별도 모델. 비유하면 "채점관." 비유가 깨지는 지점: 채점관은 보통 인간이지만, 여기서는 다른 LLM이 채점하므로 채점 자체에도 오류가 있을 수 있다.

- **Deterministic criteria** — 테스트 통과 수, 점수 임계값 등 모호함이 없는 성공 기준. 비유하면 "계산기 결과." 비유가 깨지는 지점: 코드 품질, UX 만족도 등 정성적 목표에는 적용하기 어렵다.

- **Dynamic workflows** — 여러 에이전트를 동적으로 오케스트레이션하는 기능 (research preview). 원문에 상세 메커니즘 설명은 없으며, "수백 개의 에이전트를 생성할 수 있다"고만 언급된다.

## 시각 자료

| 루프 유형 | Trigger | Stop 조건 | Primitive | 적합 작업 | 토큰 관리 핵심 |
|---|---|---|---|---|---|
| Turn-based | 사용자 프롬프트 | Claude가 완료 판단 | 기본 세션 | 일회성 짧은 작업 | 구체적인 프롬프트 + SKILL.md |
| Goal-based | 수동 프롬프트 | 목표 달성 or 턴 상한 | `/goal` | 검증 가능한 종료 조건이 있는 작업 | 명시적 완료 기준 + 턴 캡 |
| Time-based | 시간 간격 | 취소 or 작업 완료 | `/loop`, `/schedule` | 반복 작업, 외부 시스템 연동 | 간격 조정, 이벤트 기반 트리거 |
| Proactive | 이벤트+스케줄 | 개별 작업은 goal 달성, 루틴은 수동 해제 | 조합 (/schedule + /goal + dynamic workflows + auto mode) | 장기 자동화 스트림 | 작은 모델 라우팅, 파일럿 테스트 |

## 핵심 시사점 / 판단

- **(저자 주장)** 루프 설계는 프롬프트 엔지니어링의 진화 형태이며, "무엇이 좋은 결과인가"를 시스템에 인코딩하는 것이 핵심이다.
- **(저자 주장)** 모든 작업에 복잡한 루프가 필요한 것은 아니며, 가장 단순한 해결책에서 시작해 점진적으로 확장해야 한다.
- **(저자 주장)** Evaluator 모델 분리, 결정론적 기준 사용, 리뷰어 에이전트 분리는 코드 품질을 유지하는 구조적 방법이다.
- **(검증 필요·불확실)** Evaluator 모델의 정확도나 false positive/negative 비율에 대한 데이터는 원문에 없다. "evaluator가 Claude를 다시 작업으로 돌려보낸다"는 메커니즘이 실제로 얼마나 효과적으로 over-engineering을 방지하는지 불확실.
- **(검증 필요·불확실)** "수백 개의 에이전트를 생성할 수 있다"는 dynamic workflows의 실제 토큰 비용·안정성 데이터는 원문에 없음.
- **(원문에 없음)** 이 분류 체계가 Claude Code 외의 에이전트 프레임워크(Cursor, Windsurf, Copilot 등)에도 적용 가능한지는 언급되지 않았다.
