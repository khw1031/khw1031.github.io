## TL;DR
- Svelte 5는 **런타임이 아닌 컴파일러**가 반응성 코드를 생성하는 UI 프레임워크로, `$state`, `$derived`, `$effect` 등 **runes**를 통해 명시적 상태 관리와 심층 반응형 프록시, push-pull 전파 모델을 제공한다.

## 핵심
==Svelte는 다른 프레임워크가 브라우저에서 가상 DOM diffing으로 변경을 찾는 것과 달리, **빌드 시 컴파일러**가 선언형 컴포넌트(.svelte 파일)를 최적화된 JavaScript로 변환한다.== 이 때문에 런타임 오버헤드가 없고, 상태는 평범한 JavaScript 변수처럼 다루되 `$state` 같은 **rune**(컴파일러 지시어)를 붙이면 Svelte가 자동으로 반응형 프록시로 감싼다. 상태가 변경되면 의존하는 파생값과 이펙트가 **push 단계**에서 알림을 받고, 실제 재계산은 **pull 단계**(값이 읽힐 때)에 일어나는데, 이것이 push-pull 반응성 모델이다. 이렇게 컴파일러가 반응성 코드를 주입하기 때문에, 개발자는 프레임워크 전용 API 대신 일반 변수 할당(`count++`)으로 상태를 업데이트할 수 있다.

## 깊이
- [runes-$state] `$state(value)`는 숫자 같은 원시값은 그대로, 배열·단순 객체는 **deep reactive proxy**로 변환한다. 프록시는 속성 읽기/쓰기 시 Svelte가 코드를 실행하게 하여 `array.push()` 같은 메서드 호출까지 세밀한 업데이트를 트리거한다. 단, 클래스 인스턴스나 `Object.create`로 만든 객체는 프록시화 중단점이므로 그 안의 속성은 반응하지 않는다.
  - **비유 깨짐**: "프록시가 재귀적으로 감싼다"고 해서 모든 객체가 반응하는 것은 아님 — 클래스, Map, Date 등은 제외. Svelte는 `svelte/reactivity`에서 `Set`, `Map`, `Date`, `URL`의 반응형 버전을 별도로 제공한다.
- [$state.raw] `$state.raw`는 프록시화를 건너뛰고 **재할당만**으로 업데이트된다. 대용량 배열/객체에서 프록시 오버헤드를 피할 때 쓴다. 원문: `$state.raw`는 mutate 효과가 없고 reassign만 작동.
- [$state.eager] `await` 표현식과 함께 쓰일 때 상태 변경이 UI에 즉시 반영되도록 강제한다. 네비게이션 시각 피드백 등 제한적 용도로만 권장.
- [runes-$derived] `$derived(expression)`은 의존 상태가 바뀔 때 **lazy**하게 재계산된다(읽힐 때). `$derived.by(fn)`은 복잡한 파생 로직용. 파생값은 `$state`와 달리 프록시화가 안 된 원본 그대로이므로, 파생된 객체를 mutate하면 원본 배열에 영향이 간다(의도적 설계).
  - Svelte 5.25부터 파생값을 **재할당하여 덮어쓰기** 가능 (optimistic UI 용도).
- [runes-$effect] `$effect(fn)`은 DOM 마운트 후, 상태 변경 시 마이크로태스크로 실행. 동기적으로 읽은 반응값만 의존성으로 추적되며, `await`/`setTimeout` 뒤의 값은 추적되지 않는다. **return teardown function**으로 정리 가능. `$effect.pre`는 DOM 업데이트 **직전** 실행, `$effect.root`는 자동 정리 없는 수동 제어용.
  - **권장**: `$effect`는 탈출구(escape hatch)로 간주 — 상태 동기화에는 `$derived`를, DOM 조작·분석에는 `$effect`를.
- [runes-$props] 컴포넌트 입력은 구조분해로 받으며, fallback 값, 이름 변경, rest props 모두 지원. **props를 mutate하면 안 됨** — 소유하지 않은 상태를 변경했다는 경고(`ownership_invalid_mutation`)가 발생한다. 변경 전파는 콜백 prop 또는 `$bindable`로.
- [runes-$bindable] 부모→자식 단방향 props에 자식→부모 **양방향** 흐름을 추가. `bind:` 디렉티브와 함께 사용. 남용 시 데이터 흐름 예측 어려움.
- [마크업-{#each}] 배열·iterable 반복. `(key)`를 주면 Svelte가 아이템 삽입/이동/삭제를 지능적으로 처리. key가 없으면 끝부분 추가/제거 방식으로 작동.
- [마크업-{#snippet} / {@render}] Svelte 4의 slot을 대체. 템플릿 내에서 재사용 가능한 마크업 덩어리를 정의하고 파라미터를 받을 수 있음. 컴포넌트 안에 직접 선언하면 암시적 prop이 되며, 태그 사이의 콘텐츠는 `children` snippet으로 자동 전달.
- [마크업-{@attach}] Svelte 5.29+ 추가. 엘리먼트 마운트 시 또는 내부 상태 변경 시 effect로 실행되는 attachment 함수. 기존 `use:` action을 대체하며 fully reactive. 컴포넌트에도 전달 가능(Symbol prop).
- [이벤트 위임] 메모리 절약을 위해 `click`, `input`, `keydown` 등 다수 이벤트를 앱 루트에서 단일 리스너로 처리. 수동 dispatch 시 `{ bubbles: true }` 필요, `stopPropagation` 호출 시 위임 리스너 도달 못 함.
- [바인딩] `bind:value`(텍스트·숫자 입력 자동 형변환), `bind:checked`(체크박스), `bind:group`(라디오·체크박스 그룹), `bind:files`(파일 입력), `bind:this`(DOM 엘리먼트 참조), `bind:offsetWidth`(읽기 전용 크기).

## 용어 풀이
- **rune** — 컴파일러에게 "이 변수/표현식을 특별히 처리하라"고 지시하는 `$` 접두사 구문(`$state`, `$derived` 등). / 비유: JavaScript의 예약어처럼 생겼지만, Svelte 컴파일러만 알아보는 "마법 키워드". / 비유 깨짐: 런타임 함수가 아님 — 컴파일 후 다른 코드로 변환되므로, 일반 JS 환경에서 그대로 쓰면 오류.
- **push-pull reactivity** — push: 상태 변경 시 모든 의존자에게 즉시 알림. pull: 실제 파생값 재계산은 값이 읽힐 때까지 지연. / 비유: 식당에서 주문(push)은 즉시 주방에 들어가지만, 요리(pull)는 손님이 가져갈 때 완성. / 비유 깨짐: push 단계에서 "알림"만 가고 계산이 안 되므로, 의존 체인이 길어도 불필요한 중간 계산이 발생하지 않음 — 그러나 동시에 읽기 시점까지 값이 stale할 수 있다는 점은 주의.
- **state proxy** — `$state`로 만든 배열·객체를 감싸는 JavaScript Proxy. 속성 접근 시 Svelte가 변경 감지. / 비유: 유리 진열장 — 안은 그대로지만, 누가 건드리는지 센서가 알음. / 비유 깨짐: 프록시는 원본을 mutate하지 않음(내부 사본 관리). 구조분해(`let { x } = obj`) 시 반응성 끊어짐 — 일반 JS 평가 시점과 동일.
- **event delegation** — 다수 엘리먼트의 이벤트를 상위 루트 한 곳에서 처리. / 비유: 우체부 한 명이 동네 전체 우편 배달. / 비유 깨짐: `stopPropagation`을 쓰면 우체부가 중간에서 멈춤 → 루트 도달 못 함 → Svelte 핸들러 실행 안 됨.

| 상황 | mutate 효과 | 경고 발생 |

## 핵심 시사점 / 판단
- (저자 주장) Svelte는 가상 DOM 없이 컴파일러가 반응성 코드를 생성하여 런타임 오버헤드를 제거한다.
- (저자 주장) `$effect`는 탈출구로만 사용하고, 상태 동기화에는 `$derived`를 쓰는 것이 바람직하다.
- (저자 주장) props를 자식에서 mutate하면 안 되며, 콜백 또는 `$bindable`로 변경을 전파해야 한다.
- (검증 필요·불확실) "lean, tightly optimized JavaScript"라는 성능 주장 — 원문에 벤치마크 수치가 없음.
- (검증 필요·불확실) 이벤트 위임이 항상 성능에 유리하다는 주장 — 버블링 경로가 긴 경우 오버헤드 발생 가능성 있으나 원문에 비교 데이터 없음.

## 확인 질문
- Q1(전이): `$state`로 선언된 배열에서 `todos[0].done = true`가 UI를 업데이트하는 원리는? (프록시 속성 setter → push 알림 → 관련 파생/이펙트 dirty 마킹 → pull 시 재계산)
- Q2(왜·어떻게): Svelte가 `$effect` 대신 `$derived`를 상태 동기화에 권장하는 이유는? (effect에서 상태 업데이트는 무한 루프 위험 + 데이터 흐름 추적 어려움 → derived는 선언적 의존 그래프로 자동 관리)
- Q3(경계): `$state` 프록시의 구조분해(`let { x } = $state({x: 1})`)가 반응성을 잃는 이유는? (구조분해는 평가 시점의 값을 복사할 뿐, 원본 프록시와의 연결이 끊어짐 — 일반 JS의 동작과 동일)
