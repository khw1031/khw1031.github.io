## TL;DR
- 목표: 간판·광고물 사진을 받았을 때 ==계약기간이 끝난 모델==의 이미지가 쓰였는지 특정해서 잡아내기.
- 처음 떠올린 접근("현재 계약 모델만 등록 → 얼굴 유사도 threshold로 통과/탈락")은 **논리 방향이 뒤집혀 있다.** "현재 모델 아님"과 "만료 모델임"은 다른 조건이다.
- 입력이 **매장·간판 원본 사진 그대로**라면 화이트리스트는 못 쓴다. 원본에는 행인·직원·배경 인물이 다 잡혀 전부 "불일치=탈락"으로 쏟아진다.
- 정답: ==전·현직 모델을 모두 등록(blacklist)하고, 1:N 식별로 "누구인가"를 먼저 판정한 뒤 계약기간 로직을 별도로 적용==한다. 미상 얼굴은 무시, 만료 모델 히트만 사람 검토로 넘긴다.

## 문제 정의
- 회사가 특정 모델을 계약기간 동안 대표(시니어) 모델로 쓰고, 계약이 끝나면 그 모델 이미지 사용 금지.
- 받은 사진(간판·광고물)에 **현재 계약 대상이 아닌 모델**이 있는지 특정해서 파악하고 싶다.
- 이건 초상권/계약 위반 판정이라 놓침(FN)이 비싸다 → 자동 판정기보다 **재현율 우선 스크리닝 + 사람 최종 확인** 구조가 맞다.

## 화이트리스트 접근이 깨지는 이유
"현재 모델과 불일치 → 탈락"이 성립하려면 입력 얼굴이 사실상 `{현재 모델, 이전 모델}` 둘뿐이어야 한다(예: 대표 모델 얼굴로 이미 크롭된 입력). 하지만:

1. **원본 사진엔 무관한 얼굴이 섞인다** — 행인·직원·다른 포스터 인물이 전부 "현재 모델 불일치 = 탈락"으로 뜬다 → 오탐 폭발 → 도구 무용.
2. **"특정" 정보가 사라진다** — 화이트리스트는 "현재 모델 아님"까지만 말한다. 과거 모델이 여럿이면 누구인지 못 짚는다.
3. **가장 약한 구성이다** — 현재 모델만 등록하면 판정이 1:1 절대 threshold가 된다. 화질 나쁜 간판(원거리·반사·인쇄 보정)에서 정상적인 현재-모델 간판인데도 유사도가 threshold 밑으로 떨어져 **합법 광고를 위반으로 오판**한다.

원본 사진에서 "현재 모델 아님"은 거의 모든 얼굴을 가리키는 조건이라 정보가 없다. 유효한 질문은 **"이 사진에 그 만료 모델이 나오는가?"** 뿐이고, 이건 반드시 만료 모델을 등록해야 답할 수 있다.

## 판정 로직 (뒤집은 버전)
사진에서 검출된 얼굴마다:
```
등록 갤러리(현재 + 만료 모델) 대상 1:N 매칭
├─ 최고 유사도 < threshold → 미상 얼굴(행인 등) → 무시   ← 핵심
└─ 최고 유사도 ≥ threshold → 모델 M 식별
     ├─ M = 만료 → 🚩 플래그 (모델·이미지 내 위치·점수 함께)
     └─ M = 현재 → OK
```
비전(누구인가?)과 비즈니스 로직(지금 허용되나?)을 분리하는 게 핵심. threshold 하나에 계약 여부를 욱여넣지 않는다.

## 정확도를 크게 올리는 포인트
간판에 박힌 만료 모델 얼굴은 사실 회사가 이미 갖고 있는 **특정 광고 키비주얼(KV) 사진**이다. 즉 순수 face recognition이 아니라 **"알려진 이미지 찾기(near-duplicate/이미지 검색)"** 문제이기도 하다.

- 만료 모델의 **실제 광고 원본 크리에이티브(KV 크롭)를 그대로 갤러리에 등록** → "거의 같은 이미지" 매칭이 일반 얼굴 대조보다 정밀.
- 간판은 풍화·각도·가림이 있어 단순 perceptual hash는 깨짐 → 기하/광도 변형에 강한 **로컬 특징 매칭(ORB/SIFT + 호모그래피)** 또는 학습된 이미지 임베딩 사용.
- **face recognition은 fallback으로 병행** — 알려진 KV에 없는 새 사진이 쓰였을 때는 얼굴 대조만이 잡는다.
- 정리: **알려진 크리에이티브 near-dup(정밀도↑) + 얼굴 대조(미확인 이미지 커버)** 조합.

## 필요한 스택 (자체 호스팅 기준)
먼저 분기: **클라우드 API**(AWS Rekognition face search 등)는 착수 빠르지만 얼굴 생체정보 외부 반출(개인정보보호법 이슈)+호출당 과금. **자체 호스팅**은 반출 없음·비용 통제, 모델 수 수십~수백이면 부담 작음 → 권장.

| 레이어 | 하는 일 | 추천(최소) | 갈아탈 시점 |
|---|---|---|---|
| 얼굴 검출+임베딩 | 얼굴 찾고 벡터화 | **InsightFace**(SCRFD/RetinaFace + ArcFace 한 패키지) | 정확도 부족 시 최신 임베딩 모델 |
| 크리에이티브 매칭 | KV 포스터를 원본에서 찾기 | **OpenCV** ORB/SIFT + 호모그래피(RANSAC) | 대량이면 이미지 임베딩(CLIP/DINO 계열)로 후보 추림 |
| 벡터 저장/비교 | 유사도 계산 | 규모 작음 → **NumPy 인메모리 코사인** | 벡터 수만 넘으면 **FAISS**, 더 크면 pgvector/Qdrant |
| 메타데이터 DB | 모델·계약기간·참조 이미지 | **PostgreSQL/SQLite** + 이미지 스토리지 | 벡터도 DB에 → pgvector |
| 파이프라인/서비스 | 사진→검출→매칭→판정 | **Python**(+ 서비스화 시 FastAPI) | 처리량↑ → 큐(Celery/RQ) |
| 사람 검토 UI | 플래그+매칭 크롭+후보·점수 확인 | 가벼운 웹앱 | 물량↑ → 라벨링 툴 |
| 캘리브레이션 | threshold 튜닝, P/R 측정 | **scikit-learn**(ROC) | — |

최소 의존성: `insightface`, `onnxruntime`(GPU면 `-gpu`), `opencv-python`, `numpy`, `scikit-learn`, (서비스화 시) `fastapi`+`uvicorn`, `psycopg`/`sqlite3`.

데이터 흐름: `원본 사진 → InsightFace 검출/임베딩 → {현재+만료} 1:N 코사인 + (병행) OpenCV KV 매칭 → 만료 히트만 플래그 → 사람 검토`

## 주의점
- **재현율 우선 + 사람 확인** — 자동 탈락 말고 스크리닝. 플래그는 매칭 얼굴 크롭 + 후보 모델명 + 점수 + 계약기간을 사람에게 제시.
- **작은 얼굴** — 멀리 찍은 전경에서 간판 얼굴이 작으면 검출 실패/임베딩 손상 → 고해상도 입력 + 다중 스케일 검출(필요시 타일 분할).
- **참조는 모델당 여러 장** — 단일 임베딩은 취약. 각도·조명·연도별 + KV 크롭 포함, max/mean 유사도.
- **threshold는 계산이 아니라 캘리브레이션** — 실제 간판 사진 라벨셋으로 P/R 뽑아 운영점 결정. 고정값 위험.
- **NumPy로 시작** — 모델 수 적은데 처음부터 벡터 DB는 과설계. 실제로 커진 뒤 FAISS로.
- **생체정보 저장은 법적 이슈** — 얼굴 임베딩은 개인정보보호법상 생체정보 소지 가능 → 모델 동의·보관/파기 정책 사전 확인.
- **버전 확인** — InsightFace/OpenCV/FAISS는 수년째 안정적이나, 채택 직전 현재 시점 최신 버전·대안 임베딩 모델을 검색 확인.

## 다음에 정할 것
- 모델 수(수십 vs 수천) → 벡터 DB 필요 여부.
- 사진 처리량(하루 몇 장) → GPU·서비스화·큐 도입 시점.
- 자체 호스팅 vs 클라우드 최종 결정.
