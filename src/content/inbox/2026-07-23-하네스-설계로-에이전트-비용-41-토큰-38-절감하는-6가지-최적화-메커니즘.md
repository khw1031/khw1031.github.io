## TL;DR
- LLM 에이전트의 비용·속도는 모델 지능이 아니라 하네스(오케스트레이션 껍데기)의 구조적 설계에서 결정되며, 6가지 메커니즘으로 비용 41%·토큰 38% 절감이 가능하다는 **저자 주장**이 있으나 자체 평가 한계가 명확하다.

## 큰 그림
```
              ┌─────────────── 하네스 (Orchestration Layer) ───────────────┐
              │                                                            │
  입력 ──►   │  ① 2구역 프롬프트 분리 ──► 캐시 적중률 99.9%              │
              │  ② Compaction (압축)   ──► 토큰 증가: 기하→선형          │
              │  ③ Offload (덜어내기)  ──► 컨텍스트 8KB 이하 유지         │
              │       ┌────────────────────────────────────────┐           │
              │       │          LLM (모델 본체)               │           │
              │       └────────────────────────────────────────┘           │
              │  ④ Zero-token waiting ──► 유휴 시 토큰 0                  │
              │  ⑤ 서킷 브레이커       ──► doom loop 3회 차단             │
              │  ⑥ 모델 불가지론 뼈대  ──► 중소형 모델 호환              │
              │                                                            │
              └────────────────────────────────────────────────────────────┘
                                   ▼
                      비용↓41% · 토큰↓38% · 속도↑44%
                         (Writer 자체 평가, n=22)
```

## 핵심
Writer 연구진은 에이전트가 "똑똑한 모델"만으로 효율을 얻는 것이 아니라, 모델을 감싸는 하네스가 토큰 경제를 좌우한다고 **주장**한다. 이 하네스는 LLM에 넘길 정보를 사전에 줄이고(①②③), LLM이 쉬어야 할 때 완전히 멈추게 하며(④), 실패 시 무한 재시도를 차단하고(⑤), 어떤 모델을 꽂아도 포맷이 깨지지 않게 정제(⑥)하는 6개의 구조적 레이어로 구성된다. 각 메커니즘은 독립적이지만 서로 연결되어 있다: 프롬프트를 잘 분리해야(①) 압축(②)과 오프로드(③)의 캐시 효율이 극대화되고, 서킷 브레이커(⑤)가 있어야 압축된 컨텍스트가 doom loop로 다시 팽창하는 것을 막는다.

## 깊이
**① 2구역 프롬프트 분리** — 프롬프트를 '불변 구역'(도구 스키마, 시스템 지침)과 '가변 구역'(현재 시간, 상태)으로 물리적으로 나눈다. LLM 제공사의 prompt caching API는 앞부분이 동일하면 캐시된 토큰을 재사용하므로, 불변 구역을 앞쪽에 고정하면 입력 토큰 단가가 약 1/10로 떨어진다. 이는 Claude의 prompt caching이나 GPT의 cached tokens 할인 등 이미 존재하는 기능을 구조적으로 극대화하는 설계다.

**② Compaction (구조화된 압축)** — 대화 이력이 길어지면 단순 자르기(truncation) 대신, 값싼 보조 모델이 '핵심 요약·기억' 등 4가지 고정 체크포인트로 압축한다. 토큰 증가 곡선이 기하급수적→선형으로 안정화된다는 **주장**이지만, 압축 과정에서 맥락 손실이 발생할 수 있다는 점은 원문에서 다루지 않는다.

**③ Offload (컨텍스트 덜어내기)** — 긴 터미널 출력이나 웹페이지를 외부 파일 시스템으로 빼고, LLM에는 짧은 프리뷰만 전달한다. 서브 에이전트도 8KB 이하 요약만 상위 에이전트에 넘긴다. 이는 "모델이 읽을 필요 없는 정보는 주지 않는다"는 원칙으로, ①·②와 함께 **입력 토큰을 3중으로 줄이는 구조**를 형성한다.

## 용어 풀이
- **Harness** — 모델을 감싸는 오케스트레이션 껍데기 / 비유: LLM이 엔진이면 하네스는 변속기·브레이크·냉각계 / 비유가 깨지는 점: 실제 하네스는 모델 내부 추론을 바꾸지 않지만, 프롬프트 조작을 통해 추론 자체에 간접 영향을 준다.
- **Doom loop** — 에이전트가 같은 에러를 무한 반복하는 상태 / 비유: 문이 안 열리는데 계속 같은 열쇠로 시도하는 것 / 비유가 깨지는 점: 실제 에이전트는 루프 중에도 매번 토큰 비용이 발생한다.
- **Circuit breaker** — 실패 횟수 초과 시 자동 차단 / 소프트웨어 공학의 서킷 브레이커 패턴에서 유래.
- **Model-agnostic** — 특정 모델에 종속되지 않는 설계. 원문에 따르면 중소형 모델의 JSON 파싱 에러 등을 하네스가 전단에서 정제.

## 시각 자료

| # | 메커니즘 | 목표 | 절감 대상 | 원문 기준 핵심 수치 |
|---|----------|------|-----------|---------------------|
| ① | 2구역 프롬프트 분리 | 캐시 극대화 | 입력 토큰 비용 | 캐시 히트 99.9%, 단가 1/10 |
| ② | Compaction | 이력 압축 | 토큰 증가율 | 기하→선형 안정화 |
| ③ | Offload | 컨텍스트 경량화 | 입력 토큰 수 | 서브 에이전트 출력 ≤8KB |
| ④ | Zero-token waiting | 유휴 비용 제거 | 대기 토큰 | 완전 중단→이벤트 재개 |
| ⑤ | 서킷 브레이커 | 무한 재시도 차단 | 실패 토큰 | 3회 실패 시 강제 종료 |
| ⑥ | 모델 불가지론 뼈대 | 모델 호환성 | 교체 비용 | JSON/도구 호출 전단 정제 |

## 핵심 시사점 / 판단
- **(저자 주장)** 에이전트의 실용적 효율은 모델 지능보다 오케스트레이션 설계에서 나오며, 6가지 하네스 메커니즘으로 비용 41%·토큰 38%·속도 44% 개선이 가능하다.
- **(저자 주장)** Qwen 3.6, GLM 5.1 등 저성능 모델은 복잡한 오케스트레이션 지시를 소화하지 못해 성능이 떨어지므로, 이 모든 최적화는 모델 성능이 뒷받침되어야 한다.
- **(검증 필요·불확실)** 평가 표본 22개, 저자 32명 전원 Writer 직원, 타 하네스 비교 부재, 소스 코드 비공개 → 수치(41%, 38%, 44%)의 외부 재현성이 확인되지 않았다.
- **(검증 필요·불확실)** Compaction 시 맥락 손실 영향, Offload 8KB 임계값의 근거 등은 원문에 없음.
