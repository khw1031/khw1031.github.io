# OKF idea-doc template

`src/content/idea/{slug}.md`에 쓰는 평문 형식. 다른 노트와 같은 OKF 정신(한 문서 = 한 아이디어,
읽히는 산문 §핵심, 링크로 관계 표현). frontmatter는 `baseFrontmatter`(title·pubDate 필수)를 따른다 —
encrypt 스크립트가 저장 전에 이 스키마로 검증한다.

```markdown
---
title: (아이디어를 한 문장으로)
pubDate: 'YYYY-MM-DD'
description: (한 줄)
summary: (권장: 독립적 초록 — 아이디어 + 왜 유효한가)
tags: [...]
---

> 한 줄 명제: (이 아이디어가 결국 주장/제안하는 한 문장)

## 핵심
(산문. 아이디어가 무엇인지 → 왜 다룰 가치가 있는지 → 지금의 확신도. 키워드 덤프 금지.)

## 방법론
(고른 프레임워크 이름 + 왜 이 아이디어에 맞는지 한두 줄. methodologies.md 참조.)

## 전개
(고른 방법론의 단계를 실제로 밟은 결과. 예: JTBD면 job·상황·경쟁 대안; Pre-mortem이면 실패
 시나리오와 방어책. 방법론의 구조가 이 절의 구조가 된다.)

## 근거
(도메인 리서치. 링크마다 1차/2차 표시 + 한 줄 요약. bare URL 금지. 버전·시점 의존 내용은
 확인일/버전 명시. 1차 출처 최소 1개.)

## 다음 액션
(이 아이디어를 진전시키기 위한 검증 가능한 다음 스텝. 가설·실험·필요한 정보.)

## 열린 질문 / 내 관점
(아직 모르는 것, 내가 어떻게 보는지. 확신과 불확실을 분리해서.)
```

## 하위 아이디어가 생기면

한 파일로 무거워지면 note/spec과 같은 승격 규칙을 쓴다: 기본은 `{slug}.md` 단일 파일, 진짜 하위
아이디어가 생겼을 때만 `{slug}/index.md`(허브) + `{slug}/{child}.md`로 나눈다. 허브는 직속 자식을
목차로 자동 렌더한다.
