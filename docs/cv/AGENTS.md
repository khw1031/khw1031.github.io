# CV Writing Rules

## Scope

이 문서는 CV의 `주요 프로젝트` 섹션을 작성하거나 수정할 때 적용한다.

대상 파일은 주로 `src/data/cv.ts`이며, CV 문구를 마크다운이나 다른 데이터 파일로 옮기더라도 동일한 원칙을 따른다.

## Project Workspace

- `docs/cv/projects/`를 `주요 프로젝트` 작성 전용 작업공간으로 사용한다.
- 프로젝트 파일은 시작 월 기준 `YYYY-MM-slug.md` 형식으로 만든다.
- 신규 프로젝트는 `docs/cv/projects/_new-project-template.md`를 기준으로 작성한다.
- `docs/cv/projects/README.md`의 큐를 오래된 프로젝트부터 처리한다.
- 프로젝트 파일의 `status`는 `draft`, `needs-input`, `ready`, `published` 중 하나로 관리한다.
- `ready` 상태가 되기 전에는 `src/data/cv.ts`에 최종 반영하지 않는다.
- 사용자와 상호작용할 때는 한 번에 하나의 프로젝트만 다루고, 질문은 최대 3개로 제한한다.
- `docs/cv/project-index.md`를 posts, CV, evidence 문서의 연결 인덱스로 사용한다.
- 각 프로젝트는 `docs/cv/evidence/<project-id>/` 아래에 `metrics.md`, `interview-questions.md`, `concepts.md`, `references.md`, `rewrite-notes.md`를 둘 수 있다.
- `evidence` 문서는 공개용 원고가 아니라 작성/면접 대비용 작업 문서다. 저장소 자체는 공개 저장소로 간주하지 않는다. 다만 외부 게시물이나 공개 CV로 옮길 때는 별도 공개 가능성 검토를 한다.

## Project Entry Structure

- 각 프로젝트는 `What`, `Why`, `How`, `Impact`를 내부 점검축으로 삼는다.
- 화면에 `What`, `Why`, `How`, `Impact`를 세부 제목으로 노출하지 않는다.
- `Why`는 프로젝트의 필요성이나 문제 배경이 설명력을 높일 때만 포함한다.
- 최종 결과물은 1~2개의 문단형 항목으로 작성한다.
- 각 문단은 하나의 완결된 경력기술 문장으로 구성한다.

## Content Order

- 첫 문장은 `What`을 정의한다: 무엇을 만들었는지, 어떤 범위의 프로젝트였는지 명확히 쓴다.
- 필요하면 첫 문장 안이나 바로 다음 문장에 `Why`를 붙인다: 어떤 문제, 병목, 조직적 필요 때문에 시작했는지 쓴다.
- 다음 문장은 `How`를 설명한다: 사용한 워크플로우, 설계 방식, 기술 스택, 협업 방식, 자동화 방식을 쓴다.
- 마지막 문장은 `Impact`를 설명한다: 일정 단축, 품질 개선, 재사용성, 표준화, 운영 성과처럼 검증 가능한 결과를 쓴다.

## Sentence Style

- 기본 문체는 설명문이 아니라 경력기술서 문체로 쓴다.
- `-했다`, `-하였다`, `-했습니다`로 끝나는 회고형 문장은 피한다.
- 문장은 `개발`, `구축`, `전환`, `표준화`, `자동화`, `검증`, `운영`, `개선`, `확보`, `도입` 같은 수행 결과 명사로 끝내는 것을 우선한다.
- 자연스러운 문장 연결이 필요할 때만 `달성`, `기여`, `완료`를 사용한다.
- 한 문장은 너무 길게 늘리지 않는다. 문장 하나에 `What`, `How`, `Impact`를 모두 넣어 가독성이 떨어지면 두 문장으로 나눈다.

## Preferred Patterns

좋은 패턴:

- `Claude Code 기반 5단계 AI 워크플로우를 활용한 인테리어 플래너 웹뷰 개발.`
- `요구사항 분석, 시스템 설계, 태스크 분해, 구현, 리뷰 단계를 문서 기반으로 고정해 3인 팀의 병렬 구현과 재개 가능한 개발 흐름 구축.`
- `21일 개발 완료, 기존 예상 대비 62% 일정 단축, AI 협업 비율 86.8%, 테스트:소스 비율 0.80 확보.`

피해야 할 패턴:

- `인테리어 플래너 웹뷰를 개발했다.`
- `요구사항 분석부터 리뷰까지 진행했다.`
- `좋은 성과를 만들었다.`

## Detail Level

- 구체적인 수치가 있으면 우선 사용한다.
- 수치가 없는 경우에는 관찰 가능한 결과를 쓴다: `검색 가능`, `설치 가능`, `재사용 가능`, `동일 절차 적용 가능`, `운영 가능`.
- 내부 URL, 사내 저장소명, 비공개 인명, 민감한 조직 정보는 작업 문서에 기록할 수 있다. 단, 외부 게시물이나 공개 CV로 옮길 때는 별도 마스킹 또는 제거가 필요하다.
- 기술 스택은 나열만 하지 말고, 무엇을 가능하게 했는지와 함께 쓴다.

## References and Materials

- 프로젝트 문구를 작성하기에 근거가 부족하면 추측하지 말고 사용자에게 필요한 자료를 요청한다.
- 요청 대상은 프로젝트 문서, 공개 링크, 릴리즈 노트, 블로그 글, 스크린샷, 다이어그램, 성능 지표, 일정/티켓 요약, 전후 비교 이미지처럼 문구의 신뢰도를 높이는 자료다.
- 사용자가 제공한 자료가 내부 시스템 링크이거나 비공개 자료여도 이 작업공간에는 근거로 기록할 수 있다. 외부 게시물이나 공개 CV에 노출할지는 별도로 확인한다.
- `References`에는 내부 자료와 외부 공개 자료를 함께 둘 수 있다. 외부 게시 시 공개 가능 여부를 다시 확인한다.
- 이미지 자료는 작업 문서에 원본 위치나 설명을 남길 수 있다. 사내 화면, 고객 정보, 내부 URL, 비공개 수치가 보이는 이미지는 외부 게시 전에 마스킹 또는 제거한다.
- `References`는 본문 문단과 분리된 보조 영역이다. `What / Why / How / Impact` 문단을 대체하지 않는다.
- 근거 자료가 없으면 `References` 영역을 만들지 않는다.

## Formatting Rules

- `주요 프로젝트`의 각 프로젝트는 세부 제목 배열이 아니라 문단형 `prose` 항목으로 구성한다.
- 문단 수는 기본 1개, 복잡한 프로젝트는 최대 2개로 제한한다.
- 불릿 목록은 사용하지 않는다. 단, 사용자 요청으로 상세 버전을 별도 작성할 때만 허용한다.
- `References` 영역은 예외적으로 링크 목록 형식을 사용할 수 있다.
- 한 프로젝트 안에서 같은 수치를 반복하지 않는다.
- 같은 의미의 표현을 반복하지 않는다: 예를 들어 `표준화`, `공통화`, `재사용 가능`을 한 문단에 모두 넣지 않는다.

## Review Checklist

프로젝트 문구를 확정하기 전에 다음을 확인한다.

- 무엇을 만들었는지 첫 문장에서 바로 보이는가?
- 왜 필요했는지가 필요한 프로젝트에만 들어갔는가?
- 어떻게 했는지가 워크플로우나 실행 방식으로 구체화되어 있는가?
- 결과가 수치나 관찰 가능한 변화로 표현되어 있는가?
- 근거가 부족한 수치나 표현에 대해 사용자에게 자료를 요청했는가?
- 외부 게시용 레퍼런스와 내부 근거 자료가 구분되어 있는가?
- 문장이 `-했다` 중심의 회고형이 아니라 경력기술형 문체인가?
- `What / How / Impact` 같은 세부 제목이 화면에 노출되지 않는가?
